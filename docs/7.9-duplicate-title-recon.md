## 2. Frontend Error-Handling Current State

### Where the mutation lives

`InputConfigStep.tsx` calls `useInitialiseToolboxTalk()` (defined in
`web/src/features/toolbox-talks/components/learning-wizard/hooks/useInitialiseToolboxTalk.ts`),
which wraps `mutateAsync` against `POST /toolbox-talks/initialise`. The hook has no `onError` callback â€”
it only has `onSuccess`.

### What happens on a 400

The Axios response interceptor in `web/src/lib/api/client.ts` (line 176) simply re-rejects non-401 errors:

```typescript
return Promise.reject(error);  // client.ts:176
```

The error that propagates is an `AxiosError`, which extends `Error`. Its `.message` property is the generic
Axios string `"Request failed with status code 400"`. The server body (`{ message: "..." }`) lives in
`error.response.data.message` â€” Axios does not surface it on `.message`.

### Where the message gets lost

`InputConfigStep.tsx` lines 320â€“323:

```tsx
} catch (err) {
    const message = err instanceof Error ? err.message : 'Failed to create learning';
    toast.error(message);
}
```

`err instanceof Error` is **true** for an `AxiosError` (it extends `Error`), so `err.message` is read â€” but
that's `"Request failed with status code 400"`, not the server body.

**What the user sees:** a toast with `"Request failed with status code 400"`.

Additionally, the inline error banner at `InputConfigStep.tsx` lines 999â€“1008 also renders
`initialiseMutation.error.message` (the same generic string) because TanStack Query sets `isError = true`
whenever `mutateAsync` throws, regardless of whether the catch block already handled it.

So the user sees **two** generic error indicators and zero actionable information.

## 3. Implementation Shape Recommendation

**Frontend-only fix.** The backend message is already correct and specific.

### New talk creation (InputConfigStep)

**Change:** `InputConfigStep.tsx` `onSubmit` catch block (lines 320â€“323) and the inline mutation-error banner
(lines 999â€“1008).

**Approach:** Extract `error.response?.data?.message` from the Axios error. If it mentions "already exists",
surface it as a field-level error on the `title` `<FormItem>` via `form.setError`. For any other 400, fall
back to a toast. Remove the inline error banner â€” with field-level errors covering the title-collision case,
the banner only ever shows the generic Axios message for other 400s, making it worse than a toast.

```tsx
} catch (err) {
    // err is AxiosError â€” response body is in err.response?.data
    const data = (err as { response?: { data?: { message?: string } } }).response?.data;
    const serverMessage = data?.message;
    if (serverMessage?.includes('already exists')) {
        form.setError('title', { type: 'manual', message: serverMessage });
        form.setFocus('title');
    } else {
        toast.error(serverMessage ?? (err instanceof Error ? err.message : 'Failed to create learning'));
    }
}
```

The string `"already exists"` is part of the hard-coded error thrown by the command handler (`"A learning
with title '...' already exists."`) â€” no risk of false positives among other 400 cases from this endpoint
(the only other 400 sources are FluentValidation failures from the `_validator.ValidateAndThrowAsync` call,
which return a different shape: `{ message: ..., errors: [...] }`).

The inline mutation-error banner should be **removed** â€” it is redundant and always shows a less-informative
message than either the field-level error or the fallback toast.

**No changes needed to the hook or API function.**

## 4. Edit-Path Coverage

When a user renames an existing talk on Step 4 (Settings), the call is:
`PUT /api/toolbox-talks/{id}/settings` â†’ `UpdateLearningSettings` controller action.

This action returns a **409 Conflict** (not 400) with a different body shape:


Note: the key is `error` (not `message`), and a structured `code` field is also returned â€” which makes the
frontend disambiguation unambiguous (no message-string sniffing needed).

## 8. Out-of-Scope Items Flagged

- **Title uniqueness is per-tenant and covers all statuses** â€” the `AnyAsync` in the handler inherits the
  `HasQueryFilter` predicate which filters `!IsDeleted`, so the comparison is against active (non-deleted)
  talk titles only. If a title was previously used and then deleted, it can be reused. No concern here â€” this
  is correct behaviour and the error message is accurate.

- **The inline error banner** (`initialiseMutation.isError` block) currently shows for all errors, not just
  title collisions. Removing it is the right move (the catch block handles display), but it is a UX choice
  rather than a pure bug fix. If there's any hesitation about removing it, it can instead be conditionally
  hidden (`!form.formState.errors.title && initialiseMutation.isError`). Either way, the banner as-is is
  actively harmful â€” it always shows the wrong message.

- **`AxiosError` import** â€” `InputConfigStep.tsx` and `SettingsStep.tsx` do not currently import from
  `axios`. Duck-typing the cast (`err as { response?: { ... } }`) avoids a new import and is consistent with
  how the rest of the frontend handles Axios errors (no other component file imports `AxiosError`). Recommend
  duck-typing for consistency.

- **The legacy wizard's create/edit path** (`/admin/toolbox-talks/create`, `POST /api/toolbox-talks`) is a
  separate surface. From `toolbox-talks.ts` line 92: `createToolboxTalk` hits `POST /toolbox-talks` which
  maps to a different controller action than `Initialise`. That action's error handling is out of scope for
  Â§7.9, which explicitly targets the new wizard. If the legacy wizard has the same problem, it should be a
  separate backlog entry.

- **FluentValidation errors on Initialise** â€” the command has a `_validator` called via
  `ValidateAndThrowAsync`. The controller catches `FluentValidation.ValidationException` separately (lines
  369â€“372) and returns `BadRequest(new { message = ex.Message, errors = ex.Errors })`. This is a different
  body shape from the `InvalidOperationException` 400. The proposed fix (`serverMessage?.includes('already
  exists')`) correctly distinguishes them â€” FluentValidation messages describe field constraints ("Title must
  not be empty" etc.), not existence collisions.

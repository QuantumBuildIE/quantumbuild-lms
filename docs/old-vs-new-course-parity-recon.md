# Old vs. New Course Creation — Parity Recon

**Date:** 2026-07-13
**Scope:** Read-only investigation. No code changes.
**Purpose:** Exhaustively enumerate the legacy split-and-author course flow and the new compose-existing course flow (already shipped per `docs/course-in-new-wizard-scoping-v2.md` Chunks 1-4, git log `aafd20d`..`f6a37fa`), compare feature-for-feature, and flag any gap not already documented as a deliberate deprecation.

**Prior docs read in full before starting** (all in `docs/`): `course-in-new-wizard-scoping-v2.md`, `course-in-new-wizard-scoping.md` (superseded), `course-in-new-wizard-recon.md`, `course-in-new-wizard-impl-first-recon.md`, `course-dead-hooks-and-edit-toggle-recon.md`, `compose-existing-form-recon.md`, `course-in-new-wizard-smoke-checklist.md`.

---

## 1. Headline verdict

**Parity gaps found: 1 Category D item, cosmetic/low-impact.**

The compose-existing cutover (Chunks 1-4) is functionally complete and, on the metrics that matter for a compliance-training product, is a **strict improvement** over the legacy flow: it fixes both known legacy defects (quiz-question duplication across course items, and course-level settings — certificate/refresher/auto-assign — being silently dropped at publish) simply by not having the bug's precondition (no split-and-author step exists in the new flow to duplicate anything). The single Category D gap found (course-level Validation tab shows a generic empty state for every compose-existing-created course, forever, with no explanation that this is expected) is a minor, easily-explained UX rough edge, not a functional or compliance risk — the underlying validation coverage is unaffected because it lives at the item (talk) level, where it was already being generated when each item talk was authored.

No Category D item rises to "material" — nothing blocks the toggle default or the trial-customer cutover described in `course-in-new-wizard-scoping-v2.md`.

### Known bugs excluded per task instructions (not counted as preservable old-flow features)

- **Bug A — Quiz-question duplication across course items.** `SectionIndex` is captured on each question (`ContentCreationSessionService.cs:1035`) but never read/filtered anywhere; `PublishAsCourseAsync` passes the full unfiltered question list to `SyncQuizQuestionsToTalk` for every course item (`ContentCreationSessionService.cs:1974`). A 5-section/15-question document publishes 5 talks, **each carrying all 15 questions**. Documented in `docs/course-in-new-wizard-impl-first-recon.md:47-49` (§2.4) and its Q2 recommendation ("fix during any port"). **Verified still present** — `SyncQuizQuestionsToTalk(talk.Id, quizQuestions, ...)` call at line 1974 is unchanged; no filter on `SectionIndex` anywhere in the file (confirmed by grep).
- **Bug B — Course-level settings silently dropped.** `PublishAsCourseAsync` creates `ToolboxTalkCourse` with only `Title`/`Description`/`IsActive=true`/`RequireSequentialCompletion=true` (hardcoded) — `GenerateCertificate`, `RequiresRefresher`, `AutoAssignToNewEmployees` all land at entity defaults (`false`) regardless of what the admin configured in the Settings step, with no warning in `PublishStep.tsx`'s summary UI. Documented in `docs/course-in-new-wizard-impl-first-recon.md:55` (§2.5). **Verified still present** — `ContentCreationSessionService.cs:1846-1854` unchanged (confirmed by direct read; most recent touch to this file, commit `4dccbe6`, only affected an unrelated external-review DTO in a different method).

### Present in code but not reachable (dead code)

- **"AI Suggested" badge on the Course output-type card.** `ParseContentAsync` writes `session.OutputType = parseResult.SuggestedOutputType` and the UI shows a Sparkles badge on whichever card matches, but `ContentParserService.SuggestOutputType(int sectionCount)` unconditionally `return OutputType.Lesson;` — the badge can never actually appear on the Course card. A `CourseThreshold = 3` constant exists and is never referenced. Documented in `docs/course-in-new-wizard-impl-first-recon.md:23-33` (§2.1). Not counted as an old-flow "feature" since it never functions for Course in practice.

### Excluded as belonging to the shared Talk wizard, not the Course wizard being retired

- Input & Config step (upload, title, category, translation settings, quiz-inclusion toggle, sector, audit metadata) — output-type-agnostic, shared verbatim with standalone Lesson creation; not course-specific.
- `SectionList.tsx` reorder/rename/delete UI — same component used for Lesson sections; only its *consequence* (final course-item order) is course-specific.
- Translate/Validate step UI (`TranslateStep.tsx`, `ValidateStep.tsx`) — zero `OutputType`/`Course` grep hits; identical for both.

## 5. New-flow additions (Part 4 — reverse check)

Features present in the new (compose-existing) flow that have no equivalent in the old flow — informational, for a "what's new" note:

1. **Published-only enforcement at the data layer.** Old flow has no concept of talk "status" gating course membership (it always authors brand-new `Published` talks by construction). New flow explicitly rejects Draft/ReadyForReview talks from being composed into a course, both in the picker UI (`AddTalksDialog.tsx:33`) and at the API layer (`CreateToolboxTalkCourseCommandHandler.cs:76-82`, `UpdateCourseItemsCommandHandler.cs:55-61`) — closing a real data-integrity hole the compose-existing-form-recon flagged as "not hypothetical."
2. **Per-item "Required" toggle**, exposed and editable in the UI (`CourseForm.tsx:174-182`). Old flow hardcodes every item as required.
3. **Reuse of already-vetted content.** Every course item in the new flow is, by construction, a talk that has already been through its own full authoring lifecycle (parse, quiz, settings, translate, validate, publish) via the standalone new wizard — meaning course items always carry real per-item validation coverage, real per-item translations, and a real admin-written description/category, none of which the old flow's freshly-split talks necessarily had at the same maturity level before course-level translate/validate ran once, in bulk, after the fact.
4. **URL-based smoke-testing override** (`?coursemode=new`/`?coursemode=old`), mirroring the wizard-toggle pattern (`useCoursePreference.ts:28-30`) — no equivalent existed for the old flow (it had no toggle to override, being the only path).
5. **Atomic, transactional-in-intent create.** `CreateToolboxTalkCourseCommandHandler` does the course + all items in one `SaveChangesAsync` call (`CreateToolboxTalkCourseCommandHandler.cs:112-113`) with no cross-entity intermediate state. Old flow's `PublishAsCourseAsync` explicitly has **no transaction wrapper** across its 396 lines and multiple sequential `SaveChangesAsync` calls (documented as a known risk in `course-in-new-wizard-impl-first-recon.md:80`) — new flow doesn't inherit that risk since it never runs multi-step publish logic.
6. **Course-list toggle-aware routing** — `CourseList.tsx`'s Create New button and `courses/new/page.tsx` both resolve the same `useCoursePreference()` gate, giving a single consistent entry-point decision instead of the old flow's implicit single-path assumption.

## 6. Adjacent surface notes (Part 5)

Spot-check only, not exhaustive.

- **Course assignment (`AssignCourseCommandHandler`) is fully shared and wizard-agnostic** — confirmed by direct read: it fans out into `ScheduledTalk` rows keyed off `ToolboxTalkCourseItem` regardless of which flow produced the course or its items. The `CourseComposeExistingHappyPathTests.cs` integration test exercises create → add-talks → assign → verify-operator-visibility end-to-end and passes against the real handler chain (test file confirmed present and current, `tests/QuantumBuild.Tests.Integration/ToolboxTalks/CourseComposeExistingHappyPathTests.cs:79-148`).
- **Employee/operator-facing course view is fully wizard-agnostic** — `web/src/app/(authenticated)/toolbox-talks/courses/[id]/page.tsx` reads exclusively from `useMyCourseAssignment` (`ToolboxTalkCourseAssignment` + `ScheduledTalks` DTOs) with zero reference to authoring path, `OutputType`, or any flow-specific field. Confirmed by full read — no parity gap here regardless of which flow created the course.
- **Not a parity gap, but worth flagging (pre-existing, affects both flows equally):** `AssignCourseCommandHandler` has no role-awareness — a Supervisor caller with `Learnings.Schedule` can assign a course to any employee in the tenant, not just their assigned operators, per `BACKLOG.md:820` (logged as a documented follow-up, §3.17, alongside the equivalent already-fixed gap on schedule creation). This affects courses created via either flow identically — it is not something the compose-existing cutover introduced or could fix by itself, but it sits directly in this feature's adjacent surface and is worth the same follow-up attention as §3.14 received.
- **Course-level validation display gap (see Part 4 row 8/15)** sits at the boundary between "course creation flow" and "adjacent surface" — flagged once, in the parity table, to avoid duplicate reporting here.
- No scheduling-side (`ToolboxTalkSchedule`/`ToolboxTalkScheduleAssignment`) interaction was found for either flow — course assignment has always bypassed the per-talk scheduling engine entirely, via its own one-shot fan-out (`AssignCourseCommandHandler.cs`), confirmed unchanged and identical for both authoring paths.

## 7. Recommendations per Category D item

| Item | Recommendation | Rationale |

## 8. Notes for customer communication

- Course creation is moving from "upload one document and let AI split it into a course" to "pick your already-published Learnings and arrange them into a course" — a simpler, faster mental model matching how most admins actually want to build a course (from content they've already vetted and published).
- **Existing legacy courses are voided at cutover**, per the already-communicated plan in `course-in-new-wizard-scoping-v2.md` — the trial customer should be told (as already planned) before the release, and any courses they've built under the old flow will need to be recreated by picking the same (already-published) Learnings in the new course builder, which takes only a few clicks since the underlying Learnings are unaffected.
- **Two known bugs in the old course flow are fixed automatically by this change**, not as separate work: quiz questions no longer get accidentally duplicated across course items, and course-level settings (certificate, refresher, auto-assign) now reliably save instead of silently reverting to "off."
- **New capability:** admins can now mark individual course items as optional (not every learning in a course has to be required for completion) — this did not exist before.
- **One minor known rough edge:** the course edit page's "Validation" tab will always show "no validation runs" for courses built the new way — this is expected, not a bug. Validation results live on each individual Learning's own page instead, where they were generated when that Learning was authored and published.
- The "AI splits one document into a whole course" capability is not gone forever — it's deliberately deferred, and would be rebuilt (if there's real demand) as a "generate several draft Learnings from one document, then compose them into a course" feature, keeping the two concerns (content generation vs. course composition) cleanly separate rather than fused as they were before.

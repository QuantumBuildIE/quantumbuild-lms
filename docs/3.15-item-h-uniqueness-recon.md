## 3. Schema Change Details

### 3a. Current index — confirmed

**File:** `src/Core/QuantumBuild.Core.Infrastructure/Data/Configurations/SupervisorAssignmentConfiguration.cs` (lines 13–15)

```csharp
builder.HasIndex(e => new { e.TenantId, e.SupervisorEmployeeId, e.OperatorEmployeeId })
    .IsUnique()
    .HasDatabaseName("IX_SupervisorAssignments_TenantId_SupervisorEmployeeId_OperatorEmployeeId");
```

- **Columns:** `(TenantId, SupervisorEmployeeId, OperatorEmployeeId)`
- **Unique:** yes
- **Filter clause:** none — covers both active and soft-deleted rows

### 3b. Proposed new index

- **Name:** `IX_SupervisorAssignments_TenantId_OperatorEmployeeId_Active`
- **Columns:** `(TenantId, OperatorEmployeeId)`
- **Filter:** `WHERE "IsDeleted" = false` (PostgreSQL partial index syntax)
- **Effect:** an operator can only appear once per tenant in active assignments — the single-supervisor constraint

EF Core configuration:

```csharp
builder.HasIndex(e => new { e.TenantId, e.OperatorEmployeeId })
    .IsUnique()
    .HasFilter("\"IsDeleted\" = false")
    .HasDatabaseName("IX_SupervisorAssignments_TenantId_OperatorEmployeeId_Active");
```

### 3c. Decision on the existing index — **Option A: keep it**

The existing `(TenantId, SupervisorEmployeeId, OperatorEmployeeId)` index (no filter) should be **retained**.

**Reasoning:**

The restore-on-reassign pattern in `AssignOperatorsAsync` (lines 142–149) queries soft-deleted rows via `IgnoreQueryFilters()` for the specific `(supervisor, operator)` pair before deciding whether to restore or insert. The existing no-filter index is the DB-level guard ensuring that at most one row (active or deleted) can exist for any `(TenantId, SupervisorEmployeeId, OperatorEmployeeId)` triple. Dropping it would leave restore-on-reassign relying entirely on application logic with no DB backstop against the rare case where two concurrent requests attempt to insert the same pair.

Option B (drop) would be acceptable if the application-layer check and partial index together fully covered the invariant, but the existing index is essentially free to keep and removes an entire class of potential concurrency gap.

## 5. Application-Layer Validation

### 5a. Where to add the check

**File:** `src/Core/QuantumBuild.Core.Application/Features/Employees/SupervisorAssignmentService.cs`  
**Method:** `AssignOperatorsAsync` (line 102)

Insert the new check after the self-assignment guard (line 131) and before the restore-on-reassign block (line 142). The loop over `dto.OperatorEmployeeIds` (lines 120–133) validates each operator — extend it:

```csharp
// Existing guards (lines 122-132) ...

// NEW: reject if operator is already actively supervised by a different supervisor
var currentAssignment = await _context.SupervisorAssignments
    .Where(sa => sa.OperatorEmployeeId == operatorId
              && sa.SupervisorEmployeeId != supervisorEmployeeId)
    .FirstOrDefaultAsync();

if (currentAssignment != null)
    return Result.Fail<List<SupervisorAssignmentDto>>(
        "This operator is already assigned to another supervisor. Unassign them first before reassigning.",
        FailureCode.Conflict);
```

The `HasQueryFilter` on `SupervisorAssignments` already filters to `!IsDeleted`, so this query only sees active assignments. No `IgnoreQueryFilters()` is needed.

### 5b. Error code

`FailureCode.Conflict` does not exist yet in `src/Core/QuantumBuild.Core.Application/Models/FailureCode.cs`. Add it:

public enum FailureCode

### 5c. Error message privacy

The error message does **not** name the current supervisor: "This operator is already assigned to another supervisor." This is intentional — exposing which supervisor currently holds the assignment may be inappropriate in multi-admin tenants (Admins can see all employees but may not need cross-team visibility).

### 5d. Race-condition backstop — DbUpdateException handling

If two concurrent `AssignOperatorsAsync` calls pass the application-layer check simultaneously, one will fail on the DB constraint with a `DbUpdateException` wrapping a `PostgresException` with `SqlState == "23505"`. The existing `catch (Exception ex)` at line 204 returns `Result.Fail($"Error assigning operators: {ex.Message}")`, which would expose Npgsql internals to the caller.

The implementation prompt should add a specific `catch (DbUpdateException ex) when (ex.InnerException is PostgresException pgEx && pgEx.SqlState == "23505")` handler before the general catch, returning the same user-facing `Conflict` message.

## 6. Impact on Existing Code Paths

### 6a. `AssignOperatorsAsync` — explicit check needed

As described in §5. This is the only method that creates or restores `SupervisorAssignment` rows. All assignment paths (My Team page and admin employee detail page) converge here.

### 6b. `UnassignOperatorAsync` — no change needed

Soft-delete only. Sets `IsDeleted = true`, which removes the row from the partial unique index's scope. No constraint impact.

### 6c. Restore-on-reassign under the new constraint

The existing restore-on-reassign logic (lines 142–149) is scoped to soft-deleted rows for `(supervisorEmployeeId, operatorId)`:

```csharp
var softDeletedAssignments = await _context.SupervisorAssignments
    .IgnoreQueryFilters()
    .Where(sa => sa.SupervisorEmployeeId == supervisorEmployeeId
              && sa.IsDeleted
              && dto.OperatorEmployeeIds.Contains(sa.OperatorEmployeeId))
    .ToListAsync();
```

With the new application-layer check firing first, the restore path is only reached when the operator has **no active assignment to any other supervisor**. The restore is therefore safe: restoring `IsDeleted=false` on a `(supervisorId, operatorId)` pair where no other active assignment exists will not violate the new partial unique index.

No change to the restore-on-reassign logic is required. The check ordering (application-layer conflict guard → restore path → new-row path) is correct by construction.

**Edge case documented:** Supervisor A previously unassigned Operator X. Supervisor B is now active on Operator X. Supervisor A tries to re-assign Operator X:
1. Application-layer check fires: finds B→X active, returns `Conflict`. Done.
2. Restore path never reached.

This is the correct behaviour.

## 7. Available-Operators Query Tightening (Recommended)

**Current behaviour:** `GetAvailableOperatorsAsync` excludes operators already assigned to *this supervisor*, but does not exclude operators actively assigned to *other supervisors*. A user can currently select an already-supervised operator from the dialog — they would only hit the error at submission time.

**Recommended change** in `GetAvailableOperatorsAsync` (line 68–72):

```csharp
// Change: remove the supervisor filter to exclude ALL actively-assigned operators
var assignedOperatorIds = await _context.SupervisorAssignments
    .Select(sa => sa.OperatorEmployeeId)
    .Distinct()
    .ToListAsync();
```

The `SupervisorAssignments` DbSet has the `!IsDeleted` query filter, so this query is already scoped to active assignments only. Operators assigned to this supervisor are already in the assigned list (not the available list), so removing the supervisor filter does not cause an operator to "disappear" from their current supervisor's team view.

**Effect:** the assign dialog only ever shows operators who are genuinely free to be assigned. The `Conflict` error becomes a rare concurrent-edit backstop rather than a first-line guard.

**File:** `src/Core/QuantumBuild.Core.Application/Features/Employees/SupervisorAssignmentService.cs`, `GetAvailableOperatorsAsync`, lines 68–72.

## 10. Risks and Edge Cases

### 10a. Production data violations at migration time

If Production has rows where `COUNT(*) > 1` for `(TenantId, OperatorEmployeeId)` with `IsDeleted = false`, the `CreateIndex` migration step will fail with a unique-constraint violation.

**Pre-deploy checklist item:** run the audit query from §2 against Production. If any violations exist:

1. Do not apply the migration
2. For each violating operator, retain the assignment with the most recent `UpdatedAt` and soft-delete the others (`IsDeleted = true`)
3. Re-run the query to confirm 0 rows
4. Proceed with migration

This cleanup requires a DBA script or a one-off admin action — it is not part of the migration itself.

### 10b. Privacy — naming the current supervisor

The error message does not name the current supervisor (see §5c). This is a deliberate choice. If a future product decision changes this, the service has access to the supervisor's employee record to look up their name and include it in the message.

### 10c. Race condition — two admins assigning the same operator concurrently

The application-layer check is not atomic. Two concurrent requests can both pass the check and then race to `SaveChangesAsync`. The second will fail with a `DbUpdateException` (PostgreSQL error 23505). The specific `DbUpdateException` catch in §5d catches this and returns the same `Conflict` error with a clean message. No data corruption occurs — the DB constraint is the authoritative guard.

### 10d. Operator with soft-deleted row to one supervisor + active assignment elsewhere

Scenario: Operator X was assigned to Supervisor A (soft-deleted) and is now active with Supervisor B. Admin tries to re-assign X to A. 

Result: application-layer check finds B→X active, returns `Conflict`. Soft-deleted A→X row is never restored. The restore-on-reassign is only reachable for operators with no active assignment anywhere else (see §6c).

### 10e. Supervisor is also an Operator elsewhere

The `GetExcludedEmployeeIdsAsync` method excludes employees whose linked User has Admin, Supervisor, or SuperUser roles from the assignment pool. A Supervisor cannot be assigned as an Operator. The single-supervisor constraint therefore only applies to employees with the Operator role — consistent with the constraint's intent.

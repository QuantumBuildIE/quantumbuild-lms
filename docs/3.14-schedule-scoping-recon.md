# §3.14 — Supervisor Schedule Scoping — Recon

**Date:** 2026-06-22
**Branch:** `transval`
**Status:** Read-only recon — no application files modified

---

## 1. One-line summary

`CreateToolboxTalkScheduleCommandHandler` (and its Update sibling) have no awareness of the caller's role — a Supervisor can schedule any tenant employee, not just their assigned operators. The fix requires adding role detection to `ICurrentUserService`, branching in both handlers, and constraining the schedule creation form in `ScheduleDialog.tsx`.

### 2.5 Role detection — ICurrentUserService gap

**File:** [ICurrentUserService.cs](src/Core/QuantumBuild.Core.Application/Interfaces/ICurrentUserService.cs)

The interface exposes `IsSuperUser` (bool), `HasPermission(string)`, `EmployeeId` (Guid?), `TenantId`, `UserId`, `UserName`. It does **not** expose `Roles`.

Detecting "Supervisor but not Admin/SuperUser" cannot be done from permissions alone because Admin also has `Learnings.Schedule`. The `SupervisorAssignmentService.CheckSupervisorAccessAsync` precedent (lines 279–318) queries `UserManager.Users` with `Include(UserRoles.Role)` — which works but is a DB round-trip in the Application layer.

**Recommended approach:** Add `IReadOnlyList<string> Roles { get; }` to `ICurrentUserService`, backed by reading `ClaimTypes.Role` JWT claims (the same way `IsSuperUser` reads the `is_super_user` claim). Role claims are already written into the token at login and in the test factory. This avoids a DB query and is consistent with how the service already works.

The Supervisor-only check then becomes:
```csharp
var isSupervisorOnly = !_currentUserService.IsSuperUser
    && !_currentUserService.Roles.Contains("Admin", StringComparer.OrdinalIgnoreCase)
    && _currentUserService.Roles.Contains("Supervisor", StringComparer.OrdinalIgnoreCase);
```

### 3.4 Frontend UX recommendation — Option B

**Option A** (filter client-side from full employee list): Simple, but leaks all employee names to the component even though the Supervisor cannot schedule them. Fragile if the tenant has many employees.

**Option B** (swap data source): When the user is Supervisor-only, call `useMyOperators()` instead of `useAllEmployees()`. `useMyOperators` returns only the supervisor's operators. This is clean, scoped at the data layer, and matches the backend's scoping.

**Recommendation: Option B.** The implementation in `ScheduleDialog`:
1. Detect Supervisor-only: `const isSupervisorOnly = user.roles.includes('Supervisor') && !user.roles.includes('Admin') && !user.isSuperUser`
2. Conditionally call either hook; map `SupervisorOperatorDto` to the same shape used for `Employee` in the picker
3. Hide the "Assign to all employees" checkbox entirely when `isSupervisorOnly`
4. Update the "Select employees" label to "Select team members" for Supervisors

**Label update:** Keep the change minimal — only the placeholder/heading inside the picker needs updating. The dialog title can stay "Schedule Training".

## 4. Role-resolution pattern (the precedent to follow)

**File:** [SupervisorAssignmentService.cs:279–318](src/Core/QuantumBuild.Core.Application/Features/Employees/SupervisorAssignmentService.cs#L279)

`CheckSupervisorAccessAsync` is the closest existing precedent. It checks `IsSuperUser` first, then loads the user from UserManager and inspects `UserRoles`. The §3.14 fix should NOT replicate this pattern in the handler for two reasons:

1. Querying `UserManager.Users` with `Include` inside a command handler that may be called frequently adds unnecessary DB load.
2. Role claims are already in the JWT; reading from `ICurrentUserService.Roles` (once the property is added) is a zero-DB-cost alternative.

**The §3.14 pattern (new precedent):**
```csharp
// In handler, after existing validation:
var isSupervisorOnly = !_currentUserService.IsSuperUser
    && !_currentUserService.Roles.Contains("Admin", StringComparer.OrdinalIgnoreCase)
    && _currentUserService.Roles.Contains("Supervisor", StringComparer.OrdinalIgnoreCase);

if (isSupervisorOnly)
{
    var supervisorEmployeeId = _currentUserService.EmployeeId;
    if (!supervisorEmployeeId.HasValue)
        throw new InvalidOperationException("Supervisor account is not linked to an employee record.");

    if (request.AssignToAllEmployees)
        throw new InvalidOperationException("Supervisors cannot schedule training for all employees. Please select your assigned team members.");

    var assignedIds = await _supervisorAssignmentService.GetAssignedOperatorIdsAsync(supervisorEmployeeId.Value);
    if (!assignedIds.Success)
        throw new InvalidOperationException("Could not verify supervisor assignments.");

    var unauthorised = request.EmployeeIds.Except(assignedIds.Data!).ToList();
    if (unauthorised.Any())
        throw new InvalidOperationException(
            $"{unauthorised.Count} selected employee(s) are not in your assigned team and cannot be scheduled.");
}
```

The count-based message (not listing IDs) is intentional — it avoids leaking employee IDs that the Supervisor shouldn't be able to enumerate.

## 7. Risks and edge cases

### User has both Supervisor and Admin roles
The role check `!Roles.Contains("Admin")` ensures users with both roles are treated as Admin (unrestricted). This is the correct product intent from §3.16.

### SuperUser scheduling
`ICurrentUserService.IsSuperUser` is checked first and bypasses the supervisor branch entirely. SuperUser sees no restriction.

### Supervisor with zero assigned operators
`GetAssignedOperatorIdsAsync` returns `Result.Ok([])` (empty list, not a failure). If a Supervisor with no assignments tries to schedule any employee: `unauthorised = request.EmployeeIds.Except([])` = all requested IDs → `unauthorised.Count > 0` → rejected with count message. If they try `AssignToAllEmployees=true`: blocked immediately. If they send `EmployeeIds = []`: the FluentValidator already rejects this (validator requires either `AssignToAllEmployees=true` or at least one `EmployeeId`). Edge case handled cleanly.

### `EmployeeId` is null for a Supervisor
Supervisors must have an employee record linked (that's how assignments work). If `ICurrentUserService.EmployeeId` is null for a Supervisor-role user, the handler throws `"Supervisor account is not linked to an employee record."` — a configuration error, not a permission error. This is an acceptable failure mode (it would surface a data integrity issue during setup).

### Devtools bypass on frontend
A Supervisor who manually sets `assignToAllEmployees=true` via browser devtools will hit the backend check and receive a 400. The frontend change is defence-in-depth UX, not the security boundary.

### Employee list caching (TanStack Query)
`useAllEmployees()` is cached by TanStack Query. When `ScheduleDialog` mounts with `useMyOperators()` instead, it fetches a different query key. No stale-data risk — the two data sources have separate cache entries.

### Supervisor removing themselves from the form after reassignment
If a Supervisor's team changes between opening the dialog and submitting, the backend check validates against the current live assignments. A stale frontend list causes a 400 from the backend naming the count of unauthorised employees — a clean error the user can address by reopening the dialog (which re-fetches operators).

### Course assignments (out of scope)
`AssignCourseCommandHandler.cs` also creates `ScheduledTalk` rows and accepts `EmployeeIds`. It has the same gap. Logged as a future BACKLOG item — not in §3.14 scope.

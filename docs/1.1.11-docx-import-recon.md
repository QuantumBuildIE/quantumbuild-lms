## 1. One-Line Summary

The fix is not as simple as BACKLOG suggests: `ExtractFromDocxAsync` exists in the **LessonParser** module
(a separate pipeline), not in the ToolboxTalks content-creation wizard. Wiring DOCX into the wizard
requires a new extraction service interface + implementation in ToolboxTalks, a new `InputMode.Docx`
enum value, a new `ContentSource.Docx` enum value, a NuGet dependency addition, and updates to both
frontend wizard InputConfigStep components. That is ~7 discrete changes, not a one-liner.

### 2.1 `ExtractFromDocxAsync` — Where It Actually Lives

**File:** `src/Modules/LessonParser/QuantumBuild.Modules.LessonParser.Infrastructure/Services/DocumentExtractorService.cs:99`

```csharp
public async Task<ExtractionResult> ExtractFromDocxAsync(
    Stream docxStream, string fileName, CancellationToken ct = default)
```

- Implemented via `DocumentFormat.OpenXml.Packaging.WordprocessingDocument` (OpenXML SDK v3.2)
- Walks all `Paragraph` descendants of the `Body`, assembles plain text
- Returns `ExtractionResult { Content, Title, CharacterCount, IsEmpty }`
- Minimum 50-character guard; throws `InvalidOperationException` on failure
- Registered in DI: `LessonParserInfrastructureExtensions.cs:44`

**Critical constraint:** `IDocumentExtractor` and `DocumentExtractorService` live in the LessonParser
module. ToolboxTalks must NOT take a dependency on LessonParser (dependency direction is
LessonParser → ToolboxTalks.Application, not the reverse). The OpenXML implementation must be
duplicated or wrapped in a new ToolboxTalks-owned service.

**`DocumentFormat.OpenXml` is not a current dependency of ToolboxTalks.Infrastructure:**
`src/Modules/ToolboxTalks/.../QuantumBuild.Modules.ToolboxTalks.Infrastructure.csproj` — no OpenXml
package reference. It must be added.

### 2.5 New Service Required: `IDocxExtractionService`

The ToolboxTalks Application layer already has `IPdfExtractionService` at:
`src/.../Abstractions/Pdf/IPdfExtractionService.cs`


```csharp
public interface IDocxExtractionService
{
    Task<DocxExtractionResult> ExtractTextFromUrlAsync(
        string docxUrl, CancellationToken ct = default);
}

public record DocxExtractionResult(
    bool Success,
    string? Text,
    string? ErrorMessage);
```

Implementation lives in ToolboxTalks.Infrastructure. It:
1. Downloads the file via `HttpClient.GetStreamAsync(url)` (same pattern as `PdfExtractionService`)
2. Calls `WordprocessingDocument.Open(stream, false)` and walks paragraphs
3. Returns `DocxExtractionResult` (no page count, unlike PDF)
4. Registered in DI in `ServiceCollectionExtensions.cs`

### 2.7 R2 Upload Path — No Changes Required

The presigned-URL upload mechanism at `POST /api/toolbox-talks/learning-wizard/upload-source-url`
(ToolboxTalksController.cs:390) is content-type agnostic:

```csharp
var key = $"uploads/{tenantId}/{Guid.NewGuid():N}/{request.FileName}";
var presignedUrl = await _r2StorageService.GenerateUploadUrlAsync(key, request.ContentType, ...);
```

A DOCX file uploaded with `contentType = "application/vnd.openxmlformats-officedocument.wordprocessingml.document"`
stores at `uploads/{tenantId}/{guid}/filename.docx` and is publicly accessible at the returned
`publicUrl`. No backend changes to the upload endpoint.

### 2.8 File Size Limit

Backend controller (`ToolboxTalkFilesController.cs`) defines:
```csharp

```
DOCX files are typically much smaller than their PDF equivalents; 50MB is a safe limit for both.
No separate limit required — document this explicitly to frontend.

## 5. Risks and Edge Cases

### 5.1 Slideshow Generation — Hard Blocker for DOCX

**Risk level: HIGH.** The slideshow feature (`GenerateSlidesFromPdf`) converts PDF pages to images and
renders them as slides. It is mechanically impossible for DOCX source files because there are no page
images available.

The wizard's Settings step (step 4) exposes a "Generate slides from document" toggle. If a user
selects DOCX mode and then enables that toggle, `ContentGenerationJob` would attempt to run slideshow
generation against a DOCX source — it will fail or silently skip.

**Required mitigation:** In the Settings step, hide or disable the "Generate slides from document"
toggle when `InputMode === 'Docx'`. The backend `ContentGenerationJob` already guards on
`GenerateSlidesFromPdf` being true, but the UI should also make it obvious that slideshows are not
available for Word documents.

### 5.2 Password-Protected DOCX

The OpenXML SDK throws `FileFormatException` (subclass of `IOException`) when opening a
password-protected DOCX. `DocumentExtractorService.ExtractFromDocxAsync` wraps non-`InvalidOperationException`
exceptions into a generic `InvalidOperationException`. The service implementation to be written for
ToolboxTalks should explicitly catch and surface a user-friendly message ("The Word document is
password-protected. Please remove the password before uploading.") rather than a generic failure.

Precedent: `ContentExtractionService.GetUserFriendlyPdfError()` at line 810 already does this for
PDF password errors.

### 5.3 `.doc` Legacy Format (Pre-2007 Word)

`.doc` files are **not** supported by the OpenXML SDK. Users will receive a cryptic failure if they
upload a `.doc` file. The file input `accept` attribute must be `.docx` only (not `.doc`). The UI
label should say "Word Document (.docx)" not "Word Document" to set that expectation.

### 5.4 DOCX with Embedded Images, Charts, or Tables

`DocumentExtractorService.ExtractFromDocxAsync` walks only `Paragraph` descendants. Content in:
- Embedded images → silently dropped (no alt text extracted)
- Charts → silently dropped
- Tables → table cell text is extracted as flat paragraphs; structure lost

For safety-training documents (mostly prose), this is acceptable. Document it as a known limitation
in UI hint text ("Best results with text-based Word documents; tables are extracted as flat text,
images are skipped").

### 5.5 MIME Type Ambiguity in Browser File Picker

Some browsers and OS configurations report `.docx` files with MIME type
`application/octet-stream` rather than the correct
`application/vnd.openxmlformats-officedocument.wordprocessingml.document`. The frontend validation
in `handleFileSelected`/`handleFileSelect` checks `file.type`. If validation blocks on MIME type
alone, `.docx` files from those environments will be rejected.

**Recommended fix:** In the DOCX validation branch, accept the file if `file.type` is the correct
MIME type **or** if the filename extension is `.docx` (case-insensitive). The PDF branch already
uses MIME-only validation (acceptable there because `application/pdf` is universally consistent).

### 5.6 File Size — DOCX vs PDF

DOCX files are generally smaller than PDFs for equivalent text content. The existing 50MB limit is
well above what any SOP document would reach. No change required, but note that the backend
`[RequestSizeLimit]` attribute on `UploadPdf` (50MB) is on the **old** upload endpoint
(`ToolboxTalkFilesController.cs:128`), which is not used by the new wizard. The new wizard uses the
presigned-URL path, which has no backend size check — the frontend check is the only enforcement.
This is the same gap that exists for PDF in the new wizard today.

### 5.7 `ContentSource.Docx` — Downstream Display

Adding `ContentSource.Docx = 5` requires a search for every `switch` or `if` on `ContentSource`
that might miss the new value. The most visible site is `MapToDto` in
`ParseToolboxTalkContentCommandHandler.cs:244-249`:

```csharp
SourceDisplay = s.Source switch
{
    ContentSource.Manual => "Manual",
    ContentSource.Video  => "Video",
    ContentSource.Pdf    => "PDF",
    _                    => s.Source.ToString()     // Docx falls here → "Docx" (acceptable default)

```

The `_` default means this won't break, but the displayed string will be "Docx" (enum name) rather
than "Word". Add an explicit case `ContentSource.Docx => "Word"` for polish.

## 7. What The BACKLOG Got Right and Wrong

The BACKLOG said: *"mostly-wiring: `ExtractFromDocxAsync` already exists; the wizard just needs the
rails extended."*

**Right:** The core extraction logic exists and is battle-tested. The R2 upload path is already
content-type agnostic. The downstream `ContentParserService` just needs a hint string.

**Wrong:** `ExtractFromDocxAsync` is in a **different module** (LessonParser). It cannot be
referenced from ToolboxTalks without reversing the module dependency direction. The extraction logic
must be re-implemented (or wrapped) in a new ToolboxTalks-owned service — about 60 lines including
DI registration. The BACKLOG also predates the §5.27 wizard split, so it names only one
InputConfigStep; two exist.

The total work is ~7 discrete changes across ~10 files, plus one test stub. Compact but not trivial.

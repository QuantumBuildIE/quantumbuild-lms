# Scheduled Refresh Flow — Recon

Date: 2026-07-08
Scope: read-only recon, no code changes. Verified against code at HEAD on `transval`.

## Headline

**Refresh flow appears functional end-to-end for the Monthly/Quarterly/Annually cases — recommend a verification walk covering those three intervals, the reminder emails, and two confirmed edge-case landmines (see "Gaps found").** There is no scanning "refresh job" as the task brief assumed — refresher scheduling is triggered synchronously at talk/course completion, not by a background scan for "who's due." This is a materially different architecture than BACKLOG's phrasing implies, and it matters for how you'd test it (see verification walk).

## 2. The schedule-processing job — clarifying what actually exists

There is **no job that scans "which talks/courses need a refresh cycle."** The refresh-creation trigger is **completion-time, not schedule-scan-time**:

- `CompleteToolboxTalkCommandHandler.Handle` calls `_refresherSchedulingService.ScheduleRefresherIfRequired(scheduledTalk, ...)` immediately after marking a **standalone** (non-course) `ScheduledTalk` as `Completed` — [CompleteToolboxTalkCommandHandler.cs:204-207](../src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Application/Commands/CompleteToolboxTalk/CompleteToolboxTalkCommandHandler.cs#L204).
- `CourseProgressService.UpdateProgressAsync` calls the course-assignment overload when a course assignment transitions to `Completed` — [CourseProgressService.cs:65-67](../src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Application/Services/CourseProgressService.cs#L65).
- Both run inline, synchronously, in the same request/transaction as the completion — not via Hangfire, not via a nightly scan.

The Hangfire jobs that *do* exist around scheduling are for the **initial** assignment pipeline and for **reminders on already-created rows**, not for generating refresh cycles:

| Job | Cron (Ireland TZ unless noted) | Role in refresh flow |
|---|---|---|
| `ProcessToolboxTalkSchedulesJob` | `30 6 * * *` | Creates `ScheduledTalk` rows from `ToolboxTalkSchedule` recurrence config — this is the **initial/ad-hoc scheduling** engine, shared code but a distinct concept from refreshers (out of scope per brief, noted only). |
| `UpdateOverdueToolboxTalksJob` | `0 * * * *`, UTC | Marks any `Pending`/`InProgress` `ScheduledTalk` (refresher or not) `Overdue` once `DueDate` passes — [UpdateOverdueToolboxTalksJob.cs:42-50](../src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Infrastructure/Jobs/UpdateOverdueToolboxTalksJob.cs#L42). |
| `SendToolboxTalkRemindersJob` | `0 8 * * *` | Generic overdue-reminder job — applies to refresher rows too since it has no `IsRefresher` filter, it just finds any overdue-and-under-max-reminders row. |
| `SendRefresherRemindersJob` | `0 9 * * *` | The refresher-specific 2-week/1-week pre-due reminder job — see §4. |

Registered in `Program.cs:443-464` via `IRecurringJobManager.AddOrUpdate<T>(...)` with the concrete job class as the type parameter (correctly following Note 21's "enqueue via concrete class" rule).

## 3. What determines "this operator is due for a refresh"

Entirely computed at completion time, not evaluated later. In `RefresherSchedulingService.ScheduleRefresherIfRequired` ([RefresherSchedulingService.cs](../src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Application/Services/RefresherSchedulingService.cs)):

1. Look up the `ToolboxTalk`; bail if `!talk.RequiresRefresher` (line 19).
2. **Idempotency guard** — if a non-completed, non-cancelled `ScheduledTalk` already exists for `{EmployeeId, ToolboxTalkId, IsRefresher=true}`, skip (lines 22-36). This prevents duplicate refreshers if completion logic is somehow invoked twice.
3. `completedAt = completedTalk.Completion?.CompletedAt ?? DateTime.UtcNow` — **the driving date is the completion timestamp of the specific `ScheduledTalk` instance that was just completed**, not a lookback across all historical completions for that employee/talk pair (line 39). If an employee has completed the same talk multiple times (e.g. once ad-hoc, once as a refresher), only the completion that just happened drives the next interval — there is no "find the latest completion" query.
4. `refresherDueDate = completedAt.AddMonths(talk.RefresherIntervalMonths)` (line 40) — pure `DateTime.AddMonths` on a UTC-kind timestamp.
5. New `ScheduledTalk` created with `RequiredDate = refresherDueDate.AddDays(-14)` (available 2 weeks early), `DueDate = refresherDueDate`, `Status = Pending`, `IsRefresher = true`, `OriginalScheduledTalkId` linking back, `RefresherDueDate` mirroring `DueDate`.

Course path (`ScheduleRefresherIfRequired(ToolboxTalkCourseAssignment)`) is structurally identical: same idempotency guard, same `AddMonths` computation off `completedAssignment.CompletedAt`, then creates a refresher `ToolboxTalkCourseAssignment` **plus** one `ScheduledTalk` per course item (lines 108-126).

**Weekly note confirmed:** `RefresherFrequencyMapper.ToCanonicalFields` maps `ToolboxTalkFrequency.Weekly → (RequiresRefresher: false, ...)` with an explicit comment: "Weekly has no months equivalent and was never functional for refresher scheduling (RefresherSchedulingService uses integer months)." This is real and current — `RefresherSchedulingService` has no weekly/day-based path at all, only `AddMonths`. Any `ToolboxTalk` row still carrying legacy `Frequency = Weekly` in the database, if it ever passes back through `ToCanonicalFields` (e.g. an old-wizard edit save), will silently flip to `RequiresRefresher = false` — no exception, no warning logged.

## 4. Notification mechanism after the new ScheduledTalk row is created

**There is no "you've just been scheduled for a refresher" notification at creation time.** `RefresherSchedulingService` has no `IToolboxTalkEmailService` (or any notification) dependency at all — it only touches the DbContext. Compare to the **initial**-assignment path, where `ProcessToolboxTalkScheduleCommandHandler` explicitly calls `_emailService.SendTalkAssignmentEmailAsync(...)` right after creating the `ScheduledTalk` ([ProcessToolboxTalkScheduleCommandHandler.cs:137](../src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Application/Commands/ProcessToolboxTalkSchedule/ProcessToolboxTalkScheduleCommandHandler.cs#L137)) — and `AssignCourseCommandHandler` does the course equivalent (`SendCourseAssignmentEmailAsync`, line 193). Refreshers get no analogous immediate email.

What refreshers *do* get is the pre-due reminder cadence from `SendRefresherRemindersJob` (email only, no in-app/SignalR channel found):

- **2-week reminder**: fires when `RefresherDueDate <= now+14d AND RefresherDueDate > now+7d AND !ReminderSent2Weeks`, calls `SendRefresherReminderAsync(talk, employee, "2 weeks", ...)`, sets `ReminderSent2Weeks = true` ([SendRefresherRemindersJob.cs:54-82](../src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Infrastructure/Jobs/SendRefresherRemindersJob.cs#L54)).
- **1-week reminder**: same pattern with `<= now+7d`, sets `ReminderSent1Week` (lines 85-112).
- Same two-tier logic duplicated for `ToolboxTalkCourseAssignment` refreshers (lines 114-171).
- Not gated on any tenant setting that I found — no `ToolboxTalkSettings` flag checked in this job (contrast with `SendToolboxTalkRemindersJob`, which does read `settings.MaxReminders` / `settings.EscalateAfterReminders`).
- Because `RequiredDate = DueDate - 14 days`, a refresher becomes visible/actionable in the employee portal at essentially the same moment the 2-week reminder email would fire — the two are not independently tunable.
- Once overdue, the refresher also flows into the **generic** `SendToolboxTalkRemindersJob` (no `IsRefresher` distinction there), so an overdue refresher gets the same escalating reminder cadence as any other overdue assignment, up to tenant `MaxReminders`.

**Escalation to managers is stubbed out** — `SendToolboxTalkRemindersJob.cs:120-142` has a fully-commented-out escalation block with the note "manager escalation is not yet implemented" pending an `Employee.ManagerId` field that doesn't exist yet. This applies to overdue refreshers exactly as it does to overdue initial assignments.

## 5. Operator portal visibility

`GetMyToolboxTalksQueryHandler` ([GetMyToolboxTalksQueryHandler.cs](../src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Application/Queries/GetMyToolboxTalks/GetMyToolboxTalksQueryHandler.cs)) is the query behind `/pending`, `/in-progress`, `/overdue`, `/completed`. It:

- Filters only on `TenantId`, `EmployeeId`, `!IsDeleted`, `Status != Cancelled`, and the requested status (lines 24-34) — **no filter on `ToolboxTalk.IsActive`**, consistent with §5.30's note that the field is functionally decorative on this path.
- Orders `Completed` last, then by `DueDate` ascending (lines 41-42) — **a refresher is not sorted any differently from an initial assignment**; it sits wherever its due date puts it.
- Projects `IsRefresher` and `RefresherDueDate` into `MyToolboxTalkListDto` (lines 65-66), which the frontend uses to render an orange "Refresher" badge with a `RefreshCw` icon on `TalkCard.tsx` ([TalkCard.tsx:83-88](../web/src/features/toolbox-talks/components/TalkCard.tsx#L83)) — so a refresher **is** visually distinguishable from an initial assignment once rendered, just not specially ordered.

## Edge case findings

| Edge case | What the code does today |
|---|---|
| **Operator inactive (`Employee.IsActive = false`) when refresh would fire** | No check anywhere. `RefresherSchedulingService` never reads `Employee.IsActive`. `SendRefresherRemindersJob` and `SendToolboxTalkRemindersJob` don't filter on it either — `IsActive` is not part of any EF query filter (only `IsDeleted` is — [ApplicationDbContext.cs:320](../src/Core/QuantumBuild.Core.Infrastructure/Data/ApplicationDbContext.cs#L320)). An inactive-but-not-deleted employee gets the refresher created and the reminder emails sent exactly as an active employee would. |
| **Operator has left the tenant (soft-deleted `Employee`)** | `Employee` has a global query filter `!IsDeleted && (BypassTenantFilter \|\| TenantId == TenantId)`. Both reminder jobs call `.IgnoreQueryFilters()` on the root `ScheduledTalks`/`ToolboxTalkCourseAssignments` query and then `.Include(st => st.Employee)`. In EF Core (this stack is EF Core 9), `IgnoreQueryFilters()` suppresses filters for **every** entity type touched by the query, including included navigations — so a soft-deleted employee's row is **not** excluded, and `talk.Employee` will be the departed employee's record (with their last-known email), not null. Net effect: refresher reminder emails can go out to employees who have been offboarded, as long as their `ScheduledTalk`/`ToolboxTalkCourseAssignment` row itself isn't deleted (and nothing in this flow deletes those rows on employee departure). This is a genuine, verifiable gap, not a hypothetical — worth a real test. |
| **Operator has completed the same talk multiple times — which completion drives the interval?** | Whichever `ScheduledTalk.Completion.CompletedAt` belongs to the specific instance that was just marked `Completed` in that request. There's no query that looks up "the employee's most recent completion of this talk" — the interval is always `(this completion's timestamp) + RefresherIntervalMonths`. In the normal lifecycle (refresher created → employee completes refresher → next refresher created from that completion) this is correct by construction. It could only misbehave if two `ScheduledTalk` rows for the same employee/talk pair could be completed out of order or exist as duplicates — but the idempotency guard (step 2 in §3) prevents a second pending refresher from being created while one is already outstanding, which limits that risk. |
| **DST / month-boundary computation** | `DateTime.AddMonths` on a `DateTimeKind.Utc` value is pure calendar arithmetic — no timezone conversion happens, so there's no DST-shift bug in the due-date computation itself. The real (minor) risk is .NET's standard end-of-month clamping: a completion on Jan 31 with a 1-month interval yields Feb 28 (or 29), not "Mar 3." This is standard `AddMonths` behavior, not a bug, but worth knowing if a tester expects exact day-of-month stability across multi-year refresher chains. Separately, the Hangfire recurring jobs (`ProcessToolboxTalkSchedulesJob`, `SendToolboxTalkRemindersJob`, `SendRefresherRemindersJob`) are pinned to `TimeZoneInfo.FindSystemTimeZoneById("GMT Standard Time")` (Ireland, which does observe DST) via `RecurringJobOptions.TimeZone` — Hangfire's recurring job scheduler is DST-aware for this pattern, so this is a library-handled concern rather than a homegrown one. `UpdateOverdueToolboxTalksJob` runs on plain UTC hourly cron with no `TimeZone` override, which is fine for an hourly job. |
| **Talk deactivated (`ToolboxTalk.IsActive = false`)** | Confirmed decorative on this path exactly as §5.30 flagged: `GetMyToolboxTalksQueryHandler` never checks `IsActive`, so a previously-scheduled refresher for a now-deactivated talk still surfaces, is still completable, and (since `RefresherSchedulingService` also never checks `IsActive`) will still spawn its *next* refresher on completion. Deactivating a talk does not stop its refresher cycle. |
| **Admin changes frequency after some refreshers have already fired** | Editing `RequiresRefresher`/`RefresherIntervalMonths` on the `ToolboxTalk` only affects future computations — `ScheduledTalk.RefresherDueDate` is stamped at creation time and is never recalculated retroactively, so an already-pending refresher keeps its original due date even if the admin changes the interval afterward. Only the *next* refresher (created at the next completion) picks up the new interval. This is sane, expected behavior — flagging it only so a tester doesn't mistake it for a bug when a mid-cycle interval change doesn't move an already-scheduled due date. The genuine landmine here is the legacy-Weekly interaction from §3: if an old-wizard edit round-trips a talk whose `Frequency` is still `Weekly`, the mapper silently sets `RequiresRefresher = false`, silently killing that talk's refresher cycle with no error surfaced to the admin. |

## Recommended verification walk

1. **Baseline setup**: one talk with `RequiresRefresher = true`, `RefresherIntervalMonths = 1` (fastest to observe), assigned standalone (not via course) to a test employee.
2. **Completion → refresher creation**: complete the talk as that employee. Confirm in the DB (or via `/api/my/toolbox-talks/{id}` or the admin scheduled-talks list) that a new `ScheduledTalk` row exists with `IsRefresher = true`, `OriginalScheduledTalkId` pointing at the just-completed row, `DueDate` ≈ completion + 1 month, `RequiredDate` ≈ `DueDate` − 14 days.
3. **Idempotency**: attempt to trigger a second refresher for the same employee/talk while the first is still pending (e.g. by re-invoking completion logic in a test harness) — confirm no duplicate row is created (existing-refresher guard).
4. **Portal visibility & badge**: confirm the new refresher entry appears in the employee's My Learnings pending list once `RequiredDate` passes, with the orange "Refresher" badge rendered, and that its position in the list is governed by `DueDate` only (no special "refresher-first" ordering).
5. **2-week / 1-week reminder emails**: with a `RefresherDueDate` set to exactly 13 days and exactly 6 days out (to land inside each window), manually trigger `SendRefresherRemindersJob` (or wait for the 9:00 AM Ireland-time cron) and confirm the two emails fire once each and `ReminderSent2Weeks`/`ReminderSent1Week` flip — then confirm re-running the job does **not** resend either.
6. **Overdue handoff**: let `DueDate` pass, confirm `UpdateOverdueToolboxTalksJob` flips status to `Overdue` within the hour, and confirm `SendToolboxTalkRemindersJob` picks up the row and increments `RemindersSent` on its normal daily cadence (this job doesn't distinguish refreshers, so this should behave identically to an overdue initial assignment).
7. **Inactive/departed employee check (the one most likely to surprise someone)**: create a refresher for an employee, then either set `Employee.IsActive = false` or soft-delete the employee, then run `SendRefresherRemindersJob` manually and check whether the reminder email actually goes out to that employee's address. This directly tests the gap in the Edge Case table above — expect it currently **does** send.
8. **Course refresher smoke check** (light touch, since out of scope): complete a course with `RequiresRefresher = true` and confirm a refresher `ToolboxTalkCourseAssignment` plus per-item `ScheduledTalk` rows are created, mirroring the standalone flow.
9. **Weekly legacy landmine**: if any pre-existing talk in a lower environment still has `Frequency = Weekly` in the DB, open it in the old `ToolboxTalkForm.tsx` edit form, save without changing frequency, and confirm whether `RequiresRefresher` gets silently flipped to `false` on save (tests the `RefresherFrequencyMapper.ToCanonicalFields` Weekly branch in a live round-trip, not just by reading the mapper code).

## Gaps found

- **No immediate notification when a refresher is scheduled** — by design or by omission is unclear from the code alone (no comment addressing why this differs from the initial-assignment path), but it is a real behavioral gap relative to what the boss is likely picturing ("operator gets notified and the learning gets scheduled again"). The operator currently finds out either by noticing the entry in their portal once `RequiredDate` passes, or via the 2-week reminder email — there is no "you have a new refresher" email at the moment of creation.
- **Reminder emails are not gated by employee active/deleted status** — confirmed via `IgnoreQueryFilters()` + `Include(Employee)` interaction in both reminder jobs. This is the most concrete, testable gap in the whole flow (see verification step 7) and the one I'd prioritize confirming before telling the boss "it works."
- **Legacy `Frequency = Weekly` + old edit form round-trip can silently disable refreshers** — no exception, no admin-facing warning. Low real-world frequency (requires stale Weekly data plus a save through the legacy form) but silent-data-loss-of-configuration is worth knowing about.
- **Manager escalation for overdue refreshers is not implemented** — stubbed out with a `TODO` pending `Employee.ManagerId`. Applies to overdue initial assignments too, not refresher-specific, but relevant if "operator gets notified" in the original ask was assumed to include escalation.

## Notes for the boss

The core mechanic works the way you'd expect for the common case: when someone completes a talk configured to refresh monthly/quarterly/annually, the system automatically creates their next refresher due date the moment they complete it, and they'll get reminder emails 2 weeks and 1 week before it's due, plus the same overdue-reminder emails everyone else gets if they miss it. Two things are worth knowing before we call this fully verified. First, there's no "hey, you've got a new refresher scheduled" email at the moment it's created — the employee only hears about it via the 2-week reminder or by noticing it in their portal, which may or may not match what you had in mind by "gets notified." Second, and more important: the reminder emails don't currently check whether an employee has left the company or been marked inactive, so if someone completes a talk and then leaves before their next refresher comes due, the system may still try to email them. Neither of these is a crash or data-loss bug — the scheduling math itself (due dates, intervals, avoiding duplicate refreshers) checks out — but I'd want to confirm the departed-employee email behavior in a real test before we say this is production-solid, since it's the one piece I can't fully confirm without running it. Recommend we run the verification steps above and fix the two gaps if they land the way the code suggests.

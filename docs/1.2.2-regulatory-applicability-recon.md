# §1.2.2 — Regulatory Applicability Surface Recon

**Date:** 2026-06-23  
**Status:** Read-only recon — no application files modified  
**Branch:** `transval`

---

## 1. One-Line Summary

The backend carries enough information to detect "no approved requirements" but does not expose it on either the translate trigger path or the score history DTO; the results screen conflates "not scored yet", "no sector configured", and "sector configured but requirements in Draft" into a single silent button state; the fix requires one new lightweight endpoint, one DTO extension, one translate-step pre-flight banner, and three new result-screen states.

**EF Configurations:**

### Critical Distinction: `RegulatoryCriteria` vs `RegulatoryRequirement`

These are **different things** used in different workflows:

| Entity | Role | Used By |
|---|---|---|
| `RegulatoryCriteria` | Assessment rubric (scoring weights, criteria text) | `RegulatoryScoreService` — fed into Claude prompt |
| `RegulatoryRequirement` | Compliance checklist items (individual obligations) | `RequirementMappingJob`, compliance browse, pending mappings UI |

This distinction drives the entire analysis: **regulatory scoring uses `RegulatoryCriteria`, not `RegulatoryRequirement`**. A tenant can receive a regulatory score even when all requirements are Draft. The Draft/Approved distinction only gates the compliance-tracking workflow, not the scoring itself.

### Tenant → Profile Relationship

A tenant "has" regulatory profiles via their assigned sectors (`TenantSector` → `Sector.Key` → `RegulatoryProfile.SectorKey`). A tenant can have multiple profiles (one per document-sector pair across their assigned sectors). Visibility of requirements is controlled by `GetBrowsableRequirementsAsync` which filters `IngestionStatus == Approved` and tenant's sector keys:

```csharp
// RequirementIngestionService.cs:261
var requirements = await _dbContext.RegulatoryRequirements
    .IgnoreQueryFilters()
    .Where(r => !r.IsDeleted
        && r.IngestionStatus == RequirementIngestionStatus.Approved
        && tenantSectorKeys.Contains(r.RegulatoryProfile.SectorKey))
    // ...
```

## 3. Translation Flow — When Regulatory Validation Runs

### Translation validation pipeline

```
User clicks "Start Validation"
  → POST /api/toolbox-talks/{talkId}/validation/validate
  → TranslationValidationController.StartValidation()
      - Creates TranslationValidationRun (SectorKey stored from request, can be null)
      - Enqueues TranslationValidationJob via Hangfire
      - Returns 202 with RunId
  → TranslationValidationJob.ExecuteAsync()
      - Back-translates each section (Claude Haiku, DeepL, etc.)
      - Runs consensus engine (multi-round)
      - Persists TranslationValidationResult per section
      - Sends SignalR events (SectionCompleted, ValidationComplete)
```

**Regulatory scoring is NOT part of this pipeline.** It is a completely separate, manually triggered step.

### Regulatory scoring pipeline (separate)

```
User clicks "Score [type]" button on results screen
  → POST /api/toolbox-talks/validation-runs/{runId}/regulatory-score
  → RegulatoryScoreController.Score()
      - Pre-flight: verifies run exists, has completed sections
      - For RegulatoryTranslation: verifies SectorKey is set on run
      - Calls RegulatoryScoreService.ScoreAsync(runId, scoreType)
          - Loads run + completed section results
          - Loads RegulatoryProfile by SectorKey (null-safe)
          - Loads RegulatoryCriteria (assessment rubric)
          - Calls appropriate Claude Sonnet scoring prompt
          - Persists ValidationRegulatoryScore
          - Returns RegulatoryScoreResultDto
```

### Pre-flight check for approved requirements?

**No.** Neither the translate endpoint nor the scoring endpoint checks whether the tenant has any `Approved` requirements for the sector. The `StartValidation` controller action (`TranslationValidationController.cs:51`) verifies only: talk exists, language code provided, translation exists for that language. The `Score` controller (`RegulatoryScoreController.cs:50`) verifies only: run exists and belongs to tenant, completed sections exist, sector key set (for `RegulatoryTranslation` type only).

### `RegulatoryScorePanel.tsx` (complete file at `web/src/features/toolbox-talks/components/RegulatoryScorePanel.tsx`)

The panel renders **three columns** (Source Document, Pure Translation, Regulatory Translation) side by side.

**Current state logic per column:**

| Condition | Rendering |
|---|---|
| `sectorKey` is null | Amber warning: "No sector configured for this run. Select a sector to enable regulatory scoring." — all three buttons remain active except Regulatory Translation which is disabled |
| `sectorKey` set, no scores yet | Button visible; enabled (except Regulatory Translation disabled if sector null — already handled) |
| Score exists | Score card with overall score, verdict, category breakdown, summary |
| `isApproved` (verdict contains "APPROVED FOR DISTRIBUTION") | Regulatory Translation button is also disabled (success lock) |

**What the column `isDisabled` check looks like (line 212):**
```typescript
const isDisabled =
  isScoring ||
  !hasCompletedSections ||
  (col.scoreType === 'RegulatoryTranslation' && !sectorKey) ||
  (col.scoreType === 'RegulatoryTranslation' && isApproved);
```

There is **no gate on whether approved requirements exist**. A tenant with a sector key set but zero Approved requirements sees the same enabled button as a tenant with 50 approved requirements.

## 5. Backend DTO State — What the API Returns Today

### Run detail DTO

`ValidationRunDetailDto` (carried by `GET /api/toolbox-talks/{talkId}/validation/runs/{runId}`) has no regulatory score fields at all. Regulatory data is fetched separately via `GET /api/toolbox-talks/validation-runs/{runId}/regulatory-score/history`.

### Score history DTO

`RegulatoryScoreHistoryDto` returned by the history endpoint contains only existing scores and no signal about whether requirements exist. `SourceScore == null` means "never scored" but cannot mean "not applicable".

### Existing applicability surface

`GET /api/regulatory/browse` (`RegulatoryBrowseController.cs:1`) returns the full requirement hierarchy filtered to Approved-only for the tenant's sectors. It can serve as an applicability check (empty result = no Approved requirements) but is heavyweight and sector-scoped to the tenant's assigned sectors, not a per-sector check for a specific validation run's sector.

No dedicated "do I have approved requirements for sector X?" endpoint exists.

## 6. Current UX — Four Scenarios Traced

### Scenario A: Tenant has no sector configured on the run

**Path:** User starts validation without selecting a sector (sector field left blank).

- Translate fires, validation runs, `TranslationValidationRun.SectorKey = null`
- Results screen: `RegulatoryScorePanel` shows amber warning "No sector configured" at header level
- Regulatory Translation button: disabled with tooltip "No sector configured — cannot run regulatory scoring"
- Source Document and Pure Translation buttons: enabled and usable
- Compliance browse: empty (no sector to match)

**Current experience:** Correctly blocked. This scenario is adequately handled.

### Scenario B: Tenant has sector set on run, but no `RegulatoryProfile` for that sector

**Path:** User picks sector "healthcare" but no `RegulatoryProfile` with `SectorKey == "healthcare"` exists in the system.

- Validation runs normally
- Scoring call: `RegulatoryScoreService.ScoreAsync` tries `FirstOrDefaultAsync(p => p.SectorKey == "healthcare")` → returns null → `profile = null`
- Scoring continues: `ParseCategoryWeights(null)` → empty list; Claude receives empty criteria → scores on general knowledge
- **`ValidationRegulatoryScore` row is created** with `RegulatoryProfileId = null`, `OverallScore` from Claude, verdicts may appear meaningful
- No amber warning; buttons appear enabled and functional
- Compliance browse: returns empty for this sector

**Current experience:** Misleading. Score appears but criteria were empty. In practice unlikely (profiles are system-seeded for all active sectors), but the gap is structural.

### Scenario C: Tenant has sector, profile exists, requirements all Draft (the primary gap)

**Path:** Admin ingested a regulatory document but hasn't approved any requirements yet. Tenant picks that sector for their translation.

- Validation runs normally
- Scoring call: `RegulatoryScoreService.ScoreAsync` loads the profile → loads `RegulatoryCriteria` (rubric rows, which may exist independent of requirement approval) → calls Claude with criteria
- **Score IS generated** — scoring uses `RegulatoryCriteria`, not `RegulatoryRequirement`
- Results screen shows a score; user may believe regulatory compliance is confirmed
- Compliance browse (`GET /api/regulatory/browse`): returns **empty** — no Approved requirements visible
- `RequirementMappingJob`: nothing to map to — compliance checklist remains empty after publish
- No warning anywhere in the flow

**Current experience:** Silent disconnection. The score appears legitimate, but the compliance mapping chain (which tenants use to prove compliance) is non-functional. The user has no signal that their regulatory setup is incomplete.

### Scenario D: Tenant has sector, approved requirements, talk passes

Normal success path. Score generates, compliance mappings are created post-publish, checklist populates. No issues.

## 7. Recommended Fix Shape

### 7.1 Translate-time warning (non-blocking)

**What to show:** When the user has a sector selected in the wizard's translate step and that sector has zero Approved requirements, show an inline amber informational banner in the `TranslateValidateStep` component before (or immediately after) validation starts. The banner should not block the translate action — it warns, it doesn't prevent.

**Why non-blocking:** The scoring still works (uses `RegulatoryCriteria`). The missing piece is compliance mapping, not the score itself. Blocking would be disproportionate and confusing ("you can't translate because requirements are Draft").

**Recommended copy:**
> "The regulatory requirements for **{SectorName}** haven't been approved yet. Translation and scoring will proceed, but the compliance checklist will be empty until requirements are reviewed in Regulatory → System → {DocumentName}."

**Where:** Inline banner in `TranslateValidateStep.tsx`, rendered when sector is selected and applicability check returns `approvedRequirementCount == 0`.

### 7.2 Results-screen distinction

The `RegulatoryScorePanel` needs to render four states for the Regulatory Translation column (the other two columns are sector-independent and need no change):

| State | Condition | Rendering |
|---|---|---|
| **Not configured** | `sectorKey == null` | Existing amber warning (already handled) |
| **No profile** | `sectorKey` set, `applicability.hasRegulatoryProfile == false` | Amber callout: "No regulatory profile exists for this sector" — button disabled |
| **Requirements pending** | Profile exists, `applicability.approvedRequirementCount == 0` | Amber callout: "Requirements not yet approved — compliance mapping unavailable. Scoring will proceed against general criteria." — button enabled (scoring still works) |
| **Ready / Scored** | `approvedRequirementCount > 0` OR score already exists | Current behavior (button or score card) |

The "Requirements pending" state uses **amber** (not red) because scoring is still functional — it's the compliance mapping that's missing, not the score itself. This matches the existing design language (amber = warning but not blocked).

### 7.3 Backend extension needed

**Two additions:**

#### 1. New lightweight endpoint: Regulatory applicability check

```
GET /api/regulatory/applicability?sectorKey={key}
Policy: Learnings.Admin
→ { hasRegulatoryProfile: bool, approvedRequirementCount: int, profileName: string? }
```

Used by the wizard's translate step pre-flight check (before a run exists, so can't use the history endpoint).

Implementation: query `RegulatoryProfiles` by `SectorKey`, then count `RegulatoryRequirements` where `IngestionStatus == Approved` for that profile. Two simple queries.

Controller: add to `RegulatoryBrowseController` as a new action (`GetApplicability`). Keeps all "read-only regulatory browse for tenant admins" in one controller (note 24 pattern).

#### 2. Extend `RegulatoryScoreHistoryDto`

```csharp
public record RegulatoryScoreHistoryDto {
    public Guid ValidationRunId { get; init; }
    public RegulatoryScoreResultDto? SourceScore { get; init; }
    public RegulatoryScoreResultDto? PureScore { get; init; }
    public List<RegulatoryScoreResultDto> RegulatoryScores { get; init; } = new();
    // New:
    public RegulatoryApplicabilityDto? Applicability { get; init; }
}

public record RegulatoryApplicabilityDto {
    public bool HasRegulatoryProfile { get; init; }
    public int ApprovedRequirementCount { get; init; }
    public string? ProfileName { get; init; }
}
```

`GetScoreHistoryAsync` already loads the run — add a query for the sector key from the run, then check profile + requirement count. Two additional queries to populate `Applicability`. This powers the results-screen states without any extra frontend API call.

## 9. Risks and Edge Cases

### R1 — `GetScoreHistoryAsync` receives a run with `SectorKey = null`

`Applicability` should be null (not empty struct) when `SectorKey` is null. Frontend already handles `sectorKey == null` with the existing amber warning. Map: `SectorKey == null` → `Applicability = null` in the DTO. Frontend: `if (!sectorKey) { <existing warning> }` else `if (!history?.applicability) { <loading> }` else `{ <four-state logic> }`.

### R2 — Multiple profiles for a single sector key

`GET /api/regulatory/browse` already handles this by grouping. The applicability check should count approved requirements **across all profiles** matching the sector key (a HIQA tenant with two documents, both partially ingested). Use `Sum()` across profiles, not per-profile.

### R3 — The `TranslateValidateStep` is deprecated

`TranslateValidateStep.tsx:1` is marked `@deprecated — split into TranslateStep.tsx (step 5) and ValidateStep.tsx (step 6)`. The pre-flight banner should go into **`TranslateStep.tsx`** (step 5, the one with the sector selector) not the deprecated combined component. The deprecated file exists as legacy-wizard support; only the active wizard files matter for the warning. Confirm the actual active file paths before implementation.

### R4 — Warning copy clarity

The warning must distinguish "your sector has no regulatory profile in the system" (B scenario — contact support) from "your sector's document requirements haven't been approved yet" (C scenario — go approve them). The copy recommendations above handle this via different messages, but the UX wording should be reviewed with product before shipping.

### R5 — Scoring with empty `RegulatoryCriteria`

Scenario B (profile loads but `RegulatoryCriteria` table has no rows for this profile) causes `ScoreRegulatoryTranslationAsync` to send a prompt with empty criteria. Claude returns a score anyway based on general regulatory knowledge. The resulting `ValidationRegulatoryScore` row has `RegulatoryProfileId = null` (profile was found but criteria were empty? — verify this code path). This edge case may produce phantom scores. Low priority — system-seeded profiles should always have criteria — but worth a defensive check in `ScoreAsync`.

### R6 — `useRegulatoryApplicability` call timing in the wizard

The pre-flight check fires when sector is selected. If the user changes the sector mid-wizard, the check must re-fire. Use the sector key as a query key dependency. The check should not block the translate button — it's informational only.

### R7 — The applicability check endpoint is per-sector, not per-run

The wizard check (`GET /api/regulatory/applicability?sectorKey={key}`) fires before a run exists. The results-screen check is run-scoped (uses `history.applicability` populated from the run's `SectorKey`). These are two different data flows solving the same problem at different points in the lifecycle. Both are needed; they don't overlap.

# Slideshow Generation — New Wizard Recon

**Date:** 2026-06-26  
**Branch:** transval  
**Purpose:** Demo readiness check — can the boss show slideshow generation from the new wizard?

---

## 1. Headline

**Slideshow toggle IS present in the new wizard at Step 4 (Settings), but the generation trigger path is broken for the wizard flow.**

The toggle is gated on a PDF being uploaded (`talk?.pdfUrl` truthy). When enabled, it saves `generateSlidesFromPdf: true` to the talk entity. However, no new-wizard step actively invokes slideshow generation — the `generateSlides()` API call is only reachable from the **talk edit page** (`ToolboxTalkForm`), not from any wizard step.

**Demo implication:**  
- If the boss demos the **new wizard with a PDF input**, the "Generate slideshow from PDF" toggle will appear in the Settings step. He can enable it. But the slideshow will **not be generated** during the wizard. The PublishStep's "Slideshow — Ready" badge will not appear.  
- Slideshow generation would require navigating to the edit page after publish and clicking "Regenerate Slideshow".  
- If he demos the **legacy wizard with a PDF input**, the toggle appears in its SettingsStep, and slideshow generation is passed through to `ContentGenerationJob` and runs automatically as part of content generation. The PublishStep shows "Generating in background".

## 3. Data Path

### Fields on `ToolboxTalk` that gate slideshow generation

| Field | Type | Role |
|-------|------|------|
| `GenerateSlidesFromPdf` | `bool` | User preference; set by the new wizard's SettingsStep `saveField()` |
| `SlidesGenerated` | `bool` | Set to `true` by `SlideshowGenerationService` on completion |
| `SlideshowHtml` | `string?` | Full generated HTML; populated by `SlideshowGenerationService` |
| `SlideshowGeneratedAt` | `DateTime?` | Timestamp set on completion |

**Computed property:** `ToolboxTalkDto.HasSlideshow` → `!string.IsNullOrEmpty(SlideshowHtml)`

**Does the new wizard set the preference field?** Yes — SettingsStep saves `generateSlidesFromPdf: true/false` immediately on toggle. The field is persisted to the entity.

**Does the new wizard consume it to generate slides?** No — nothing in the wizard reads this field back and enqueues a generation job. The field sits set on the entity but is never acted upon within the wizard flow.

### Ordering problem

The new wizard's step order is:

```
1. InputConfigStep  (upload PDF)
2. ParseStep        ← content generation runs here
3. QuizStep
4. SettingsStep     ← slideshow toggle lives here
5. TranslateStep
6. ValidateStep
7. PublishStep
```

`ParseStep` (step 2) fires content generation before `SettingsStep` (step 4) has set `generateSlidesFromPdf`. Even if `ContentGenerationJob`'s `AutoGenerateSlidesAsync()` falls back to the entity's `GenerateSlidesFromPdf` flag, the flag is not set at the time Parse runs. The toggle in Settings arrives too late to influence the Parse-triggered job.

## 4. Trigger Path

### Backend generation chain

```
POST /api/toolbox-talks/{id}/generate-slides?source=pdf
  └─ ToolboxTalksController.GenerateSlides()
       └─ SlideshowGenerationService.GenerateSlideshowAsync()
            ├─ Source "pdf"      → GenerateFromPdfAsync()
            ├─ Source "sections" → GenerateFromSectionsAsync()
            └─ Source "video"    → GenerateFromVideoTranscriptAsync()
            On success:
              talk.SlideshowHtml = html
              talk.SlideshowGeneratedAt = now
              talk.SlidesGenerated = true

ContentGenerationJob.ExecuteAsync() (Hangfire, on-demand)
  └─ After sections/questions generated → AutoGenerateSlidesAsync()
       ├─ Uses options.SlideshowSource if provided
       ├─ Falls back to entity.GenerateSlidesFromPdf flag
       └─ Calls SlideshowGenerationService.GenerateSlideshowAsync()
```

### Frontend callers of `generateSlides()`

| Location | File | Triggered by |
|----------|------|--------------|
| Edit page | `web/src/features/toolbox-talks/components/ToolboxTalkForm.tsx` lines 220–236 | "Regenerate Slideshow" button in the "Animated Slideshow" section |
| **New wizard** | — | **Not called** |
| **Legacy wizard** | — | Indirectly via ContentGenerationJob (see §5) |

`smartGenerateContent()` is defined in `web/src/lib/api/toolbox-talks/toolbox-talks.ts` (lines 575–584) but is **never called** from any frontend component.

## 5. Legacy Wizard Comparison

| Feature | Legacy wizard | New wizard |

### Key legacy wizard mechanics (for parity reference)

`SettingsStep.tsx` lines 67–70 (auto-default):
```tsx
if (session?.inputMode === 'Pdf' && !merged.generateSlideshow) {
  merged = { ...merged, generateSlideshow: true };
}
```

The legacy wizard passes `generateSlideshow` and `slideshowSource` through `ContentCreationSettings` all the way into the `ContentGenerationJob` options, so the job receives them at enqueue time — before any race condition on step ordering can occur.

## 6. Post-Publish Path (Talk Detail / Edit Page)

**Yes — slideshow generation is reachable post-publish from the edit page, independent of the wizard.**

| Route | File | Action |
|-------|------|--------|
| `/admin/toolbox-talks/talks/[id]/edit` | `ToolboxTalkForm.tsx` lines 220–236, 970–1026 | "Animated Slideshow" section with **"Regenerate Slideshow"** button |

`handleRegenerateSlides()` in `ToolboxTalkForm.tsx` calls `generateSlides(id, source)` → `POST /api/toolbox-talks/{id}/generate-slides?source=pdf`. This works on any published talk regardless of how it was created.

`getAdminSlideshowHtml()` (read-only) is available from the talk detail page for viewing the generated HTML once it exists.

## 7. Gap Statement

**The new wizard's slideshow toggle is a UI-only preference saver.** It writes `generateSlidesFromPdf: true` to the entity but does not enqueue or trigger slideshow generation at any point in the wizard flow. The root cause is a step-ordering mismatch: content generation fires in ParseStep (step 2), the toggle appears in SettingsStep (step 4), and by the time the user enables the toggle, the generation job has already finished without knowing about it. The PublishStep shows no "Generating in background" indicator, and the "Slideshow — Ready" badge only appears if slides were somehow already generated (e.g., from a previous edit-page trigger on the same draft talk).

**For Friday's demo:**

| Demo path | Slideshow achievable? | Notes |
|-----------|----------------------|-------|
| Legacy wizard + PDF | **Yes** — auto-triggered during Parse/generation job | Toggle auto-defaults to true; "Generating in background" shown at Publish |
| New wizard + PDF | **No** — toggle appears but nothing generates slides | Would require post-publish visit to edit page |
| Edit page (either path, post-publish) | **Yes** — via "Regenerate Slideshow" button | Manual step; not part of wizard demo |

If the boss walks through lesson creation and needs to show a slideshow generating as part of the wizard, **use the legacy wizard** (accessible via `?wizard=old` URL parameter or the `UseNewWizard` tenant setting set to `false`).

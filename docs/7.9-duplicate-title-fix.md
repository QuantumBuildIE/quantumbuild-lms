# §7.9 — Duplicate-title error UX: Implementation Report

_Date: 2026-06-19_

---

## Summary of Changes

Two files modified. No backend changes. No new dependencies. No migration.

### `web/src/features/toolbox-talks/components/learning-wizard/steps/InputConfigStep.tsx`

**Change 1 — `onSubmit` catch block (was lines 320-323):**
Replaced the generic 3-line catch with Axios-aware extraction. The new block reads
`err.response?.data?.message` and checks for the substring `"already exists"`. On match:
calls `form.setError('title', { type: 'manual', message: serverMessage })` and
`form.setFocus('title')`. For all other 400s: falls back to `toast.error` with
`serverMessage ?? generic`. A mandatory comment documents why the string-match is safe
and what its failure mode is.

**Change 2 — Inline mutation-error banner (was lines 998-1008):**
Removed the entire `{initialiseMutation.isError && (<Alert>...</Alert>)}` block. With
the catch block now handling display (field-level error or toast), this banner was
redundant and always showed the generic Axios `"Request failed with status code 400"`
string, making the UX worse than the field-level error alone.

### `web/src/features/toolbox-talks/components/learning-wizard/steps/SettingsStep.tsx`

**Change 1 — `saveField` catch (was line 132-134):**
Replaced `} catch {` with `} catch (err) {` + status/code check. Reads
`err.response?.status === 409 && err.response?.data?.code === 'TitleNotUnique'` — the
structured `code` field means no string matching. On match: `form.setError('title', ...)`
+ `form.setFocus('title')`. All other errors fall back to `toast.error`.

**Change 2 — `handleContinue` catch (was lines 163-165):**
Identical replacement to `saveField` catch. Both paths hit the same `PUT
/toolbox-talks/{id}/settings` endpoint and receive the same 409 response, so the same
logic applies.

## Implementation Notes

### "already exists" string-match documented in code
The comment is present in the new `onSubmit` catch block in `InputConfigStep.tsx`. It
explains: (a) why the match is safe today (only `InvalidOperationException` from this
endpoint has this shape; FluentValidation 400s have `{ message, errors: [...] }` and
won't trigger it), and (b) the failure mode (backend message rewording → falls back to
toast, not a crash).

### Inline mutation-error banner removed
Confirmed removed. The `{initialiseMutation.isError && ...}` block is gone from the
JSX return. The catch block is now the single source of truth for error display on the
create path.

### Rename-path uses structured `code` field
`SettingsStep.tsx` catches check `response?.data?.code === 'TitleNotUnique'` — no
string-matching. The backend's `Conflict(new { error = ..., code = "TitleNotUnique" })`
is the canonical signal.

### Both forms already had `<FormMessage>` on the Title field
`InputConfigStep.tsx`: `<FormMessage id="title-error" role="alert" />` on the Title
`<FormItem>`. `SettingsStep.tsx`: `<FormMessage role="alert" />` on the Title
`<FormItem>`. `form.setError('title', ...)` will render through these without any
structural change.

### Duck-typing used — no new `AxiosError` import
Consistent with the rest of the frontend codebase (no other component file imports
`AxiosError`). Duck-typed as
`(err as { response?: { data?: { message?: string } } })` in InputConfigStep and
`(err as { response?: { status?: number; data?: { error?: string; code?: string } } })`
in SettingsStep.

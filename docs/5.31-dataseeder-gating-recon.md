## Scope-changing finding ÔÇö read first

`tests/QuantumBuild.Tests.E2E/global-setup.ts` hard-codes both the email **and** the password of the DataSeeder's admin account:

```typescript
const SEED_ADMIN = {
  email: 'admin@quantumbuild.ai',
  password: 'Admin123!',   // ÔåÉ literal from DataSeeder
};
```

`test-constants.ts` does the same (`email: 'admin@quantumbuild.ai'`, `password: 'Admin123!'`). The Playwright global-setup logs in as this account and uses it to bootstrap E2E test users before any test runs. If the seeder stops creating this account ÔÇö or creates it with a config-sourced password ÔÇö the E2E global-setup must be updated in the same chunk.

**This does NOT invalidate the fix.** It adds one small piece of work to the chunk: update the E2E global-setup and test-constants to read credentials from env vars with a fallback. See ┬ğ9.

### ┬ğ2 ÔÇö Integration test dependencies

**No integration tests depend on the seeded accounts.** Confirmed by grep (`superuser@certifiediq` and `admin@quantumbuild` return zero hits in `tests/QuantumBuild.Tests.Integration/`).

`CustomWebApplicationFactory` sets `builder.UseEnvironment("Testing")` (line 237 of `CustomWebApplicationFactory.cs`). After the gate, the Testing environment will not create the accounts. The test infrastructure is entirely JWT-forgery-based (`GenerateTestToken`) and `TestTenantSeeder` creates its own dedicated test users.

`Respawner` preserves `Users`, `Roles`, `RolePermissions`, `Permissions` in its `TablesToIgnore` list. This was a safety net for the currently-created DataSeeder accounts; after the fix those accounts won't exist in the test DB at all.

**Integration tests will not break.**

### ┬ğ3 ÔÇö Local-development login dependency

**Yes ÔÇö developers do log in with `admin@quantumbuild.ai` today** (it is the only real admin account in a fresh local Dev DB, and the E2E Playwright global-setup uses it explicitly).

The fix must preserve this capability. With dev credentials sourced from `appsettings.Development.json` (see ┬ğ4), the same values can be kept ÔÇö making the fix transparent to anyone currently using them.

The E2E test fix (reading from env vars with literal fallback) ensures those tests continue to work both in Development (seeder creates the account with the configured password) and in a future CI environment (where credentials can be overridden via env vars without touching test source).

### ┬ğ4 ÔÇö Config source for Development credentials

**Recommendation: Option A ÔÇö `appsettings.Development.json` only.**

Rationale:
- `appsettings.Development.json` is already committed to source with dev-only secrets (JWT secret, DB connection string). Dev credentials here match the existing project pattern.
- Works out-of-the-box for any fresh clone with no extra setup.
- The repo is private; these credentials only ever produce local-dev accounts.
- Env var override (Option C) adds friction for developers without adding security value when the repo is private and the values are local-only.

New section to add to `appsettings.Development.json`:

```json
{
  "Seed": {
    "SuperUser": {
      "Email": "superuser@certifiediq.ai",
      "Password": "SuperUser123!"
    },
    "Admin": {
      "Email": "admin@quantumbuild.ai",
      "Password": "Admin123!"
    }
  }
}
```

These are the same values as the current literals ÔÇö the fix is transparent to any developer currently using local Dev.

### ┬ğ5 ÔÇö Fail-fast on missing config

**Recommendation: Log a warning and skip** (not throw) if the Seed config values are absent in a Development environment.

Rationale: With Option A, the config values are always present in the committed `appsettings.Development.json`. The only way they'd be missing is if a developer explicitly removes them or overrides them to null via env vars. In that case a warning in the startup logs is sufficient ÔÇö the developer will quickly notice that no admin user was seeded when they try to log in. A hard startup block adds no practical value when the defaults are always present.

Implementation:

```csharp
if (environment.IsDevelopment())
{
    var superUserEmail = configuration["Seed:SuperUser:Email"];
    var superUserPassword = configuration["Seed:SuperUser:Password"];
    if (!string.IsNullOrEmpty(superUserEmail) && !string.IsNullOrEmpty(superUserPassword))
        await SeedSuperUserAsync(userManager, roleManager, superUserEmail, superUserPassword, logger);
    else
        logger.LogWarning("Seed:SuperUser credentials not configured ÔÇö skipping SuperUser seeding");

    // same for Admin + EnsureAdminEmployee
}
```

### ┬ğ6 ÔÇö Production behaviour after the gate

**Post-fix Production startup:**
- Existing Production accounts (`superuser@certifiediq.ai`, `admin@quantumbuild.ai`) are unchanged ÔÇö skip-if-exists means no overwrite of existing hashed passwords.
- Fresh Production deploys (new DB or DB reset) will NOT create these accounts. This is the entire point of the fix.
- `SeedToolboxTalksDataAsync` (Program.cs:335 ÔÇö system/reference data) is a separate call, unaffected, always runs.

**Bootstrap story for fresh Production / Demo environments:**

The seeder no longer bootstraps Production admin accounts. Fresh environments need a human-driven bootstrap. Two viable options, both already natural to the existing workflow:

1. **Recommended (operational, zero code):** The system-level SuperUser account must be created by direct DB insert (or a one-time migration script) before the first deploy. This matches the existing pattern documented in CLAUDE.md Note 20: "SuperUser accounts must be seeded or created directly in the database." Once a SuperUser exists, tenant Admins can be created via the application UI.

2. **For Demo (┬ğ5.7 step 14-15):** The ┬ğ5.7 task already plans manual creation of "Demo tenant" and "Demo Admin user" post-deploy. That plan is unchanged by this fix ÔÇö it just means the DataSeeder won't pre-create those accounts before the manual steps.

**No code change needed for the bootstrap story.** It is operational. The recommendation is to document it clearly in the CLAUDE.md note added by this chunk.

### ┬ğ7 ÔÇö Regression test shape

**Recommendation: Integration test in `tests/QuantumBuild.Tests.Integration/`** using the existing `CustomWebApplicationFactory` (environment = "Testing").

Rationale:
- The factory already runs `DataSeeder.SeedAsync` during its `InitializeAsync`. Environment is "Testing".
- After the fix, the Testing environment will not create the accounts. A test that asserts the accounts do NOT exist will:
  - **Fail** before the fix (accounts ARE created today, even under Testing)
  - **Pass** after the fix (accounts not created under Testing)
- No new factory, no new DB, no mocking of `IHostEnvironment` ÔÇö just assert on the actual UserManager result.
- Two tests, both asserting absence:
  1. `DataSeeder_InNonDevelopmentEnvironment_DoesNotCreateSuperUserAccount`
  2. `DataSeeder_InNonDevelopmentEnvironment_DoesNotCreateAdminAccount`

Location: `tests/QuantumBuild.Tests.Integration/Infrastructure/SeederBehaviourTests.cs`

Both tests use `Services.GetRequiredService<UserManager<User>>()` (via a scope) and call `FindByEmailAsync` for each address, asserting null.

The positive path (Development creates the accounts) is covered by observing the existing local Dev DB. A dedicated "Development creates accounts" test would require a second factory with a mock `IHostEnvironment` returning "Development" ÔÇö more setup for little additional assurance given the negative path is the security-relevant case.

### ┬ğ8 ÔÇö Migration concern

**No migration needed.** The fix is code-only:
- A conditional in `DataSeeder.SeedAsync`
- Config additions to `appsettings.Development.json`
- No schema change, no new entities, no EF model change.

**Existing Dev environments:** accounts remain in the DB unchanged. Skip-if-exists means subsequent restarts do not overwrite them.

**Fresh Dev checkout:** accounts created with the config-sourced password (identical to current literals since the defaults are unchanged). No visible difference.

**Production:** no accounts created on fresh deploy. Existing Production accounts untouched.

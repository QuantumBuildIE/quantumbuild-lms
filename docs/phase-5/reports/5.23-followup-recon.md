# §5.23 Follow-up Recon — IsActive Semantics & General Tab UI Pattern

**Type:** Read-only investigation
**Feeds:** §5.23 fix prompt (server-side tenant defaults for wizard Step 4)
**Date:** 2026-06-16

---

## Summary

**Q1 — IsActive and visibility:** `IsActive` is not a learner visibility gate. It is not a scheduling gate either. The only mechanism that controls whether a learner sees a talk is whether a `ScheduledTalk` assignment record exists for them. Neither `IsActive` nor `Status` (Draft, Published, etc.) is checked by any employee-facing query or by the scheduling job. `IsActive` is used solely for admin dashboard counts and as an optional admin list filter. Consequence: **Option B is recommended** — remove the hardcoded `IsActive = false` from `InitialiseToolboxTalkCommandHandler` and let the tenant's `DefaultIsActive` setting drive the initial value. A new draft with `IsActive = true` will not become visible to any learner until an admin creates a schedule and assigns it.

**Q2 — UI pattern:** The implemented Validation and QR Training tabs use a consistent per-section auto-save or explicit-button pattern built on **`useTenantSettings` / `useUpdateTenantSettings`** hooks — but these hooks target a **different settings store** (tenant-level key-value settings) than `ToolboxTalkSettings`. The General tab must use `useToolboxTalkSettings()` (already exists) and PUT `/api/toolbox-talks/settings` (currently a placeholder — this is a BLOCKER). The recommended pattern for the General tab is a single React Hook Form + Zod form for all five fields with a tab-level "Save Changes" button, matching the wizard's own form architecture rather than the ad-hoc local-state pattern of the other tabs.

## Q1.2 — Three-scenario walkthrough

**Scenario 1: Draft just created (IsActive=false, Status=Draft, no ScheduledTalk)**
- Learner visibility: **No** — the learner has no ScheduledTalk assignment. No assignment = not visible. IsActive=false is irrelevant.
- Gate: Absence of ScheduledTalk record for this employee.

**Scenario 2: Draft with all steps complete but not published (IsActive=false, Status=Draft or ReadyForReview, ScheduledTalk assigned)**
- Learner visibility: **Yes** — if an admin somehow assigned this talk (scheduled it), the learner sees it. There is no check that blocks a Draft-status talk from appearing in the employee portal.
- Gate: None. ScheduledTalk assignment is the only gate, and it does not check Status.

**Scenario 3: Talk published (IsActive=true, Status=Published, ScheduledTalk assigned)**
- Learner visibility: **Yes** — standard case.
- Gate: ScheduledTalk assignment.

**Scenario 4: Talk published then deactivated (IsActive=false, Status=Published, ScheduledTalk assigned)**
- Learner visibility: **Still yes** — deactivating the talk does not remove or suppress existing ScheduledTalk assignments.
- Gate: None for existing assignments. An admin would need to manually cancel or delete the assignments.

**Summary:** The only learner-visibility gate that exists is the ScheduledTalk assignment record. `IsActive` and `Status` are both inert with respect to what learners see.

## Q1.3 — Consequences of DefaultIsActive = true on new draft

If `InitialiseToolboxTalkCommandHandler` reads `DefaultIsActive = true` from settings and creates a new draft with `IsActive = true`:

1. **The draft is not visible to any learner** — because no ScheduledTalk assignment exists yet.
2. **The draft does not become schedulable automatically** — schedules are created explicitly by admins.
3. **The admin dashboard counts it as an "active talk"** — which is cosmetically accurate (the talk is not deactivated/archived).
4. **The admin list shows it without needing to clear the IsActive filter** — also fine.
5. **No other material effect was found.** `IsActive` is not read by reports, certificate generation, subtitle processing, or any background job except the dashboard aggregation query.

**Dashboard evidence:** `GetToolboxTalkDashboardQueryHandler.cs:23-30` counts `activeTalks = talks.Count(t => t.IsActive)`. A new draft with `IsActive=true` would increment this count, which is correct behavior — it signals "this talk is active, not archived."

**Risk from DefaultIsActive = false (current hardcoded behavior):** Admin creates a talk, completes all wizard steps, publishes — but `IsActive` was `false` from creation. The wizard's Step 4 renders the toggle, the admin may never notice it defaulted to inactive. This is a latent usability bug: admins who don't touch the toggle get an inactive published talk that won't appear in active-talk counts.

## Q1.4 — Recommendation

**Option B: Remove the hardcoded `IsActive = false`. Let `InitialiseToolboxTalkCommandHandler` read `DefaultIsActive` from `ToolboxTalkSettings`.**

Rationale:
- `IsActive` is not a learner-visibility gate. A new draft with `IsActive=true` cannot be seen by any learner.
- The hardcoded `IsActive = false` is a usability trap: admins who don't notice the wizard toggle publish inactive talks.
- The null-coalescing fallback `settings?.DefaultIsActive ?? false` preserves current behavior for all existing tenants who have no settings row (or whose settings row doesn't have the new field set). No regression.
- The canonical wizard form (SettingsStep.tsx) already exposes `isActiveOnPublish` as a Switch the admin controls. Server-side defaults just set the starting position of that toggle.

The concern in G4 was: "if DefaultIsActive=true, does it make a draft talk visible before it's published?" The answer is definitively no — visibility is gated by ScheduledTalk assignment, which requires explicit admin action. Option B is safe.

The four implemented sections are internally consistent:

| State approach | Local state (no RHF) |
| Settings hook | `useTenantSettings()` + `useUpdateTenantSettings()` |

The General tab fields (minimumVideoWatchPercent, autoAssignDueDays, generateCertificate, refresherFrequency, isActive) live on `ToolboxTalkSettings`, exposed via:
- **Load:** GET `/api/toolbox-talks/settings` → `useToolboxTalkSettings()` (already exists at `web/src/lib/api/toolbox-talks/use-toolbox-talks.ts:70-75`)
- **Save:** PUT `/api/toolbox-talks/settings` — **currently a placeholder** (BLOCKER)

The General tab cannot use `useTenantSettings()`. It uses a different hook. The other tabs are not a template for the API contract — only for the save-flow and feedback patterns.

## Q2.5 — General tab specification

### Architecture

Single React Hook Form + Zod form (not per-section local state), matching the wizard's own form pattern. The five fields are logically related enough to warrant a single save action.

### Fields and controls

| Field | Control | Validation |
|---|---|---|
| `defaultMinimumVideoWatchPercent` | `Input type="number"` | min 50, max 100, required integer |
| `defaultAutoAssignDueDays` | `Input type="number"` | min 1, max 90, required integer |
| `defaultGenerateCertificate` | `Switch` | boolean, no validation |
| `defaultRefresherFrequency` | `Select` (Once / Monthly / Quarterly / Annually) | required string |
| `defaultIsActive` | `Switch` | boolean, no validation |

### Form structure

- Single `useForm` with `zodResolver` for all five fields.
- Load: `useToolboxTalkSettings()` hook — populate `reset()` on load (same pattern as the wizard's `useEffect`).
- Save: dedicated mutation calling PUT `/api/toolbox-talks/settings` (to be implemented in the same chunk).
- `isDirty` from RHF's `formState.isDirty` controls button visibility.
- Tab-level "Save Defaults" button, shown only when dirty — matches `PassThresholdSection` and `AuditPurposeSection` patterns.

### Loading / error states

- `Skeleton` while `isLoading` (matches other sections).
- Toast on save success (`"Default settings saved"`).
- Toast on save error (`"Failed to save default settings"`).

### Field grouping

Two natural groups using `Card` + `CardHeader` + `CardContent` (matching the broader admin UI pattern):
1. **Video & completion** — `defaultMinimumVideoWatchPercent`
2. **Talk defaults** — `defaultGenerateCertificate`, `defaultRefresherFrequency`, `defaultAutoAssignDueDays`, `defaultIsActive`

### API contract

- **Load:** `useToolboxTalkSettings()` → GET `/api/toolbox-talks/settings` (Result envelope: `response.data.data`)
- **Save:** PUT `/api/toolbox-talks/settings` with `UpdateToolboxTalkSettingsDto` containing the five new fields (must be implemented in the same chunk; not a follow-up)

### Prerequisites (must land in same §5.23 chunk)

1. Five new fields on `ToolboxTalkSettings` entity + migration
2. Five new fields on `ToolboxTalkSettingsDto` + TypeScript interface
3. `UpdateToolboxTalkSettingsCommand` (tenant-level — new command, distinct from the per-talk command of the same name) + handler
4. Controller wired to dispatch new command (remove `// TODO` placeholder)

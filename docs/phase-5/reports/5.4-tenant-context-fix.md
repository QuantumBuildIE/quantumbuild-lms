# 5.4 — Tenant Context Fix: TranslationWorkflowService Hangfire Compatibility

**Date:** 2026-06-11  
**Branch:** `transval` (HEAD: see `git log --oneline -1`)  
**Scope:** Backend only — no frontend changes

---

## Root Cause

`TranslationWorkflowService` had an implicit HTTP-context dependency that prevented it from being called correctly from Hangfire background jobs.

All DB reads went through EF Core's `HasQueryFilter` tenant predicate (`e.TenantId == currentTenantId`). In Hangfire context, `HttpContext` is null, so `ICurrentUserService.TenantId` returned `Guid.Empty`. The tenant filter then became `e.TenantId == Guid.Empty` — matching nothing. Simultaneously, `AddEvent` didn't set `TenantId` explicitly, relying on the `SaveChanges` auto-stamp interceptor, which also stamps `Guid.Empty` in Hangfire context.

**Observed symptom (Railway Development smoke test of Phase 5.4):**  
`TranslationStarted` events were present (written from HTTP context when the wizard step submitted). No `TranslationCompleted` or `ValidationCompleted` events were ever written (Hangfire context). `TranslationValidationJob.ExecuteAsync` called `RecordValidationCompleted` at line ~350 and `RecordTranslationCompleted` at line ~1032, but in each case `GetState` returned `Initial` (empty query result) rather than the correct state, so the state-machine guards rejected every call with `WorkflowInvalidState`. The wizard UI was permanently stuck at "Translating..." because `ValidationCompleted` was never written.

---

## Fix Summary

Added optional `Guid? explicitTenantId = null` to 12 non-token-based public methods on the interface and implementation. HTTP callers remain unchanged (pass no argument). Hangfire callers pass `explicitTenantId: tenantId`.

Internal reads use `IgnoreQueryFilters()` + explicit `TenantId == tenantId` predicate. Internal writes set `TenantId` explicitly on every `WorkflowEvent`, `WorkflowReview`, and `ExternalParticipantInvitation` added to the context.

A `ValidateExplicitTenantId` guard at the top of each Result-returning method returns `WorkflowInvalidState` failure immediately if `explicitTenantId == Guid.Empty` is passed explicitly (caller bug).

## Summary

The redirect block in `TenantQueryInvalidator` (`web/src/lib/providers.tsx`) has been removed. The component now performs only `queryClient.invalidateQueries()` on tenant change and does nothing else. The `usePathname`, `useRouter` hooks, their `next/navigation` import, and the `UUID_REGEX` constant were all dead code after the removal and have been deleted alongside the block. ## Edit (1) — Remove redirect block from TenantQueryInvalidator

**File:** `web/src/lib/providers.tsx`

### Before

```tsx
import { useEffect, useRef, type ReactNode } from "react";
import { usePathname, useRouter } from "next/navigation";

const UUID_REGEX = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;

function TenantQueryInvalidator() {
  const { activeTenantId } = useAuth();
  const queryClient = useQueryClient();
  const previousTenantId = useRef(activeTenantId);
  const pathname = usePathname();
  const router = useRouter();

  useEffect(() => {
    if (previousTenantId.current !== activeTenantId) {
      previousTenantId.current = activeTenantId;
      queryClient.invalidateQueries();

      // If on a detail/edit page (path contains a UUID segment),
      // redirect to the parent list page
      const segments = pathname.split("/");
      const uuidIndex = segments.findIndex((s) => UUID_REGEX.test(s));
      if (uuidIndex > 0) {
        const parentPath = segments.slice(0, uuidIndex).join("/");
        router.replace(parentPath);
      }
    }
  }, [activeTenantId, queryClient, pathname, router]);

  return null;
}
```

### After

```tsx
import { useEffect, useRef, type ReactNode } from "react";

function TenantQueryInvalidator() {
  const { activeTenantId } = useAuth();
  const queryClient = useQueryClient();
  const previousTenantId = useRef(activeTenantId);

  useEffect(() => {
    if (previousTenantId.current !== activeTenantId) {
      previousTenantId.current = activeTenantId;
      queryClient.invalidateQueries();
    }
  }, [activeTenantId, queryClient]);

  return null;
}
```

## §5.20 Interaction Note

**The SuperUser slice of §5.20 is now closed as a side effect of this fix.**

The §5.20 BACKLOG entry describes "refresh on any `/learnings/{talkId}/{step}` route lands on `/drafts` rather than the step the user was on." This was caused by `TenantQueryInvalidator` firing during hydration (the `null → storedUUID` transition) and redirecting to the parent path, which then redirected to `/drafts`. With the redirect removed, that chain no longer fires. A SuperUser refreshing on any wizard step stays on that step — the talk fetches with the restored tenant ID and renders normally.

**The regular-admin slice of §5.20 remains open.** Regular admins' `activeTenantId` is always `null` (they have no tenant-switching surface) and never transitions, so the `TenantQueryInvalidator` was never the cause for them. If regular admins also experience refresh-position loss, the cause lies elsewhere (auth layout guards or session restoration) and is not addressed by this fix.

**BACKLOG §5.20 update:** The entry should be narrowed to: "Regular-admin refresh behavior on wizard steps needs investigation — SuperUser case closed by §5.22 fix (2026-06-15)."

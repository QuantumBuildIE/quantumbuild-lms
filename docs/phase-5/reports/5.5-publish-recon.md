## Orientation Summary (Pass 1)

**What Publish does today.** Publish is a session-based operation triggered by `POST /api/toolbox-talks/create/session/{sessionId}/publish` in `ContentCreationController.cs:266`. It calls `ContentCreationSessionService.PublishAsync` which (a) validates session state, (b) re-syncs ALL session JSON (sections, quiz questions, quiz settings, behaviour settings) into the draft `ToolboxTalk` entity, (c) promotes the talk's `Status` to `Published`, (d) sets `session.Status = Completed`, and (e) fire-and-forgets a `RequirementMappingJob` (AI regulatory mapping). The operation is talk-wide — all target-language translations publish together. There is no unpublish endpoint.

**Old wizard vs new wizard.** A complete, working `PublishStep.tsx` exists for the old session-based wizard (`create-wizard/steps/PublishStep.tsx`, 727 lines). It renders a three-panel summary (content preview, per-language back-translation scores, quiz details), an audit metadata panel, and a Publish button. The new learning wizard has a 12-line placeholder at `learning-wizard/steps/PublishStep.tsx` with a TODO comment labelled "Phase 5.6" (note: this chunk is 5.5 — the placeholder was written before the phase number was finalized). Step 7 is hardcoded unreachable in `stepOrder.ts:60` with comment "5.5 owns the real rule."

**External review and publish.** External review is an optional post-validation workflow state (`AwaitingThirdParty` → `ThirdPartyReviewed`) defined in `TranslationWorkflowState.cs`. It does NOT gate publish today — `PublishAsync` checks only session status (`Parsed`, `Validated`, or `QuizGenerated`) and does not inspect external review state. A translation in `AwaitingThirdParty` can be published without cancelling the pending review. This is either intentional (optional external review as a parallel track) or an unaddressed gap. Backlog §11 defers the cancel-external-review endpoint to Phase 4.

**Critical architecture gap discovered.** The existing publish endpoint takes a `sessionId`. The new wizard works exclusively from `talkId`. `ContentCreationSession` stores `OutputTalkId` (the draft talk ID, set in `StartTranslateValidateAsync:611`), but `ToolboxTalk` has no back-reference to its session. No "get session by talk ID" endpoint exists in `ContentCreationController`. **The new wizard cannot call the session-based publish endpoint without a new backend addition.** The good news: by the time Step 7 is reached, the draft talk already has all necessary data on its entity (sections, quiz, translations, settings all persisted by prior steps) — so a direct-publish endpoint working from `talkId` alone is viable and simpler than the session-based path.

Title falls back to `SettingsJson.title` if not supplied (line 834-846). Title is still required after fallback (line 848).

**Validation logic in `PublishAsync` (ContentCreationSessionService.cs:822-850):**
```csharp
if (session.Status != Parsed && session.Status != Validated && session.Status != QuizGenerated)
    → error "Session must be in Parsed, Validated, or QuizGenerated status"

if (session.OutputType == null)
    → error "Output type must be set before publishing"
```

**Does NOT validate:**
- Whether target language translations have completed validation runs
- Whether validation outcomes are Pass (fails with Review or Fail outcomes are publishable)
- Whether external reviews are still `AwaitingThirdParty`
- Whether quiz questions exist when `RequiresQuiz = true`

**Post-publish side effects:**
- `session.Status = Completed`, `session.OutputTalkId = draftTalk.Id` (already set since StartTranslateValidate — overwritten to same value)
- `RequirementMappingJob` enqueued for lesson (lines 1683-1684, 1787-1788) and for each course (lines 2185-2186) — fire-and-forget
- Slideshow generation job queued if settings indicate `GenerateSlideshow = true` (lines 883-924)

**5.5 work implied:** A new `POST /api/toolbox-talks/{talkId}/publish` endpoint is needed. The session-based endpoint cannot be called from the new wizard (no sessionId available). The new endpoint: validates `talk.Status == Draft`, checks sections exist, sets `talk.Status = Published`, enqueues `RequirementMappingJob`. Simpler than the session path — no JSON re-sync needed since the draft talk already has all data.

### 6. State that Publish needs to read

The new wizard tracks everything by `talkId`, not `sessionId`. The `useTalk(talkId)` hook fetches `GET /api/toolbox-talks/{id}` and returns a `ToolboxTalk` object. The table below maps what Publish needs against what is already available:

| Data needed | Entity field / endpoint | Already fetched? | Note |
|---|---|---|---|
| Title, description, category | `talk.title`, `.description`, `.category` | Yes — `useTalk` | |
| Source language | `talk.sourceLanguageCode` | Yes — `useTalk` | |
| Section count | `talk.sections.length` | Yes — `useTalk` | |
| Output type (Talk vs Course) | `talk.outputType` | Need to verify field exists on DTO | See observation §A |
| Target languages | `talk.targetLanguageCodes` (JSON string) | Yes — `useTalk` | Needs `JSON.parse` |
| Video / PDF presence | `talk.videoUrl`, `talk.pdfUrl` | Yes — `useTalk` | |
| Quiz settings | `talk.requiresQuiz`, `talk.passingScore` | Yes — `useTalk` | |
| Certificate / refresher settings | `talk.generateCertificate`, `talk.requiresRefresher` | Verify on DTO | See observation §B |
| Validation run outcomes per language | `GET /api/toolbox-talks/{id}/validation/runs` | Yes — `useValidationRuns` exists | Already used in Validate step |
| Workflow state per language | `useWorkflowSubscription(talkId)` | Yes — used in Step 6 | Already available |
| Audit metadata | `talk.reviewerName`, `.reviewerOrg`, `.auditPurpose` etc. | Verify on DTO | These are on session, not talk — may need checking |

The key difference from the old wizard: **the old wizard re-read everything from `session.SettingsJson` / `session.QuestionsJson` at publish time, because those were the source of truth**. In the new wizard, the draft talk entity IS the source of truth by Step 7, so `useTalk` is sufficient for the summary panel. No session lookup is required for display.

**5.5 work implied:** Audit which `ToolboxTalk` DTO fields are currently exposed by the API (`GET /api/toolbox-talks/{id}`) to confirm all needed fields are present. Fields from session audit metadata (reviewer name, org, purpose) may not be on the talk DTO and may require the new publish endpoint to pull them from the session at publish time — or expose them separately.

## Sizing Estimate

**Estimate: Medium (~3 days)**

The UI port is the largest single piece, but the old wizard's `PublishStep.tsx` is a solid reference. The critical unknown (session vs. talk architecture) is now resolved — a new `talkId`-based endpoint is the right path, and it's simpler than the session endpoint because no JSON re-sync is needed. The step is not greenfield (strong reference exists), not blocked on any deferred phase 4 items, and has no new hub infrastructure.

**Work breakdown:**
- New backend publish endpoint (`POST /api/toolbox-talks/{talkId}/publish`) + RequirementMappingJob enqueue: ~0.5 day
- Integration test for the endpoint: ~0.5 day
- `isStepReachable(7, talk)` rule + any required wizard state changes: ~0.5 day
- `PublishStep` component implementation (summary panels, publish action, success state): ~1 day
- Wiring publish button into page wrapper footer, post-publish navigation: ~0.25 day
- E2E test for publish flow: ~0.25 day

**Top 3 unknowns:**

1. **Step 7 reachability rule.** When exactly should Step 7 become reachable? Options: (a) as soon as sections exist (most permissive — same as steps 5/6); (b) only after all translations are `AIGenerated` or better; (c) only after at least one language has a completed validation run. If the answer is (c), the rule needs to read from validation run data, which requires either a new field on the talk DTO or a hook that's not currently in stepOrder context. This decision determines whether the reachability check is cheap (local field on talk) or expensive (requires a network call or caching).

2. **What to surface when external review is pending.** The current backend has no guard. Does 5.5 add a warning banner ("1 language awaiting external review — you can still publish")? Or silently allow it? Or soft-block? The decision affects both the UI (warning component) and whether the backend endpoint needs an `allowWithPendingReview` flag.

3. **Audit metadata availability on the talk DTO.** The old wizard showed reviewer name, organisation, audit purpose, and document reference — sourced from `session.ReviewerName` etc. These fields are on `ContentCreationSession`, not on `ToolboxTalk`. The new publish step may want to display them in the summary. Whether they're exposed via the existing `GET /api/toolbox-talks/{id}` response (or need a new endpoint / session lookup) determines whether display is cheap or requires extra work.

## Observations (Read-only — do not act on during recon)

**§A — `OutputType` field on ToolboxTalk DTO.** `ContentCreationSession.OutputType` (Lesson vs Course) determines whether `PublishAsLessonAsync` or `PublishAsCourseAsync` runs. It's a session field. Whether this is exposed on the draft `ToolboxTalk` entity or its API DTO is unconfirmed. The new publish endpoint will need to know it. May require a new field or inference from `talk.isPartOfCourse` / course membership.

**§B — Certificate / refresher / auto-assign fields on talk DTO.** `GenerateCertificate`, `RequiresRefresher`, `RefresherIntervalMonths`, `AutoAssignToNewEmployees`, `AutoAssignDueDays` are persisted to the talk entity by Step 4. Verify they're included in the `GET /api/toolbox-talks/{id}` response DTO so the Publish summary panel can display them without a separate settings fetch.

**§C — Placeholder says "Phase 5.6".** `learning-wizard/steps/PublishStep.tsx:3` has `// TODO (Phase 5.6)`. This is a naming artefact — the placeholder was written before the phase number was finalized. Update the comment when implementing; no functional impact.

**§D — `session.OutputTalkId` set at translate-start, not at publish.** `ContentCreationSessionService.cs:611`: `session.OutputTalkId = talkId` is set inside `StartTranslateValidateAsync`, not `PublishAsync`. This means the session already points to the draft talk before publish. A `GET /api/toolbox-talks/create/session/by-talk/{talkId}` endpoint could be added as an alternative to a new direct-publish endpoint — it would return the `sessionId`, letting the frontend call the existing session publish path. However, the direct-publish approach (new `POST /api/toolbox-talks/{talkId}/publish`) is simpler and avoids threading session concerns into the new wizard's architecture.

**§E — Old wizard `PublishStep` re-syncs quiz question IDs.** `PublishAsLessonAsync` (lines 1525-1527) hard-deletes existing quiz questions via `ExecuteDeleteAsync` and recreates them from session JSON. This is necessary in the old wizard because the session is the source of truth. In the new wizard, quiz questions are persisted directly to the talk entity during the Quiz step — no re-sync needed at publish. The new endpoint must NOT replicate this delete-and-recreate pattern.

## Summary

Toggle infrastructure for the wizard parallel period is shipped. The `UseNewWizard` TenantSetting defaults to `"false"` (legacy wizard). When set to `"true"`, the "Create New" button on the Learnings list and the legacy wizard's "Create Another" both route to the new wizard. A live Resume-routing bug (legacy-wizard drafts in the drafts list would have navigated to a new-wizard step URL that doesn't know about them) is fixed as part of this chunk. Flipping any tenant to `"true"` remains gated on §24 (Edit workflow design).

### Edit 8 — Frontend: `drafts/page.tsx` (three changes)

`web/src/app/(authenticated)/admin/toolbox-talks/learnings/drafts/page.tsx`

**(a) `handleResume` — live bug fix.** Previously, `lastEditedStep ?? 2` meant legacy-wizard drafts (where `lastEditedStep === null`) would navigate to `/admin/toolbox-talks/learnings/{id}/parse` — a new-wizard step URL that doesn't know about legacy-wizard talks. **(b) `stepLabel(null)` fallback.** Was "Step 1 — Input & Config" (misleading — the legacy wizard has no step URLs). Changed to `'—'`.

**(c) Legacy badge in `DraftRow`.** When `draft.lastEditedStep === null`, renders a `<Badge variant="secondary">Legacy</Badge>` adjacent to the title. New-wizard drafts get no badge.

## §24 dependency note

No tenant should have `UseNewWizard` set to `"true"` until §24 (Edit workflow design) progresses to at least:
- **Chunk 1** (talk detail page edit UI — section content editing, dirty-state, save-and-cascade)
- **Chunk 3** (validate step UI accessible from talk detail page post-publish)

Without these, new-wizard talks are creation-only with no edit path, which is not a defensible production state. The toggle infrastructure itself is safe to deploy at the default position — no tenant is affected until an operator explicitly flips their setting.

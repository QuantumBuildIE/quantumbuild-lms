## 6. The Root Cause: The AIGenerated Window

The backend job `TranslationValidationJob.ExecuteAsync` processes new-wizard runs (`IsNewWizard=true`) like this:

**Phase A — Translation generation** ([TranslationValidationJob.cs:523-531](src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Infrastructure/Jobs/TranslationValidationJob.cs#L523-L531)):  
`LoadSectionsAsync` detects no existing translation → calls `GenerateTranslationForSectionsAsync` → calls Claude API to generate content → saves `ToolboxTalkTranslation` record → calls `RecordTranslationCompleted` → writes `TranslationCompleted` event → **state becomes `AIGenerated`**.

**Phase B — Validation** ([TranslationValidationJob.cs:164-341](src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Infrastructure/Jobs/TranslationValidationJob.cs#L164-L341)):  
Job loops over sections, calls `ValidateSectionAsync` for each (back-translation via Claude/DeepL/Gemini). This phase takes 10–120 seconds depending on section count and API latency.

**Phase B end** ([TranslationValidationJob.cs:362-368](src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Infrastructure/Jobs/TranslationValidationJob.cs#L362-L368)):  
`RecordValidationCompleted` → writes `ValidationCompleted` event → **state becomes `Validated`**.

**The state sequence for a new-wizard run:**
```
[API returns]  →  Translating
[Phase A done] →  AIGenerated    ← LONG window: entire Phase B duration
[Phase B done] →  Validated
```

**`Validating` is NEVER written for new-wizard runs.** The job goes directly from `AIGenerated` to `Validated`. The `AIGenerated` window spans Phase B — which is the majority of job runtime (10–120 seconds).

`ACTIVE_STATES = {'Translating', 'Validating'}`. `AIGenerated` is not in `ACTIVE_STATES`.

**The subscriber lifecycle during this window:**

When any `workflow-state` query refetch fires while a language is in the `AIGenerated` window, that language's run ID is absent from `activeRunIds`. React reconciles the `WorkflowSubscriber` list: the subscriber for that run unmounts. Cleanup closes the SignalR connection and unsubscribes from the hub group.

When Phase B ends and `ValidationComplete` fires on the hub, there is no subscriber. The event is lost. The UI stays stuck on `AIGenerated`.

**What triggers the problematic refetch?**

The most common trigger: another language's `WorkflowSubscriber` calls `onValidationComplete` (its own run finished), which calls `invalidate()`, which refetches `workflow-state`. If at that moment a different language is in Phase B (`AIGenerated`), its subscriber unmounts.

With a 1-second stagger and jobs running in parallel, multiple languages can be in Phase B simultaneously. The first to finish validation triggers a refetch that unmounts the others' subscribers.

## 7. Timing Analysis: Subscriber Mount vs Event Firing

With the 1-second stagger, each language's `onSuccess` fires ~300–500ms after its mutation is called. At that moment (< 3 seconds total), all jobs are still in `Translating` state (Phase A hasn't completed yet — Claude translation takes 30–120 seconds). So the refetches triggered by each `onSuccess` correctly return `Translating` and mount all three subscribers. **This startup phase is fine.**

The failure window opens only after Phase A completes (~30–120 seconds into the job). At that point, the subscriber IS mounted and the SignalR connection IS established. The subscriber then gets unmounted by a subsequent refetch triggered by another language's `ValidationComplete`.

**No mount-timing race at startup.** The race is a mid-job unmount, not a missed early event.

## 8. Hypothesis

**Root cause (single sentence):** `TranslationValidationJob` transitions workflow state from `Translating` to `AIGenerated` after generating translations, skips `ValidationStarted`, and only advances to `Validated` at job end — creating a long `AIGenerated` window during which `ACTIVE_STATES` excludes the language's run ID, causing its `WorkflowSubscriber` to unmount mid-job.

**Mechanism:**
1. New-wizard job runs. After translation generation completes, `RecordTranslationCompleted` writes `TranslationCompleted` event → state = `AIGenerated`.
2. While Phase B (validation) runs (10–120 seconds), state remains `AIGenerated`.
3. Another language's `ValidationComplete` event fires → `onValidationComplete` → `invalidate()` → `workflow-state` refetches.
4. Refetch returns the stuck language with `state: 'AIGenerated'` — not in `ACTIVE_STATES`.
5. React reconciles: `WorkflowSubscriber` for the stuck language unmounts. SignalR connection closes.
6. Stuck language's Phase B finishes → `ValidationComplete` SignalR event fires → no subscriber to catch it.
7. State on the backend is now `Validated`, but the UI shows `AIGenerated` indefinitely.
8. UI only resolves when the next invalidation refetch (e.g., another language finishing, or window refocus) returns `Validated` for the stuck language — at which point the subscriber was never needed.

**Fix shape:** Backend. Add a `RecordValidationStarted` call in `ExecuteAsync` immediately before the section validation loop begins (after `LoadSectionsAsync` returns). This transitions state to `Validating` before Phase B starts, keeping the language in `ACTIVE_STATES` for the entire job. The `AIGenerated` window shrinks to the gap between `RecordTranslationCompleted` and `RecordValidationStarted` — two sequential async operations, effectively zero from a UI-observability standpoint.

Alternatively: suppress the `RecordTranslationCompleted` call for new-wizard runs entirely (keep state as `Translating` throughout) and only call `RecordValidationCompleted` at the end. This is also correct but loses the audit trail of when translation vs validation phases completed.

**Fix size:** Small. Backend only. ~5 lines — one `await _workflowService.RecordValidationStarted(...)` call in `ExecuteAsync` after `LoadSectionsAsync`, before the section loop.

## 9. Out-of-Scope Observations

**A. `ValidationStarted` event type exists but is unused for new-wizard runs.**  
`WorkflowEventTypes.ValidationStarted` is defined and `EventTypeToState` maps it to `Validating`. The `RecordValidationStarted` method presumably exists on `ITranslationWorkflowService`. The fix is to call it — no new infrastructure needed.

**B. The `runId` returned by the mutation is discarded.**  
`startTalkTranslation` returns `{ runId: string }` and `useStartTalkTranslation.onSuccess` receives it in `result` but ignores it. An alternative frontend-only fix would be to track returned run IDs locally and merge them into `activeRunIds` for a brief window after mutation success. This is viable but treats the symptom rather than the cause. The backend fix is cleaner.

**C. The `AIGenerated` window between `RecordTranslationCompleted` and `RecordValidationStarted` (after the fix) will still be brief.**  
If the fix adds both calls sequentially, there is a short window (two `SaveChangesAsync` round-trips, ~5–50ms) where state is `AIGenerated`. A refetch landing in this window would still unmount the subscriber, but the probability is negligible compared to the current 10–120 second window. If this edge case needs to be eliminated entirely, the alternative is to merge both events into a single batch or to omit `RecordTranslationCompleted` from the new-wizard path.

**D. Old-wizard validation runs (IsNewWizard=false) are not affected.**  
They already have translations populated before the validation job runs; `GenerateTranslationForSectionsAsync` returns early without calling `RecordTranslationCompleted`. Their state stays `Translating` (set by the old wizard's flow) until `RecordValidationCompleted` fires. The `Validating` state is also not written there, but `Translating` remains in `ACTIVE_STATES` throughout, so the subscriber stays mounted.

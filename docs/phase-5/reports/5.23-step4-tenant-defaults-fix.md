## Summary

Five tenant-level default fields (`DefaultMinimumVideoWatchPercent`, `DefaultAutoAssignDueDays`, `DefaultGenerateCertificate`, `DefaultRefresherFrequency`, `DefaultIsActive`) were added to `ToolboxTalkSettings`, migrated, exposed in the settings admin UI General tab, and wired into `InitialiseToolboxTalkCommandHandler` so new talks receive tenant-configured values instead of hardcoded entity defaults. The hardcoded `IsActive = false` override at creation was removed — per follow-up recon Q1.4, `IsActive` is not a learner-visibility gate, so setting it to the tenant's `DefaultIsActive` preference (defaulting to `false` for tenants with no settings row, preserving existing behaviour) is safe. `DefaultPassingScore` (already present on the entity) was also wired at creation for consistency.

## Edit 4 — New command, handler, validator

New directory: `src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Application/Commands/UpdateToolboxTalkTenantDefaults/`

**`UpdateToolboxTalkTenantDefaultsCommand.cs`** — record with `TenantId` + five fields, returns `Result<ToolboxTalkSettingsDto>`.

**`UpdateToolboxTalkTenantDefaultsCommandHandler.cs`** — runs FluentValidation, looks up the settings row (creates one if absent using the create-on-first-save pattern), updates the five fields, saves, returns full DTO.

**`UpdateToolboxTalkTenantDefaultsCommandValidator.cs`** — validates ranges: `DefaultMinimumVideoWatchPercent` 50–100, `DefaultAutoAssignDueDays` 1–90, `DefaultRefresherFrequency` must be one of Once/Monthly/Quarterly/Annually.

Naming rationale: the existing `UpdateToolboxTalkSettingsCommand` is per-talk (wizard Step 4, updates a specific `ToolboxTalk`); the new `UpdateToolboxTalkTenantDefaultsCommand` is tenant-level (updates `ToolboxTalkSettings`). Different targets, different names, no collision.

## Edit 7 — InitialiseToolboxTalkCommandHandler

**File:** `src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Application/Commands/InitialiseToolboxTalk/InitialiseToolboxTalkCommandHandler.cs`

Before constructing the `ToolboxTalk` entity, the handler now:
1. Queries `ToolboxTalkSettings` for the tenant (null-safe — missing row returns `null`).
2. Maps `DefaultRefresherFrequency` string → `RequiresRefresher` + `RefresherIntervalMonths` using the same switch expression as `UpdateToolboxTalkSettingsCommandHandler`.
3. Applies all five wizard defaults (plus `PassingScore`) via null-coalescing fallbacks.

Key change — `IsActive = false` hardcoded override removed:
```csharp
// Before:
IsActive = false,

// After:
IsActive = tenantSettings?.DefaultIsActive ?? false,
```

Null-coalescing fallback (`?? false`) preserves the pre-fix `IsActive = false` behaviour for tenants with no settings row, so existing tenants are unaffected.

Full diff for the new lines:
```csharp
IsActive = tenantSettings?.DefaultIsActive ?? false,
GenerateCertificate = tenantSettings?.DefaultGenerateCertificate ?? true,
MinimumVideoWatchPercent = tenantSettings?.DefaultMinimumVideoWatchPercent ?? 90,
AutoAssignDueDays = tenantSettings?.DefaultAutoAssignDueDays ?? 14,
PassingScore = tenantSettings?.DefaultPassingScore ?? 80,
RequiresRefresher = defaultRequiresRefresher,
RefresherIntervalMonths = defaultRefresherIntervalMonths,
```

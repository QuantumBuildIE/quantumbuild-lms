# §5.24 — Frequency Conflict Recon

**Type:** Investigation report  
**Status:** Complete  
**Date:** 2026-06-16  
**Branch:** transval  

---

## 1. Summary

`ToolboxTalk` carries two overlapping refresher-scheduling representations: the legacy `Frequency` enum (Once/Weekly/Monthly/Annually) and the newer `RequiresRefresher` + `RefresherIntervalMonths` pair. The two systems are in active drift. Every new-wizard talk creation (via `InitialiseToolboxTalkCommandHandler`) writes only to the new fields and leaves `Frequency` at its entity default of `Once`. Every new-wizard Step 4 save (`UpdateToolboxTalkSettingsCommandHandler`) writes only to the new fields. The old wizard's edit form (`ToolboxTalkForm.tsx` → `UpdateToolboxTalkCommandHandler`) writes only to `Frequency` and leaves `RequiresRefresher` / `RefresherIntervalMonths` at whatever they were — meaning an admin who opens a new-wizard talk in the old edit form and saves will either wipe the refresher configuration set in Step 4 (if `Frequency` is read by any runtime consumer) or produce a silent data inconsistency.

**Critical runtime exposure:** The `RefresherSchedulingService` and `CertificateGenerationService` — the two services with actual operational consequences — read exclusively from the **new fields**. The `GetToolboxTalkDashboardQueryHandler` produces a `talksByFrequency` breakdown that reads the legacy `Frequency` field from the ToolboxTalk entity; for new-wizard talks this will always show "Once" regardless of the refresher configuration in the new fields. The admin list filter by frequency will also be incorrect for new-wizard talks. The old edit form displays `Frequency` and submits it back to `UpdateToolboxTalkCommandHandler`, which writes it to the entity — this is the overwrite risk itemised in §5.27.

**Recommended direction:** Candidate A (mirror translation in `UpdateToolboxTalkCommandHandler`). The old edit form is active and reachable for any talk including new-wizard talks. Candidate B (full removal of `Frequency`) is the correct long-term direction but requires decommissioning the old wizard and the `ToolboxTalkSchedule`/schedule-processing Frequency column (which belongs to `ToolboxTalkSchedule`, not ToolboxTalk). Those are separate entities and the schedule Frequency is unambiguous — the scope of the conflict is ToolboxTalk.Frequency only. Candidate A (two-line fix in `UpdateToolboxTalkCommandHandler` + optional back-fill of existing rows) is safe, minimal, and unblocks §5.27.

**Critical finding — old wizard edit writes `RequiresRefresher = false` for new-wizard talks:**

`ToolboxTalkForm.tsx` `onSubmit` builds `requestData` at lines 276–299. The object does NOT include `requiresRefresher` or `refresherIntervalMonths`. The API endpoint receiving this is `PUT /api/toolbox-talks/{id}` which binds to `UpdateToolboxTalkCommand`. This command's properties have defaults (`RequiresRefresher = false`, `RefresherIntervalMonths = 12`). ASP.NET Core model binding will supply those defaults for the absent fields, overwriting whatever `UpdateToolboxTalkSettingsCommandHandler` set. The net result: any admin who edits a new-wizard talk via `/admin/toolbox-talks/talks/{id}/edit` and saves will **silently reset `RequiresRefresher = false`**, disabling any refresher the Step 4 configuration established. Certificate expiry will also be cleared.

## 5. Step 4 Findings — Current drift state per creation path

### New wizard (InitialiseToolboxTalk → UpdateToolboxTalkSettings)

| Field | State after creation | State after Step 4 save |
|---|---|---|
| `Frequency` | `Once` (entity default — never written by new wizard) | `Once` (still never written) |
| `RequiresRefresher` | `false` or `true` (from tenant `DefaultRefresherFrequency` mapping) | Correctly updated by Step 4 |
| `RefresherIntervalMonths` | 12 unless tenant default maps to other interval | Correctly updated by Step 4 |

Post-creation drift: `Frequency` is always `Once` regardless of what refresher configuration was applied. Dashboard frequency breakdown, admin list frequency filter, and all DTO `frequencyDisplay` fields are wrong for new-wizard talks with refresher enabled.

### Old wizard (CreateToolboxTalk / UpdateToolboxTalk)

| Field | State after creation | State after edit |
|---|---|---|
| `Frequency` | Set from `request.Frequency` (user-submitted) | Set from `request.Frequency` |
| `RequiresRefresher` | Entity default = `false` | Reset to `false` (default in missing field) |
| `RefresherIntervalMonths` | Entity default = `12` | Reset to `12` (default in missing field) |

Old-wizard talks: `RequiresRefresher` is always `false` after any edit. Refresher scheduling via `RefresherSchedulingService` will never fire for old-wizard talks. This is pre-existing behaviour from before Phase 5 — old-wizard talks never had RefresherSchedulingService wired in.

### Mixed path (new wizard create, then old wizard edit)

This is the §5.27 gating risk. After Step 4 of the new wizard sets `RequiresRefresher = true`, opening the old edit form and clicking Save will write `RequiresRefresher = false` (absent from `requestData`, defaults to `false` in the command). The refresher is silently disabled. The certificate expiry date is also cleared. `RefresherSchedulingService` will not schedule a refresher on completion.

## 6. Step 5 Findings — Enum mismatch mapping

### Legacy `ToolboxTalkFrequency` enum

| Enum value | Integer | Meaning |
|---|---|---|
| `Once` | 1 | One-time completion |
| `Weekly` | 2 | Every week |
| `Monthly` | 3 | Every month |
| `Annually` | 4 | Every year |

### New `RefresherFrequency` enum (wizard-internal, defined in `UpdateToolboxTalkSettingsCommand.cs`)

| String value | RequiresRefresher | RefresherIntervalMonths |
|---|---|---|
| `Once` | `false` | (preserved, no change) |
| `Monthly` | `true` | 1 |
| `Quarterly` | `true` | 3 |
| `Annually` | `true` | 12 |

### Tenant default string (`DefaultRefresherFrequency` on `ToolboxTalkSettings`)

Same four values as wizard `RefresherFrequency`: Once / Monthly / Quarterly / Annually.

### Round-trip mapping gaps

| Legacy → New | Result | Gap? |
|---|---|---|
| `Once` → | `RequiresRefresher = false` | No gap |
| `Monthly` → | `RequiresRefresher = true`, 1 month | No gap |
| `Weekly` → | No matching new value | **GAP**: Weekly has no equivalent in the new wizard's `RefresherFrequency`. A talk created via the old wizard with `Frequency = Weekly` cannot be faithfully represented in the new system. The inverse mapping for Candidate A would need to treat `Weekly` as an unmapped special case (map to `RequiresRefresher = true, RefresherIntervalMonths = 0` or round to 1 month). |
| `Annually` → | `RequiresRefresher = true`, 12 months | No gap |
| `Quarterly` → | `RequiresRefresher = true`, 3 months | No gap (new-only value, no legacy equivalent) |

### Summary of mismatch

- **Weekly exists in legacy but not in new**: cannot round-trip faithfully through the new system. The refresher scheduling system (`RefresherSchedulingService`) uses `RefresherIntervalMonths` which is an integer — there is no "weekly" concept; the minimum is 1 month. Weekly refreshers set in the old wizard are not actionable by the new refresher system.
- **Quarterly exists in new but not in legacy**: no `ToolboxTalkFrequency.Quarterly` enum value. A Candidate A mirror would need to map `Quarterly (3 months)` to `ToolboxTalkFrequency.Monthly` (closest available) or introduce a new enum value. The latter requires a migration.

## 7. Step 6 Findings — Fix-direction candidates

### Candidate A — Mirror translation in `UpdateToolboxTalkCommandHandler`

**Description:** When `UpdateToolboxTalkCommandHandler` writes `RequiresRefresher` and `RefresherIntervalMonths` (which it already does from the command DTO), also derive and write `Frequency` using an inverse mapping. When `InitialiseToolboxTalkCommandHandler` creates the talk, also set `Frequency` from the same mapping. Optionally run a one-time data fix on existing rows.

**Inverse mapping:**

| RequiresRefresher | RefresherIntervalMonths | → Frequency |
|---|---|---|
| `false` | any | `Once` |
| `true` | 1 | `Monthly` |
| `true` | 2 or 3 | `Monthly` (≤3 months rounds to Monthly; or add a migration to add `Quarterly` = 5 if desired) |
| `true` | 4–11 | `Monthly` or `Annually` (ambiguous range) |
| `true` | 12 | `Annually` |
| `true` | >12 | `Annually` |

**Pros:**
- Minimal: 2-3 lines in `UpdateToolboxTalkCommandHandler`, 3-5 lines in `InitialiseToolboxTalkCommandHandler`.
- Fixes §5.27 risk: old edit form reads and re-submits `Frequency` which is now derived from the canonical new fields.
- Fixes dashboard frequency breakdown for new-wizard talks.
- No migration needed (data fix is optional and low-risk).
- Does not require touching old wizard UI.

**Cons:**
- Frequency column remains redundant forever unless Candidate B follows.
- The mapping is lossy for `Weekly` (no new-field equivalent) and imprecise for mid-range intervals.
- Does not fix the overwrite risk from the old edit form unless the old form is also updated to include `requiresRefresher` in its submitted payload — or the handler is changed to derive `RequiresRefresher` from `Frequency` instead of trusting the DTO.
- Requires clarifying which field is canonical. If `Frequency` is kept as a derived display field, the handler must be explicit that new fields win.

**Actual fix for old-wizard overwrite:** To prevent the old edit form from writing `RequiresRefresher = false` on save, `UpdateToolboxTalkCommandHandler` needs to either: (a) ignore `RequiresRefresher`/`RefresherIntervalMonths` from the DTO and instead derive them from the incoming `Frequency`, or (b) preserve existing values when the DTO sends the default values. Option (a) makes `Frequency` the single write point for `UpdateToolboxTalkCommandHandler`, which is clean for the old-wizard path. Option (b) requires sentinel detection (fragile).

**Effort:** ~1 hour backend, no frontend changes, no migration.

### Candidate B — Full migration: remove Frequency from ToolboxTalk

**Description:** Drop `ToolboxTalk.Frequency` column. Remove all read and write sites. Migrate existing rows by converting their `Frequency` to `RequiresRefresher` + `RefresherIntervalMonths`. Update the old edit form to expose `RequiresRefresher` / `RefresherIntervalMonths` directly (toggle + interval input, same as CourseForm).

**Consumer site count for ToolboxTalk.Frequency:**
- 14 backend read sites in handlers/queries (see Step 1 table above)
- 1 backend write site in `UpdateToolboxTalkCommandHandler`
- 1 backend write site in `CreateToolboxTalkCommandHandler`
- 1 EF configuration entry
- 1 migration required
- Frontend: `ToolboxTalkFrequency` type, `frequency` / `frequencyDisplay` fields on 4+ TS interfaces, `ToolboxTalkForm.tsx` form field (the frequency select dropdown)

**`ToolboxTalkSchedule.Frequency` is NOT part of this scope** — it is a different entity with its own self-consistent Frequency column. That column drives the recurring schedule logic (`ProcessToolboxTalkScheduleCommandHandler`) and is correct.

**Migration data fix:** For existing old-wizard talks:
- `Frequency = Once` → `RequiresRefresher = false` (already entity default)
- `Frequency = Monthly` → `RequiresRefresher = true, RefresherIntervalMonths = 1`
- `Frequency = Annually` → `RequiresRefresher = true, RefresherIntervalMonths = 12`
- `Frequency = Weekly` → **unmappable**: no weekly interval in new system. Decision required: round to monthly (1) or disable refresher.

**Pros:**
- Eliminates the dual-representation permanently.
- Forces the old edit form to use the canonical representation.
- Dashboard breakdown becomes accurate if rewritten to use `RequiresRefresher`.

**Cons:**
- Large surface area: 14 read sites, frontend type system, old wizard form UI.
- Weekly → new system mapping is undefined — product decision needed.
- Dashboard `talksByFrequency` KPI would need redesigning (currently groups by Frequency enum; with that gone, grouping would be by "has refresher" or "interval bucket" — a different UX).
- Blocks on §5.27 old-wizard decommission decision.
- Requires a migration (data transformation) — risk per Note 28.
- Cannot be done while the old wizard is active and writing `Frequency`.

**Effort:** 1–2 days backend + frontend. High coordination cost.

### Candidate C — Hybrid: update old edit form to write new fields, keep Frequency as display-only

**Description:** Modify `ToolboxTalkForm.tsx` to add a refresher toggle + interval that maps to the new fields (mirroring the new wizard's `SettingsStep.tsx`). Keep `Frequency` as a derived display field computed from new fields. Update `UpdateToolboxTalkCommandHandler` to derive `Frequency` from the incoming new fields (not from `request.Frequency`). Remove the `Frequency` select from the old edit form.

**Pros:**
- Fixes the overwrite risk with minimal backend change.
- Old edit form gains feature parity with new wizard for refresher.
- Frequency becomes derivation-only — no semantic loss.

**Cons:**
- Frontend change to old wizard edit form (medium effort).
- Still requires removing the Frequency select from the form.
- Does not fully eliminate the dual-representation.

**Effort:** ~2 hours (frontend change) + 30 minutes (handler change).

## 8. Step 7 Findings — §5.27 Gating Risk Assessment

### Is the old edit form reachable for new-wizard talks?

**Yes.** The route `/admin/toolbox-talks/talks/{id}/edit` renders `ToolboxTalkForm.tsx` for any talk regardless of how it was created. The edit page fetches the talk by ID and passes it as the `talk` prop. There is no guard checking `LastEditedStep` or any new-wizard flag. A new-wizard talk (identifiable by `LastEditedStep != null`) is fully editable via the old form.

### What happens when an admin saves a new-wizard talk via the old edit form?

`ToolboxTalkForm.tsx` submits `requestData` (lines 276–299) which includes `frequency` but NOT `requiresRefresher` or `refresherIntervalMonths`. The `PUT /api/toolbox-talks/{id}` endpoint binds to `UpdateToolboxTalkCommand`. The absent `RequiresRefresher` defaults to `false`; absent `RefresherIntervalMonths` defaults to `12`. `UpdateToolboxTalkCommandHandler` writes these defaults, **overwriting the Step 4 configuration**. Specifically:

1. `RequiresRefresher` is set to `false` — any configured refresher is silently disabled.
2. `RefresherIntervalMonths` is set to `12` — the interval is reset.
3. `RefresherSchedulingService` will not schedule a refresher on talk completion.
4. `CertificateGenerationService` will compute `expiresAt = null` — the certificate will have no expiry.

### Does the old edit form write Frequency back?

Yes. The form includes a `frequency` select field (lines 430–454) defaulting to `talk.frequency` (line 154). The user sees `Once` (the entity default for all new-wizard talks) and, if they do not change it, submits `Frequency = Once`. `UpdateToolboxTalkCommandHandler` writes this back. Since `Once` was already the stored value, this particular write is a no-op — but if the user changes the frequency dropdown, it writes the new value.

### Is there any report/audit surface showing inconsistency to users?

Not visibly to admins. The `frequencyDisplay` on the admin talk list and detail views will show "Once" for all new-wizard talks even when `RequiresRefresher = true`. The dashboard frequency breakdown will show all new-wizard talks in the "Once" bucket. These are silent incorrect displays, not error states. There is no warning shown when opening the old edit form for a new-wizard talk.

### §5.27 gating: should this block the parallel-period toggle?

**Yes, per BACKLOG §5.27.** The entry explicitly states §5.24 (Frequency conflict) must be resolved before any tenant is toggled to new-as-default. The risk is: an admin using the old wizard to edit a new-wizard talk silently disables the refresher. In a parallel period where both wizards are in use, this is a realistic scenario. Candidate A (2 lines in `UpdateToolboxTalkCommandHandler` to stop trusting `request.RequiresRefresher` and instead derive from `request.Frequency`, plus mirror `Frequency` back in the new-wizard paths) is sufficient to unblock §5.27.

## 9. Adjacent Observations

1. **`InitialiseToolboxTalkCommandHandler` MapToDto hardcodes `FrequencyDisplay = "Once"`** (line 208). This is accidentally correct since `Frequency` is always `Once` for new-wizard talks, but it is fragile — it would be wrong if Candidate A derives `Frequency` from the new fields and a tenant default maps to Monthly or Annually.

2. **`GetToolboxTalkDashboardQueryHandler` selects `t.Frequency`** in the talk projection (line 25) specifically to build the frequency breakdown. For a tenant using only the new wizard, the breakdown will permanently show all talks as "Once". This is a visible data quality issue on the admin dashboard.

3. **`UpdateToolboxTalkSettingsCommand.cs` defines a local `RefresherFrequency` enum** with values Once/Monthly/Quarterly/Annually. This enum is file-scoped and not shared. The `InitialiseToolboxTalkCommandHandler` duplicates the mapping logic inline as a string switch. If the mapping changes, both sites must be updated. Consider extracting to a shared static method or utility class.

4. **`ContentCreationSessionService` also writes `RequiresRefresher`/`RefresherIntervalMonths`** (lines 1454–1465, 1740–1741) during the old content creation session publish step. These writes do not touch `Frequency` either — same drift pattern. This service is associated with the old content creation session wizard path, not the new learning wizard. Its writes are consistent with the new fields.

5. **`ToolboxTalkSchedule.Frequency` is entirely separate** and operates correctly. The schedule-processing job reads `schedule.Frequency` to determine recurrence of a _scheduled session_, not the refresher interval of a _talk content_. These are semantically different concepts. The schedule frequency determines how often new assignments are created; the talk refresher frequency determines how long after completion a new assignment should be created. These should not be confused.

6. **Weekly refresher is unreachable from both wizards** in practice. The old wizard exposes it as a `Frequency` option but the `RefresherSchedulingService` uses `RefresherIntervalMonths` (integer months). Since `Frequency.Weekly` is never mapped to `RefresherIntervalMonths`, a talk with `Frequency = Weekly` will have `RequiresRefresher = false` and `RefresherIntervalMonths = 12` (entity defaults) — `RefresherSchedulingService` will never fire for it. The Weekly option in the old form is effectively a no-op for refresher scheduling.

## 10. Recommended Direction

**Candidate A, with a targeted fix to the overwrite risk.**

The minimal viable fix has two parts:

**Part 1 — Stop the old edit form from overwriting new fields.**  
In `UpdateToolboxTalkCommandHandler.Handle()`: replace the current lines that write `request.RequiresRefresher` and `request.RefresherIntervalMonths` with a derivation from `request.Frequency`:

```csharp
// Derive RequiresRefresher + RefresherIntervalMonths from Frequency (old wizard's canonical field)
(toolboxTalk.RequiresRefresher, toolboxTalk.RefresherIntervalMonths) = request.Frequency switch
{
    ToolboxTalkFrequency.Monthly  => (true, 1),
    ToolboxTalkFrequency.Annually => (true, 12),
    ToolboxTalkFrequency.Weekly   => (true, 1),  // best approximation; Weekly has no months equivalent
    _                             => (false, toolboxTalk.RefresherIntervalMonths) // Once — preserve interval
};
```

This makes `Frequency` the single write point when the old wizard edits a talk. The new fields are always consistent with it.

# Phase 5.4 — SignalR gap fix

**Date:** 2026-06-11
**Branch:** transval
**Follows:** 5.4-translate-validate.md (original 5.4 report, unchanged as historical record)

---

## Summary

Two gaps surfaced during post-5.4 verification:

1. **SignalR not wired** — `useWorkflowSubscription.ts` was using React Query polling at 3 s instead of SignalR as the 5.4 prompt's §E1 specified. Polling misses intermediate `SectionCompleted` / `ValidationProgress` hub events and doesn't meet BACKLOG §1.3.5 (fire-and-notify) or §6.4.
2. **Auth gate convention break** — `POST /translations/{code}/start-translation` was gated `ToolboxTalks.Admin`; the surrounding workflow-mutation endpoints (`accept`, `cancel-external-review`, `initiate-external-review`) all use `Learnings.Manage`.

Both are corrected in this chunk.

## §6.4 Conformance — SignalR for long-running operations

**Status: Yes**

| Location | Evidence |
|----------|----------|
| `useWorkflowSubscription.ts:19-28` | `useQuery` with no `refetchInterval` — polling removed |
| `useWorkflowSubscription.ts:55-59` | `activeRunIds` derived from states in `Translating`/`Validating`; consumers render one `<WorkflowSubscriber />` per entry |
| `WorkflowSubscriber.tsx:22` | `useValidationHub(runId)` called — subscribes to `TranslationValidationHub` WebSocket |
| `WorkflowSubscriber.tsx:24-26` | `useEffect([isComplete])` → `onComplete(runId)` → `invalidate()` — query invalidated on `ValidationComplete` event |
| `WorkflowSubscriber.tsx:28-31` | `useEffect([completedSections.size, progress])` → `onSectionCompleted` — section progress surfaced from `SectionCompleted` / `ValidationProgress` hub events |
| `TranslateStep.tsx:29-32` | Destructures `activeRunIds`, `onValidationComplete`, `onSectionCompleted` from hook |
| `TranslateStep.tsx:67-73` | Renders `<WorkflowSubscriber key={runId} ... />` per active run |
| `ValidateStep.tsx:38-44` | `handleComplete` invalidates both `workflow-state` and `validationRuns` |
| `ValidateStep.tsx:118-125` | Renders `<WorkflowSubscriber key={runId} onComplete={handleComplete} ... />` per active run |

The previous polling implementation was a scope deviation from the 5.4 prompt §E1. The sub-component pattern (`WorkflowSubscriber`) is required because React hooks cannot be called conditionally inside a loop — one hub connection per runId must each be managed by its own component instance.

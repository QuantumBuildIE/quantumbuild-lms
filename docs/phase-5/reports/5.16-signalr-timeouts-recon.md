## 1. Reference Fix — `use-validation-hub.ts`

**File:** `web/src/features/toolbox-talks/hooks/use-validation-hub.ts`

The Phase 5.4 fix applies the timeout settings **after `.build()`, before `.start()`** by direct property assignment on the connection object. It is applied in **both** the initial build and the manual-reconnect fresh build:

**Initial connection (lines 63–73):**
```ts
const connection = new HubConnectionBuilder()
  .withUrl(`${API_BASE_URL}/api/hubs/translation-validation`, {
    accessTokenFactory: () => getStoredToken('accessToken') || '',
  })
  .withAutomaticReconnect(RECONNECT_DELAYS)
  .configureLogging(LogLevel.Warning)
  .build();

// Match server-side KeepAliveInterval / ClientTimeoutInterval set in Program.cs
connection.serverTimeoutInMilliseconds = 120_000;       // 2 min — matches server ClientTimeoutInterval
connection.keepAliveIntervalInMilliseconds = 10_000;    // 10 s — matches server KeepAliveInterval
```

**Manual reconnect fresh connection (lines 87–97):**
```ts
const fresh = new HubConnectionBuilder()
  .withUrl(`${API_BASE_URL}/api/hubs/translation-validation`, {
    accessTokenFactory: () => getStoredToken('accessToken') || '',
  })
  .withAutomaticReconnect(RECONNECT_DELAYS)
  .configureLogging(LogLevel.Warning)
  .build();

// Match server-side timeout settings
fresh.serverTimeoutInMilliseconds = 120_000;
fresh.keepAliveIntervalInMilliseconds = 10_000;
```

**Pattern:** Two property assignments after `.build()`, on the named connection variable. Applied to every `HubConnectionBuilder` call in the file (both initial and fresh/reconnect builds).

## 5. Timeout Values

| Setting | Value | Rationale |
|---------|-------|-----------|
| `serverTimeoutInMilliseconds` | 120_000 (2 min) | Default is 30 s. Server waits this long for a client ping before closing. Must exceed Railway's idle-kill window (~30 s) by a comfortable margin. 2 min is conservative. |
| `keepAliveIntervalInMilliseconds` | 10_000 (10 s) | Client sends a ping every 10 s. Ensures the Railway proxy sees activity every 10 s, well inside the 30 s idle threshold. |

**Fit for each use case:**
- **Subtitle processing (`use-subtitle-hub.ts`):** Can run 5–15 minutes for large videos. The 2-minute server timeout is not a job duration ceiling — as long as the keep-alive pings flow every 10 s, the server never times out. Values appropriate.
- **Corpus runs (`use-corpus-run-hub.ts`):** Can run 3–20 minutes depending on corpus size. Same analysis — keep-alive prevents idle kill. Values appropriate.
- **Lesson parser (`use-lesson-parser-hub.ts`):** Typically completes in under 2 minutes. Values are more than sufficient and there is no concern about them being too low.
- **Subtitle processing composite (`use-subtitle-processing.ts`):** Same as subtitle hub above. Values appropriate.

No hook needs different values. All four should receive the same 120_000 / 10_000 pair.

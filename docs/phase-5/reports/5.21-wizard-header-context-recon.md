# §5.21 Learning Wizard Page Header — Recon Report

**Date:** 2026-06-15  
**Scope:** Read-only investigation. No code changes.  
**Prompt:** BACKLOG §5.21 — "Learning wizard page header inherits wrong context"

---

## Summary

Two separate mechanisms produce the visible "page header" for admin routes: a subtitle in `admin/layout.tsx` and a breadcrumb in `toolbox-talks/layout.tsx`. The subtitle issue described in the BACKLOG ("Administration / Manage employees and users") is **not reproducible with current code** — the existing `pathname.startsWith('/admin/toolbox-talks')` conditional in `admin/layout.tsx:105` already covers the learnings routes and correctly produces "Manage learnings and training". The conditional was likely narrower or absent when the BACKLOG entry was written on 2026-06-10.

The **real remaining gap** is the breadcrumb in `toolbox-talks/layout.tsx:80-86`, which is hardcoded to render "Administration / Learnings" for all sub-routes with no leaf segment. For the learnings wizard routes, users see "Administration / Learnings" with no indication of which specific page they are on (Drafts, Quiz, Settings, etc.).

The recommended fix is a targeted extension of the breadcrumb block in `toolbox-talks/layout.tsx` to compute and render a leaf segment from the current pathname. Single-file change, closes both the leaf gap and any residual subtitle ambiguity.

---

## 1. Page-Header Mechanism

Two separate components contribute to what a user sees as the "page header":

### 1a. Admin layout subtitle — `admin/layout.tsx:99-108`

```tsx
<h1 className="text-2xl font-semibold tracking-tight">
  {isSupervisorOnly ? "Learning Management" : "Administration"}
</h1>
<p className="text-muted-foreground">
  {isSupervisorOnly
    ? "Manage schedules and learning reports"
    : pathname.startsWith('/admin/toolbox-talks')
      ? "Manage learnings and training"
      : "Manage employees and users"}
</p>
```

- Uses `usePathname()` from Next.js.
- For any path starting with `/admin/toolbox-talks` (which includes all learnings routes), it produces "Manage learnings and training".
- The fallback "Manage employees and users" fires for `/admin/employees`, `/admin/users`, etc.
- Single instance shared across all admin routes. No per-section override mechanism exists.

### 1b. Toolbox-talks breadcrumb — `toolbox-talks/layout.tsx:80-86`

```tsx
<div className="flex items-center gap-2 text-sm text-muted-foreground">
  <Link href="/admin" className="hover:text-foreground">
    Administration
  </Link>
  <span>/</span>
  <span className="text-foreground">Learnings</span>
</div>
```

- Hardcoded. Renders "Administration / Learnings" for every route under `/admin/toolbox-talks/`.
- No pathname check. No leaf segment. Every toolbox-talks sub-route (talks, courses, schedules, learnings wizard steps) renders the same two-level breadcrumb.

### 1c. WizardLayout page h1 — `features/toolbox-talks/components/learning-wizard/components/WizardLayout.tsx:43`

```tsx
<h1 className="text-2xl font-semibold tracking-tight">{title}</h1>
```

- Renders "New Learning" for all wizard step pages (steps 2–7). The `title` prop is hardcoded to `"New Learning"` in every page file.
- The drafts list page (`learnings/drafts/page.tsx`) does not use `WizardLayout`; it renders its own `<h1>Draft Learnings</h1>`.

## 3. Broken Route Trace

**Route:** `/admin/toolbox-talks/learnings/[talkId]/quiz`

**Layout nesting:**
```
(authenticated)/layout.tsx
  └── admin/layout.tsx            ← h1 "Administration" + subtitle
    └── toolbox-talks/layout.tsx  ← breadcrumb "Administration / Learnings" (no leaf)
      └── learnings/[talkId]/quiz/page.tsx → WizardLayout h1 "New Learning"
```

There is **no `layout.tsx` under `learnings/`** or `learnings/[talkId]/`. All 8 wizard routes inherit directly from `toolbox-talks/layout.tsx`.

**What the user actually sees (current code):**
1. Section subtitle: "Manage learnings and training" (from `admin/layout.tsx:105` — the path starts with `/admin/toolbox-talks`)
2. Admin nav tabs: Employees · Users · **Learnings** (active, since the special `isActive` rule in `toolbox-talks/layout.tsx:72-73` marks the Learnings tab active for `/admin/toolbox-talks/learnings/*`)
3. Breadcrumb: "Administration / Learnings" — no leaf segment
4. Inner nav tabs: Overview · **Learnings** (active) · Courses · Schedules · …
5. WizardLayout h1: "New Learning"
6. Step indicator showing which step is current

**Root cause of BACKLOG description vs current code discrepancy:**  
The BACKLOG entry was written 2026-06-10 and states the subtitle shows "Manage employees and users". This is not reproducible with the current code. The `pathname.startsWith('/admin/toolbox-talks')` conditional now covers learnings routes. Most likely the conditional was added after the entry was written (possibly during an earlier Phase 5 chunk) but the entry was never marked done.

**The remaining actual gap (confirmed):**  
The breadcrumb at `toolbox-talks/layout.tsx:80-86` is hardcoded and does not render a leaf segment. For learnings wizard routes, users see "Administration / Learnings" with no indication of context ("Drafts", "New Learning", "Quiz", etc.).

## 5. Fix-Direction Assessment

### Candidate A — Extend the `toolbox-talks/layout.tsx` breadcrumb (Recommended)

Add pathname-based leaf detection to the existing breadcrumb block. The breadcrumb renders inside the layout that already runs `usePathname()`. A segment-to-label map covers the learnings routes and can be extended to other toolbox-talks sections if wanted.

```tsx
// Segment-to-label map (learnings routes; extend for other sections as needed)
const leafLabels: Record<string, string> = {
  drafts: "Drafts",
  new: "New Learning",
  parse: "Parse",
  quiz: "Quiz",
  settings: "Settings",
  translate: "Translate",
  validate: "Validate",
  publish: "Publish",
};

// Derive leaf from the segment after /learnings/ (or /learnings/[talkId]/)
const segments = pathname.split('/');
const learningsIdx = segments.indexOf('learnings');
const rawLeaf = learningsIdx >= 0
  ? segments[learningsIdx + 1] === '[talkId]' || isUuid(segments[learningsIdx + 1])
    ? segments[learningsIdx + 2]
    : segments[learningsIdx + 1]
  : null;
const leaf = rawLeaf ? leafLabels[rawLeaf] : null;
```

Then in the breadcrumb JSX:
```tsx
{leaf && (
  <>
    <span>/</span>
    <span className="text-foreground">{leaf}</span>
  </>
)}
```

**Code surface:** `toolbox-talks/layout.tsx` — one file, ~15-20 lines added.  
**Feasibility:** High. The layout already has `usePathname()` and runs on every navigation. The UUID segment (`[talkId]`) must be skipped when extracting the leaf — a simple `isUuid()` guard or checking that the segment is not in the `leafLabels` map handles this.  
**Side effects:** Other toolbox-talks sub-routes (talks/[id], courses/[id], etc.) are unaffected unless extended — the leaf derivation is scoped to the `learnings` path segment check.  
**Future-proofing:** Adding a new learnings wizard route only requires adding an entry to the `leafLabels` map. No structural change.  
**Closes breadcrumb gap:** Yes, for all 8 learnings routes.  
**Closes subtitle gap:** Not needed — subtitle is already correct per current code. If there's any residual doubt, a separate line in the map or a narrower `pathname.startsWith('/admin/toolbox-talks/learnings')` guard in `admin/layout.tsx` would make it explicit, but the current general guard covers it.

### Candidate B — Add a dedicated `learnings/layout.tsx`

Create `web/src/app/(authenticated)/admin/toolbox-talks/learnings/layout.tsx` that renders its own breadcrumb with a leaf segment, entirely replacing the `toolbox-talks/layout.tsx` breadcrumb for learnings routes.

**Feasibility:** High. Next.js layout nesting means `learnings/layout.tsx` would render inside `toolbox-talks/layout.tsx` — the `toolbox-talks` breadcrumb ("Administration / Learnings") and the `learnings` breadcrumb would both render, stacking two breadcrumbs on screen. To avoid this, the `learnings/layout.tsx` would either need to suppress the parent breadcrumb (not straightforward in React server components / layout nesting) or the parent layout would need to conditionally hide its own breadcrumb for learnings paths. This makes Candidate B messier than Candidate A without adding architectural benefit.

**Code surface:** New file + conditional suppression change in `toolbox-talks/layout.tsx`. More code, more coupling.  
**Side effects:** Breadcrumb behaviour diverges between learnings and other toolbox-talks sections, potentially inconsistently.  
**Recommended against** in favour of Candidate A.

### Candidate C — Use step-specific titles in `WizardLayout` / per-page h1

Change `title="New Learning"` in each page to `title="New Learning — Quiz"`, `title="New Learning — Settings"`, etc.

**Feasibility:** High, trivially.  
**What it closes:** The WizardLayout h1 would be more descriptive at the page level, but this does not fix the breadcrumb (which is rendered by the layout, not the page). Does not close the §5.21 gap as described.  
**Standalone value:** Low — the StepIndicator already shows which step the user is on. The h1 "New Learning" is a reasonable wizard-wide title.  
**Recommended against** as the primary fix, though adjusting the `title` prop is optional polish.

## 6. Breadcrumb Gap

**Location:** `toolbox-talks/layout.tsx:80-86`  
**Current output:** "Administration / Learnings" (hardcoded, two levels only)  
**Why the leaf is missing:** The breadcrumb block does not inspect `pathname`. It renders the section label ("Learnings") unconditionally with no mechanism to append a route-specific leaf.

**Scope:** The gap affects all toolbox-talks routes (talks/[id], courses/[id], schedules/[id], etc. also lack a leaf) but §5.21 specifically flags the learnings routes because they have distinct pages with distinct purposes. The existing toolbox-talks section pages are less affected because their inner nav tabs provide context.

**Does the same change close the subtitle gap?**  
No change is needed to close the subtitle gap — the current `admin/layout.tsx` conditional already handles it. The breadcrumb leaf is an independent fix that only touches `toolbox-talks/layout.tsx`. Both gaps trace to the same root (no pathname-aware context propagation in the toolbox-talks layout), but the subtitle gap appears already resolved.

---

## 7. Recommended Direction

**Candidate A: extend `toolbox-talks/layout.tsx` breadcrumb with a learnings-scoped leaf map.**

Reasoning:
- Single-file change in a file that already has `usePathname()`.
- Matches the existing codebase pattern — layouts use `pathname` for active-tab detection (lines 66-76 in the same file).
- Does not require a new layout file or conditional suppression logic.
- The leaf map is minimal, explicit, and easy to extend when new learnings routes are added.
- No duplication of the breadcrumb block.
- Closes the §5.21 breadcrumb gap cleanly.
- The subtitle issue in `admin/layout.tsx` is already resolved and requires no change.

The breadcrumb leaf and the subtitle are **independent** concerns — the subtitle conditional is in `admin/layout.tsx`, the breadcrumb leaf is in `toolbox-talks/layout.tsx`. The recommended fix only touches the latter.

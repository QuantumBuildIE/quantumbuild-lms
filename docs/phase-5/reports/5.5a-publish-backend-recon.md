## 4. Response Convention Gap (talkId vs sessionId)

The existing session-based publish endpoint (`ContentCreationSessionsController`) takes a `sessionId` and an explicit `talkId` in the body. The new wizard endpoint must operate purely by `talkId` without a session.

**Decision:** New dedicated action `PublishByTalkId` in `ToolboxTalksController`. Controller action fires `PublishToolboxTalkCommand(TalkId, TenantId)`, returns `PublishTalkResult` directly (no `Result<T>` wrapper — matches new endpoint convention).

---

## 5. Reachability Gate

**Product decisions (confirmed from spec):**
- `talk.Sections.Count > 0` — section presence required
- If `TargetLanguageCodes` is empty → skip translation gate (English-only talks publish freely)
- If `TargetLanguageCodes` is non-empty → at least one `TranslationValidationRun` with `Status == Completed` must exist for this `talkId` (any language qualifies; the gate is presence-based, not per-language)
- `AwaitingThirdParty` translations do **not** block publish (frontend concern only)

**Pre-publish status:** The new wizard keeps talks in `Draft` throughout Steps 1–6. Handler allows publish from any non-`Published` status (does not restrict to `Draft` only — covers edge cases where a talk might have been previously set to `ReadyForReview`).

## 6. Post-Publish Side Effect

**File:** `src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Infrastructure/Jobs/RequirementMappingJob.cs`

The requirement mapping job is fire-and-forget. Per CLAUDE.md Note 21 it must be enqueued via the concrete class, not an interface.

```csharp
BackgroundJob.Enqueue<RequirementMappingJob>(job =>
    job.MapRequirementsAsync(tenantId, talkId, null, CancellationToken.None));
```

Enqueue site: controller (not command handler) to keep Application layer clean from Hangfire dependency.

## 10. Post-publish Translation Editing

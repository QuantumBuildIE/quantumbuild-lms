# §5.22 — TenantQueryInvalidator parent-path redirect: Recon Report

**Date:** 2026-06-15  
**Type:** Read-only investigation  
**Output of:** BACKLOG §5.22 recon prompt  
**Status:** Complete — no code changes made

---

## Summary

The bug shape matches BACKLOG §5.22 verbatim, with one important update: the §5.3b
workaround (`/admin/toolbox-talks/learnings/page.tsx` redirecting to `/drafts`) has
eliminated the only current 404 case for admin routes. As of this investigation, the
known 404-producing gap (`/toolbox-talks/courses` missing its landing page) is
unreachable by SuperUsers due to an independent layout guard in `toolbox-talks/layout.tsx`
that redirects SuperUsers away from all employee-facing routes. However, **the structural
fragility remains**: the logic assumes any UUID route's parent is a valid page, and the
workaround must be applied manually to every new UUID route added under a parentless path.

The more significant observable consequence today is **UX context loss on browser
refresh**: a SuperUser who refreshes on any learnings wizard step (e.g.
`/admin/toolbox-talks/learnings/{talkId}/quiz`) loses their step position — the
invalidator fires during hydration, strips to the learnings landing page, and the landing
page redirects to `/drafts`. The user then has to resume from the drafts list.

## A. Current Behavior

**File:** `web/src/lib/providers.tsx:13-37`

```tsx
// providers.tsx:11
const UUID_REGEX = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;

// providers.tsx:13-37
function TenantQueryInvalidator() {
  const { activeTenantId } = useAuth();
  const queryClient = useQueryClient();
  const previousTenantId = useRef(activeTenantId);      // :16
  const pathname = usePathname();
  const router = useRouter();

  useEffect(() => {
    if (previousTenantId.current !== activeTenantId) {  // :21 — ref comparison
      previousTenantId.current = activeTenantId;
      queryClient.invalidateQueries();                   // :23 — always fires

      const segments = pathname.split("/");
      const uuidIndex = segments.findIndex((s) => UUID_REGEX.test(s));  // :28 — FIRST UUID only
      if (uuidIndex > 0) {
        const parentPath = segments.slice(0, uuidIndex).join("/");
        router.replace(parentPath);                      // :31 — conditional on UUID found
      }
    }
  }, [activeTenantId, queryClient, pathname, router]);   // :34

  return null;
}
```

**Key behavioral facts:**

1. **Trigger:** `previousTenantId.current !== activeTenantId`. The ref is initialised to
   the value of `activeTenantId` at mount time (`useState(null)` → `null`). Any change
   after mount fires the effect.

2. **UUID detection finds the first UUID segment only.** For
   `/admin/toolbox-talks/talks/{uuid1}/validation/{uuid2}`, `uuidIndex` is the position
   of `{uuid1}`, and `parentPath` is `/admin/toolbox-talks/talks` — not the intermediate
   validation path.

3. **Query invalidation is unconditional.** `queryClient.invalidateQueries()` fires on
   every tenant change regardless of the current path. The UUID redirect is conditional
   (`if (uuidIndex > 0)`).

4. **No short-circuit conditions.** The effect runs for any tenant transition, on any
   path, for any user. The only gate is the ref comparison.

5. **`pathname` in the dependency array** is a code smell: `pathname` is read as data
   inside the effect body, not as a trigger for action. Including it causes the effect to
   re-run on every navigation, though the ref guard makes those re-runs no-ops. Harmless
   but worth noting.

6. **No loop risk.** After the redirect fires, `pathname` changes and the effect re-runs.
   But `previousTenantId.current` was already updated to `activeTenantId` in the first
   run, so `previousTenantId.current === activeTenantId` → condition fails → no second
   redirect.

## B. Trigger Surface — When Does `activeTenantId` Change?

`activeTenantId` is `useState<string | null>(null)` in
`web/src/lib/auth/auth-context.tsx:43`. Three mutation sites exist:

### B.1 Initial hydration on page mount

**File:** `auth-context.tsx:110-143` (`initAuth()` inside `useEffect([], [loadUser])`)

```tsx
// auth-context.tsx:113-115
const storedTenantId = typeof window !== "undefined"
  ? localStorage.getItem("activeTenantId")
  : null;
if (storedTenantId) {
  setActiveTenantIdState(storedTenantId);  // :115 — null → storedUUID
}
```

This runs on **every page mount**, including browser refresh. If a SuperUser has a
stored `activeTenantId` in localStorage, this triggers the `null → storedUUID`
transition on every load.

**TenantQueryInvalidator fires:** Yes — `previousTenantId.current = null`,
`activeTenantId` becomes the UUID → condition met on next render.

**Most likely URL:** Any page the user was on at their last session, including wizard
step routes like `/admin/toolbox-talks/learnings/{talkId}/quiz`.

**This is the root cause of the "browser refresh loses step context" observation** noted
in BACKLOG §5.22 and also documented as a §5.20-adjacent issue.

### B.2 TenantSwitcher user action

**File:** `web/src/components/layout/tenant-switcher.tsx:25-31`

```tsx
const handleChange = (value: string) => {
  if (value === ALL_TENANTS_VALUE) {
    setActiveTenantId(null);
  } else {
    setActiveTenantId(value);
  }
};
```

**TenantQueryInvalidator fires:** Yes — intentional, the original design target.  
**Most likely URL:** Any admin page the SuperUser was on.

### B.3 Logout

**File:** `auth-context.tsx:206`

```tsx
const logout = useCallback(() => {
  clearStoredTokens();
  setAuthState({ user: null, token: null, isLoading: false, dpaAccepted: false });
  setActiveTenantId(null);  // :206 — storedUUID → null
}, [setActiveTenantId]);
```

**TenantQueryInvalidator fires:** Yes — but benign. On logout the user is being routed
to `/login` anyway; the redirect from the invalidator is immediately superseded.

### B.4 Login

No explicit `setActiveTenantId` on login. The tenant is restored only through the B.1
path (localStorage hydration on next mount). A freshly-logged-in SuperUser goes through
B.1 on the first render after `initAuth` completes.

## D. §5.3b Workaround Analysis

**File:** `web/src/app/(authenticated)/admin/toolbox-talks/learnings/page.tsx`

```tsx
import { redirect } from "next/navigation";

export default function LearningsIndexPage() {
  redirect("/admin/toolbox-talks/learnings/drafts");
}
```

**What it does:** A Next.js server-side redirect. Any GET to
`/admin/toolbox-talks/learnings` is immediately redirected to `/admin/toolbox-talks/learnings/drafts`. There is no rendered UI.

**How it resolves the 404:** Before Phase 5.3b, `/admin/toolbox-talks/learnings` had no
`page.tsx`. When a SuperUser refreshed on e.g. `/admin/toolbox-talks/learnings/{talkId}/quiz`,
the invalidator redirected to `/admin/toolbox-talks/learnings` → Next.js returned 404.
Adding the `page.tsx` gives that path a valid Next.js route.

**Universality:** The workaround is universal — any URL under `/admin/toolbox-talks/learnings`
with a UUID segment will redirect to that same parent, which then redirects to drafts.
It covers all six wizard step routes and any future routes added under `[talkId]/`.

**Limitation — context loss:** The workaround prevents 404 but **not context loss**. A
SuperUser refreshing on the Validate step lands at the drafts list. Per PHASE_5_STANDARDS
§5.4, "the user can refresh on any step, and within one second of the data fetching, the
UI is back where it was." The workaround does not meet that standard for SuperUser
accounts.

**Pattern applicability:** The pattern works for any UUID route whose parent path is
under a predictable common segment (`/admin/toolbox-talks/learnings`). It would need to
be applied manually to every new UUID-bearing route group where the parent currently
lacks a page. It is a mitigation, not a structural fix.

## E. Fix-Direction Assessment

### E.1 Candidate 1 — Known-safe fallback on unresolvable parent

When `router.replace(parentPath)` would point to a route with no page, fall back to a
known-good destination (e.g. `/dashboard`, or derived from path prefix).

**Implementation sketch:**
```tsx
const SAFE_FALLBACK = "/dashboard";

// After computing parentPath:
const safePath = KNOWN_PARENT_PATHS.has(parentPath) ? parentPath : SAFE_FALLBACK;
router.replace(safePath);
```

Or a prefix-based heuristic:
```tsx
const safePath = parentPath.startsWith("/admin") ? "/admin/toolbox-talks" : "/dashboard";
```

**Feasibility:** Medium. The heuristic approach is ~10 lines but requires a judgement
call about what a "safe" fallback is. `/dashboard` is universally routable. The prefix
approach is brittle if a new top-level section is added without updating the heuristic.

**Side effects:** User still loses context on any tenant transition that fires on a UUID
route — they land at the fallback instead of 404, which is better, but they've still
been navigated away from their work.

**Code surface:** 8–15 lines in `providers.tsx`.

**Interaction with §5.20 (Refresh Amendment):** No improvement. The redirect still fires
on initial hydration, so SuperUsers who refresh on a wizard step are sent to the
fallback. §5.20's "refresh stays at current step" goal is still violated for SuperUsers.

**Interaction with §5.22 (the problem being solved):** Stops future 404s if the fallback
is chosen well. Does not address the structural fragility.

---

### E.2 Candidate 2 — Manifest of valid parent paths

Maintain a static `Set<string>` of routable parent paths. Before redirecting, check the
set; if parentPath is not in it, skip the redirect (or fall back to §5.20 candidate).

**Implementation sketch:**
```tsx
const ROUTABLE_PARENTS = new Set([
  "/admin/employees",
  "/admin/regulatory",
  "/admin/regulatory/system",
  "/admin/tenants",
  "/admin/toolbox-talks/courses",
  "/admin/toolbox-talks/learnings",
  "/admin/toolbox-talks/schedules",
  "/admin/toolbox-talks/talks",
  "/admin/users",
]);

// In effect:
if (uuidIndex > 0) {
  const parentPath = segments.slice(0, uuidIndex).join("/");
  if (ROUTABLE_PARENTS.has(parentPath)) {
    router.replace(parentPath);
  }
  // else: skip redirect; query invalidation already happened
}
```

**Feasibility:** Low-medium. The set is easy to write today but it must be kept in sync
with the route tree manually — it's a second source of truth for what the router already
knows. Missing an entry means a future route quietly skips the redirect without error;
a stale entry redirects to a page that has since been removed.

**Side effects:** Same context-loss problem as Candidate 1 for paths that ARE in the
manifest. For paths not in the manifest, the invalidator silently skips the redirect
(query invalidation still fires) — the user stays on the current page with stale data
briefly visible, then React Query refetches.

**Code surface:** ~30 lines (manifest declaration + check in `providers.tsx`).

**Interaction with §5.20:** No improvement. Redirect still fires for manifested paths.

### E.3 Candidate 3 — Remove redirect; query invalidation only

Delete lines `providers.tsx:27-32` entirely. Retain line `23`
(`queryClient.invalidateQueries()`).

**Before:**
```tsx
if (previousTenantId.current !== activeTenantId) {
  previousTenantId.current = activeTenantId;
  queryClient.invalidateQueries();                   // keep this

  const segments = pathname.split("/");
  const uuidIndex = segments.findIndex((s) => UUID_REGEX.test(s));
  if (uuidIndex > 0) {
    const parentPath = segments.slice(0, uuidIndex).join("/");
    router.replace(parentPath);                      // remove these 6 lines
  }
}
```

**What the original redirect protected against:** Brief window where a SuperUser
is viewing tenant-A's data and switches to tenant-B — the old tenant's data is visible
for the duration of the React Query refetch. The redirect was a defensive guard to
take the user off the stale-data page before they could act on it.

**Worst case without the redirect:** SuperUser is on
`/admin/toolbox-talks/talks/{tenantA-uuid}` (talk detail) and switches to tenant-B. The
queries invalidate and refetch. The API returns 404 (talkA doesn't exist in tenantB).
The page renders its error state (per §6.2 of PHASE_5_STANDARDS: every fetch has an
explicit error state with "what failed" and "what to do"). The user sees "Couldn't load
this talk" with a "Back to talks" link and clicks it.

The stale-data window (between query invalidation and the new data arriving) is typically
less than 500ms on a fast connection. React Query shows the stale data while revalidating
by default (`staleWhileRevalidate`), then updates. For most pages this means brief
flicker of wrong-tenant data, then either correct data or an error state.

**Feasibility:** High. Three lines removed from one file. No new logic needed.

**Side effects:**
- Stale-while-revalidate briefly shows tenant-A data after switching to tenant-B.
  Duration: ~200-500ms on a typical connection.
- Pages without proper §6.2 error states (fetch returns 404/403, component doesn't
  handle it gracefully) could show broken or empty UI until the user navigates away.
  This is a pre-existing gap in those pages, not introduced by this fix.
- On initial hydration (browser refresh), the redirect no longer fires at all. SuperUsers
  who refresh on a UUID page stay on that page — the talk fetches with the restored
  tenant ID and either loads (if it's the right tenant) or shows an error state (if the
  stored tenant ID doesn't match this talk's tenant).

**Code surface:** Remove lines `providers.tsx:27-32`. The `const segments`, `const
uuidIndex`, `if (uuidIndex > 0)`, `const parentPath`, `router.replace()` block — 6
lines removed. Also remove `pathname` and `router` from the `useEffect` dependency array
(`providers.tsx:34`) since they'd no longer be used.

**Interaction with §5.20 (Refresh Amendment):** **Positive.** The refresh redirect that
was the primary UX problem for SuperUsers is removed. A SuperUser who refreshes on the
Validate wizard step stays on the Validate step. This partially addresses §5.20's intent
for SuperUsers (who were previously the only users affected by the redirect).

**Interaction with §5.22:** Eliminates the root cause entirely. No redirect → no 404
risk. No manifest to maintain. Future routes don't require workaround pages.

## F. Adjacent Observations

### F.1 Admin layout has its own `activeTenantId` guard (parallel, not conflicting)

`web/src/app/(authenticated)/admin/layout.tsx:52-58`:

```tsx
useEffect(() => {
  const isNonTenantScoped = nonTenantScopedPaths.some(path =>
    pathname.startsWith(path)
  );
  if (isSuperUser && !activeTenantId && !isNonTenantScoped) {
    router.replace("/admin/tenants");
  }
}, [isSuperUser, activeTenantId, pathname, router, nonTenantScopedPaths]);
```

This fires when `activeTenantId` goes to null (no tenant selected — "All Tenants" in
the switcher) while the SuperUser is on a tenant-scoped admin path. It redirects to the
tenant list. This is a separate mechanism from `TenantQueryInvalidator`. They fire
together on logout (`null → null` doesn't trigger either guard; actually `storedUUID →
null` triggers this layout guard). They don't conflict: the invalidator fires on any
tenant change, this layout fires only on the null case.

### F.2 Toolbox-talks layout gates SuperUsers out entirely

`web/src/app/(authenticated)/toolbox-talks/layout.tsx:40-44`:

```tsx
useEffect(() => {
  if (isSuperUser) {
    router.replace("/admin/tenants");
  }
}, [isSuperUser, router]);
```

This means `/toolbox-talks/courses/{id}` (which lacks a parent page) is not a
reachable surface for SuperUsers. The parent-page gap is real but harmless in
practice today.

### F.3 Interaction between §5.22 and §5.20

§5.20 (Refresh Amendment) says "refresh on any `/learnings/{talkId}/{step}` route lands
on `/drafts` rather than the step the user was on." This is a consequence of the current
`TenantQueryInvalidator` redirect firing during hydration. Removing the redirect
(Candidate 3) would **automatically fix the §5.20 case for SuperUsers** — the most
important part of that gap. Regular admins (whose `activeTenantId` is always null and
never changes) are already unaffected by the invalidator; for them, the §5.20 refresh
behavior would need investigation elsewhere (auth layout guards or session restoration).

### F.4 `pathname` in the dependency array is unnecessary if Candidate 3 is chosen

If the redirect block is removed, `pathname` and `router` are no longer used inside the
effect body. They should be removed from the dependency array (`providers.tsx:34`) to
avoid spurious re-runs of the effect on every navigation. Only `activeTenantId` and
`queryClient` remain as meaningful dependencies.

### F.5 One more route tree gap: `/admin/toolbox-talks/courses/[id]` has no `page.tsx`

The agent survey found that `/admin/toolbox-talks/courses/[id]/` has no `page.tsx` (only
`/edit` and `/validation/[runId]` are routable). However, the invalidator's first-UUID
logic means routes under this segment redirect to `/admin/toolbox-talks/courses`, not to
`/admin/toolbox-talks/courses/{id}`. The non-existent `courses/[id]/page.tsx` is not
directly implicated by the invalidator.

## Recommended Direction

**Candidate 3 — remove the redirect block entirely.**

Rationale:
1. **The current codebase has no actively broken 404 paths.** The §5.3b workaround
   has patched the only case that was blowing up. There is no urgency to patch 404s
   that don't currently exist.
2. **The primary remaining symptom is context loss on refresh.** SuperUsers who refresh
   on a wizard step lose their step position. This is the concrete user-facing harm.
   Candidates 1 and 2 don't address it; Candidate 3 eliminates it as a side effect.
3. **The protective value of the redirect is low.** The stale-data window without the
   redirect is sub-second, React Query already invalidates all queries, and the worst
   case is a 404 error state on a page that should have one anyway per §6.2.
4. **Candidates 1 and 2 preserve fragility.** They require ongoing maintenance
   (fallback heuristics or manifests) with no principled guarantee that they stay
   correct as the route tree evolves.
5. **Code surface is three lines removed in one file.** Lowest-risk possible change.
6. **§5.20 partial improvement is a side effect.** Not the goal of §5.22's fix, but a
   meaningful bonus.

**One pre-condition worth checking before implementation:** Audit the pages that would
be most exposed to the "stale data briefly visible, then 404" path — specifically
`/admin/toolbox-talks/talks/[id]` and any new wizard steps — and confirm they have
§6.2-compliant error states (a "Couldn't load this" error with a back link). If they
do, the worst-case user experience is a one-second error flash before the user clicks
back. If they don't, the error state gap should be patched in the same chunk.

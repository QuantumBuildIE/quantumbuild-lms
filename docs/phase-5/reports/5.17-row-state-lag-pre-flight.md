## 2. Old Wizard vs New Wizard Path

**Distinguishing field:** `run.IsNewWizard` (bool on `TranslationValidationRun`).

**New wizard (`IsNewWizard = true`):**
- Translation generated inline by `GenerateTranslationForSectionsAsync`
- `RecordTranslationCompleted` called → state becomes `AIGenerated`
- `StartValidation` call is valid from `AIGenerated`
- **This is the path the fix applies to**

**Old wizard (`IsNewWizard = false`):**
- Translation already exists before the job runs; `LoadSectionsAsync` returns early from `GenerateTranslationForSectionsAsync`
- `RecordTranslationCompleted` is NOT called (guarded by `if (isNewWizard)`)
- State stays `Translating` (set by the old wizard's `StartTranslation` HTTP call)
- `StartValidation` from `Translating` would fail with `WorkflowInvalidState`
- **This path should NOT call `StartValidation`**

**Guard used in fix:** `run.ToolboxTalkId.HasValue && run.IsNewWizard`

## 3. `StartValidation` Preconditions and Idempotency

**Source:** `TranslationWorkflowService.StartValidation` at line 263–289.

**Legal source states:** `AIGenerated` or `Validating` (comment: "re-entrant calls when the job restarts after a crash").

**Idempotency when already `Validating`:** NOT idempotent — writes a second `ValidationStarted` event. State stays `Validating`. Functionally harmless; duplicate audit event is the documented design for retry scenarios.

**Decision:** No idempotency check added. The extra round-trip is unnecessary since the normal path enters from `AIGenerated`, and the retry path (entering from `Validating`) is the intended use case per the method's comment.

# §5.27 Cutover Toggle — Recon Report

**Date:** 2026-06-16  
**Scope:** Read-only investigation. No code changes.  
**Purpose:** Validate §5.27 design decisions against current codebase; locate touch points for each implementation chunk; surface surprises.

---

## Summary

The §5.27 locked design (tenant-level toggle + URL param override, in-flight drafts stay with creating wizard, manual cutover) is **sound and fits the codebase as it stands**. No design decisions need revisiting.

Two shifts from what the §5.27 entry implicitly assumed when written:

1. **`lastEditedStep` is the right discriminator for in-flight drafts** — new-wizard talks set it from Step 1; old-wizard talks never set it. The drafts list at `/admin/toolbox-talks/learnings/drafts` already fetches all Draft-status talks (not just new-wizard), so no backend query change is needed — only the Resume routing and a visual badge.

2. **The toggle belongs in `TenantSettings` (key-value store), not `ToolboxTalkSettings` (typed entity)** — this is the same surface used by `SkipValidationStep` and `QrLocationTrainingEnabled`, requires no migration, and follows an established pattern that already has a working settings UI in the General tab.

The fix is frontend-heavy (4 of 5 chunks are frontend-only). The backend work is one line in `TenantSettingKeys.cs` and one line in `TenantSettingsService.GetAllSettingsAsync`.

## Step 3 — Toggle storage and configuration surface

### Two tenant-level settings patterns exist

**Pattern A — `TenantSettings` key-value store** (correct surface for the toggle)

- Entity: `TenantSetting` rows with `{TenantId, Key, Value}` string pairs
- Service: `TenantSettingsService` — `GetAllSettingsAsync` / `SetSettingAsync`
- Keys: `TenantSettingKeys.cs` — `src/Core/QuantumBuild.Core.Application/Features/TenantSettings/TenantSettingKeys.cs`
- Frontend read: `useTenantSettings()` → `settings?.['KEY'] === 'true'`
- Frontend write: `useUpdateTenantSettings()` → `{ settings: [{ key, value }] }`
- Existing examples: `SkipValidationStep` (old wizard reads it at `CreateWizard.tsx:170`), `QrLocationTrainingEnabled`
- **No migration needed** — `TenantSettings` table already exists

**Pattern B — `ToolboxTalkSettings` typed entity** (wrong surface)

- Entity: `ToolboxTalkSettings` — one row per tenant, typed C# fields
- Every new field requires a migration
- Used for: wizard step-4 defaults (video watch %, due days, certificate, refresher, active)
- **Not appropriate** for the wizard toggle — routing preference is not a content config value

**Decision confirmed: use Pattern A (`TenantSettings` key-value store).**

### Proposed new key

```csharp
// TenantSettingKeys.cs
public const string UseNewWizard = "UseNewWizard";
```

Default value: `"false"` (old wizard is default, per §5.27 design decision 1).

Add to `TenantSettingsService.GetAllSettingsAsync` defaults dictionary:
```csharp
result[TenantSettingKeys.UseNewWizard] = "false";
```

### Existing `SkipValidationStep` is the model

`SkipValidationStep` in `CreateWizard.tsx:170`:
```typescript
const skipValidationStep = tenantSettings?.['SkipValidationStep'] === 'true';
```

The wizard toggle hook (`useWizardPreference`) will follow this same pattern. The `useTenantSettings()` hook caches the result via TanStack Query, so there is no extra network call if the hook is called in multiple components.

## Step 4 — In-flight draft handling

### Discriminator confirmed: `lastEditedStep`

- `ToolboxTalk.LastEditedStep` (`int?`) — `src/Modules/ToolboxTalks/.../Domain/Entities/ToolboxTalk.cs:254`
- `ToolboxTalkListItem.lastEditedStep: number | null` — `web/src/types/toolbox-talks.ts:177`
- Projected in `GetToolboxTalksQueryHandler.cs:85`

**Rule:**
- `lastEditedStep != null` → created by new wizard
- `lastEditedStep == null` → created by old wizard (or any pre-Phase-5 path)

This is reliable. The new wizard calls `InitialiseToolboxTalk` at Step 1 which sets `lastEditedStep`. The old wizard never calls this endpoint.

### Current drafts list behaviour

`useDraftsList` (`web/src/features/toolbox-talks/components/learning-wizard/hooks/useDraftsList.ts:11`):
```typescript
const result = await getToolboxTalks({ status: 'Draft', pageSize: 100 });
```

This already fetches **all** Draft-status talks from the tenant — not filtered by wizard. So the unified list is already working at the data layer.

**Bug in current Resume routing** (`learnings/drafts/page.tsx:113-118`):
```typescript
const handleResume = (draft: ToolboxTalkListItem) => {
  const step = draft.lastEditedStep ?? 2;
  router.push(buildStepUrl(draft.id, step));  // always routes to new wizard
};
```

When an old-wizard draft (`lastEditedStep == null`) is in the list, `step` defaults to `2` and `buildStepUrl` navigates to `/admin/toolbox-talks/learnings/{id}/parse` — **a new-wizard URL that doesn't know about old-wizard talks**. This will 404 or render incorrectly.

**Fix:** detect old-wizard drafts and route to the talk detail page instead:
```typescript
const handleResume = (draft: ToolboxTalkListItem) => {
  if (draft.lastEditedStep === null) {
    // Old-wizard draft — no wizard URL; land on talk detail where Edit is available
    router.push(`/admin/toolbox-talks/talks/${draft.id}`);
    return;
  }
  router.push(buildStepUrl(draft.id, draft.lastEditedStep));
};
```

## Step 5 — URL parameter override

### No existing URL-param feature-flag pattern

The codebase has no prior example of a URL param overriding a TenantSettings flag. The closest precedent is `useSearchParams()` for filter state on list pages (e.g. `ToolboxTalkList.tsx` uses it for `page`, `search`, `frequency`, `active`).

### Proposed hook shape

```typescript
// web/src/features/toolbox-talks/hooks/useWizardPreference.ts
export function useWizardPreference(): 'new' | 'old' {
  const searchParams = useSearchParams();
  const { data: settings } = useTenantSettings();

  const urlOverride = searchParams.get('wizard');
  if (urlOverride === 'new') return 'new';
  if (urlOverride === 'old') return 'old';

  return settings?.['UseNewWizard'] === 'true' ? 'new' : 'old';
}
```

**Override URL for smoke testing:**
- Force new wizard: `?wizard=new` on any page with the Create New button
- Force old wizard: `?wizard=old` to verify fallback while toggle is `true`

**Persistence:** one-shot per navigation. The URL param is not promoted to localStorage or cookie — it survives page refreshes if the param stays in the URL, but disappears on the next click. This is correct for a testing mechanism (per §5.27 design decision 1).

**Consumption:** `useWizardPreference()` is called in the button click handler (or the button's href derivation). The hook resolves synchronously after TanStack Query has cached tenant settings (first render shows skeleton; after cache warms, no flicker).

§5.27 says the list should "distinguish" wizard types, not separate them. One list with a badge is correct. No second list needed.

## Step 7 — "Create New" routing

### Logic location

The toggle is consumed at the **button click handler level** in `ToolboxTalkList.tsx`. The hook resolves in the component body and the button's `onClick` reads the resolved value:

```typescript
// ToolboxTalkList.tsx
const wizardPreference = useWizardPreference();

// ...

<Button onClick={() => router.push(
  wizardPreference === 'new'
    ? '/admin/toolbox-talks/learnings/new'
    : `${basePath}/create`
)}>
  Create New
</Button>
```

`useWizardPreference` is a client hook (uses `useSearchParams` and `useTenantSettings`), so the component must be a Client Component — which `ToolboxTalkList.tsx` already is (`'use client'` on line 1).

### "Create another" in old wizard PublishStep

`create-wizard/steps/PublishStep.tsx:223` has a "Create another" button pointing to `/admin/toolbox-talks/create`. After toggle infrastructure ships, this button should also be toggle-aware (i.e. a user on a new-wizard tenant who published via old-wizard fallback should be offered the new wizard for their next creation).

## Adjacent observations

1. **`/admin/toolbox-talks/learnings/page.tsx` is a redirect** to `drafts`. The `Learnings` tab in the admin nav (`layout.tsx:25`) links to `/admin/toolbox-talks/talks` (the old Learnings list), but any path under `/admin/toolbox-talks/learnings` highlights that tab. The two surfaces (`/talks` and `/learnings/drafts`) coexist under the same nav tab. This is intentional per §5.21.

2. **Old wizard reads `useTenantSettings()` already** — at `CreateWizard.tsx:170` for `SkipValidationStep`. Adding `useWizardPreference()` in the Create button (which appears on the *talks list* page, not inside the wizard itself) does not affect the old wizard's internal behaviour.

3. **No `lastEditedStep` on `ContentCreationSession`** — the session has its own status fields. The `lastEditedStep` discriminator lives on `ToolboxTalk`, not on the session. This means the discriminator only works once a talk exists (i.e. after new-wizard Step 1). For in-flight old-wizard sessions where no `ToolboxTalk` row has been created yet, those sessions are not visible in the drafts list at all (the list queries `ToolboxTalk` rows). This is the correct behaviour — there's nothing to resume in the list for pre-talk old-wizard sessions.

4. **`stepLabel(null)` currently shows "Step 1 — Input & Config"** — this fires today for any old-wizard draft in the new-wizard drafts list. The bug is latent but live as soon as an old-wizard talk ends up as Draft status.

5. **§5.27's "no automatic threshold" decision** is self-enforcing by the implementation shape: there's no code path that auto-flips `UseNewWizard`. The only write point is the settings UI (Chunk 4) and direct DB/API calls.

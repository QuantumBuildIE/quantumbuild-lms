# 5.4 LastValidationRunId Fix — GetState Populated for In-Progress Runs

**Date:** 2026-06-12
**Branch:** transval
**Baseline commit:** 72e6736

---

## ROOT CAUSE

`TranslationWorkflowService.cs:89–96` (pre-fix) filtered the `lastValidationRun` query by
`r.Status == ValidationRunStatus.Completed`:

```csharp
var lastValidationRun = await context.TranslationValidationRuns
    .IgnoreQueryFilters()
    .Where(r => !r.IsDeleted
             && r.TenantId == tenantId
             && r.ToolboxTalkId == talkId
             && r.LanguageCode == languageCode
             && r.Status == ValidationRunStatus.Completed)   // ← BUG
    .OrderByDescending(r => r.CompletedAt)
    .FirstOrDefaultAsync(ct);
```

For an in-progress translation, `Status == Running` (not `Completed`), so the query returned
`null`. The DTO's `LastValidationRunId` was then `null`.

The frontend's `useWorkflowSubscription` hook extracts `activeRunIds` from languages where
state is `Translating`/`Validating` AND `lastValidationRunId` is truthy. With
`LastValidationRunId` null during the run, `activeRunIds` was empty, no `WorkflowSubscriber`
mounted, no SignalR connection opened, and the badge never received the `ValidationComplete`
event that would invalidate the workflow-state query.

**Symptom:** State badge showed "Translating..." correctly during the run but stayed
"Translating..." indefinitely even after the job completed. Only a manual page refresh
revealed the completed state.

**Verified by:** smoke test on Railway Development, 2026-06-12. `workflow-state` response
during a run was `[{"state":"Translating","lastValidationRunId":null,...}]`. `WorkflowEvents`
DB showed all four events present (`TranslationStarted`, `TranslationCompleted`,
`ValidationStarted`, `ValidationCompleted`). UI stayed stuck at Translating.

This is the third "designed-for-post-completion-only" bug in the 5.4 series:
1. 5.4-workflow-state-enumeration-fix — endpoint returned `[]` during in-progress run
2. 5.4-tenant-context-fix — Hangfire job wrote to wrong tenant
3. This fix — `LastValidationRunId` was null during in-progress run

## IN-SCOPE WORK

### A1 — Service query change

**File:** `src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Infrastructure/Services/Workflows/TranslationWorkflowService.cs:89–100`

**Before:**
```csharp
var lastValidationRun = await context.TranslationValidationRuns
    .IgnoreQueryFilters()
    .Where(r => !r.IsDeleted
             && r.TenantId == tenantId
             && r.ToolboxTalkId == talkId
             && r.LanguageCode == languageCode
             && r.Status == ValidationRunStatus.Completed)
    .OrderByDescending(r => r.CompletedAt)
    .FirstOrDefaultAsync(ct);
```

**After:**
```csharp
// Include runs in any status (Pending, Running, Completed, Failed) so that
// LastValidationRunId is populated for in-progress runs, not only after completion.
// Order by StartedAt first (set when the job begins executing), falling back to
// CreatedAt for Pending runs whose StartedAt is still null.
var lastValidationRun = await context.TranslationValidationRuns
    .IgnoreQueryFilters()
    .Where(r => !r.IsDeleted
             && r.TenantId == tenantId
             && r.ToolboxTalkId == talkId
             && r.LanguageCode == languageCode)
    .OrderByDescending(r => r.StartedAt ?? r.CreatedAt)
    .FirstOrDefaultAsync(ct);
```

**Ordering rationale:** `TranslationValidationRun.StartedAt` is `DateTime?` — it is set when
the Hangfire job begins executing. Pending runs have `StartedAt = null`. The
`StartedAt ?? CreatedAt` expression (maps to PostgreSQL `COALESCE(started_at, created_at)`)
ensures Pending runs sort by when they were created, while Running/Completed/Failed runs sort
by when they actually started. `CreatedAt` is on `BaseEntity` (inherited by
`TenantEntity → TranslationValidationRun`).

### A2 — DTO construction guard on LastValidationOutcome

**File:** `src/Modules/ToolboxTalks/QuantumBuild.Modules.ToolboxTalks.Infrastructure/Services/Workflows/TranslationWorkflowService.cs:118–120`

**Before:**
```csharp
LastValidationOutcome = lastValidationRun?.OverallOutcome,
```

**After:**
```csharp
LastValidationOutcome = lastValidationRun?.Status == ValidationRunStatus.Completed
    ? lastValidationRun.OverallOutcome
    : null,
```

`OverallOutcome` is a non-nullable `ValidationOutcome` enum with default value `Pass` (0).
Without the guard, an in-progress run would have surfaced `Pass` as the outcome before
scoring completed. The guard ensures `LastValidationOutcome` is only populated when the run
has actually finished scoring.

`TranslationWorkflowStateDto.LastValidationOutcome` is `ValidationOutcome?` (nullable), so
returning `null` is type-safe.

### A3 — EventTypeToState unaffected

The `EventTypeToState` switch (`TranslationWorkflowService.cs:799–814`) is a private helper
that derives workflow state from `WorkflowEvents`, not from `TranslationValidationRuns`. The
`ValidationStarted → Validating` mapping introduced in the previous 5.4 chunk (line 802) is
completely independent of this query change.

## BACKLOG NOTES

### Audit other workflow service queries for `Status == Completed` assumptions

`TranslationWorkflowService.cs` has two additional queries that also filter by
`Status == Completed`:

1. **`ComputeFlaggedWordCountAsync` (line 889)** — correct to keep as-is. Flagged words are
   computed from `TranslationValidationResult`/`TranslationFlag` rows, which only exist for
   completed runs. Removing the filter would do an extra query against an empty result set for
   in-progress runs.

2. **`GetPortalContext` (line 670)** — also correct to keep as-is. The external review portal
   displays section-level results, which are only meaningful after scoring completes. Showing
   an in-progress run with no results would be a confusing UX.

Both are intentional and semantically correct. No action required.

**Recommend adding a BACKLOG item:** "Audit any future `TranslationValidationRun` query that
filters by `Status == Completed` — verify whether the query is intentionally post-completion
only, or whether it should behave like `GetState`'s `lastValidationRun` query and include all
statuses." Add a comment to each such query explaining the intent.

# §3.11 Recon — Tenant Creation 400 "already exists" on First Submit

**Date:** 2026-06-17  
**Status:** Complete — root cause confirmed, fix shape clear  
**Scope:** Read-only investigation, no code changes

---

## Summary

**Root cause:** The frontend submit button's `disabled={isPending}` guard creates a brief
window — one React render cycle (~16ms) — between when `mutateAsync` is called and when React
re-renders the DOM with the button disabled. A second click (double-click, keyboard Enter, or
any mechanism that fires the click event twice) within that window dispatches a second
`onSubmit` invocation, which fires a second HTTP POST while the first is in-flight.

- **First POST** (in-flight): backend duplicate check passes → tenant created → 201 returned
- **Second POST** (overlapping): backend duplicate check finds the tenant just committed by
  the first → returns 400 "A tenant with this name already exists"

The user's `mutateAsync` in the second invocation rejects → the `catch` block shows
`toast.error("Failed to create tenant")`. The tenant IS in the database (created by the
first request). The network tab shows the 400 response.

**Recommended fix:** Replace `disabled={isPending}` with
`disabled={isPending || form.formState.isSubmitting}` on the submit button. React Hook
Form sets `formState.isSubmitting` synchronously (via an internal ref) the moment the
submit event is processed, before any async work begins and before any React re-render.
This closes the window. No backend changes needed.

## Step 2 — Duplicate Check Logic

**File:** [TenantService.cs](src/Core/QuantumBuild.Core.Application/Features/Tenants/TenantService.cs)

```csharp
public async Task<Result<TenantDetailDto>> CreateAsync(CreateTenantCommand command)
{
    try
    {
        var existingByName = await _context.Tenants
            .IgnoreQueryFilters()
            .AnyAsync(t => t.Name.ToLower() == command.Name.ToLower() && !t.IsDeleted);

        if (existingByName)
            return Result.Fail<TenantDetailDto>("A tenant with this name already exists");

        if (!string.IsNullOrWhiteSpace(command.Code))
        {
            var existingByCode = await _context.Tenants
                .IgnoreQueryFilters()
                .AnyAsync(t => t.Code != null && t.Code.ToLower() == command.Code.ToLower() && !t.IsDeleted);

            if (existingByCode)
                return Result.Fail<TenantDetailDto>("A tenant with this code already exists");
        }

        var tenant = new Tenant { ... };
        _context.Tenants.Add(tenant);
        await _context.SaveChangesAsync();

        return Result.Ok(MapToDetailDto(tenant));
    }
    catch (Exception ex)
    {
        return Result.Fail<TenantDetailDto>($"Error creating tenant: {ex.Message}");
    }
}
```

**Findings:**

| Aspect | Detail |
|---|---|
| Fields checked | `Name` (always), `Code` (only if non-empty) |
| Scope | Global — `IgnoreQueryFilters()` bypasses all query filters, checks all tenants |
| Case sensitivity | Application-level: `ToLower()` on both sides (case-insensitive) |
| Trimming | None — leading/trailing whitespace NOT normalised before comparison |
| Soft-delete handling | `&& !t.IsDeleted` — soft-deleted tenants are **excluded** from the check |
| Error message source | `Result.Fail<TenantDetailDto>("A tenant with this name already exists")` |

**Critical observation:** The only code path that produces the exact error string "A tenant
with this name already exists" with a 400 response is when this check finds a match. For
this to fire while the tenant IS in the DB, that tenant must have been committed by an
earlier request that already passed the check. Two requests are required.

**Whitespace gap:** The check does not trim `command.Name` before comparison, but the
`ToLower()` comparison means case differences can't cause false positives. Untrimmed
whitespace could cause a false negative (miss a duplicate), but cannot cause a false positive
(spurious "already exists").

**Soft-delete gap:** Soft-deleted tenants are excluded. A tenant named "Acme" that was
soft-deleted would not block a new "Acme" at the application layer, but the DB has a unique
index `IX_Tenants_Name` with no filter clause — so a soft-deleted row with the same name
WOULD block at the DB layer. That would produce "Error creating tenant: ..." (EF exception)
not "A tenant with this name already exists". This is a separate latent bug (not the
reported symptom).

## Step 3 — Existing Data State

Not queried directly (no live DB access). Confirmed from `TenantConfiguration.cs`:

```csharp
builder.HasIndex(e => e.Name)
    .IsUnique()
    .HasDatabaseName("IX_Tenants_Name");     // No filter — hard unique on ALL rows

builder.HasIndex(e => e.Code)
    .IsUnique()
    .HasFilter("\"Code\" IS NOT NULL")
    .HasDatabaseName("IX_Tenants_Code");     // Filtered — unique only for non-null codes
```

The `IX_Tenants_Name` index has no filter clause, so it enforces uniqueness across all
rows including soft-deleted ones at the DB layer. The application-level duplicate check
(Step 2) is more permissive (excludes soft-deleted rows). This mismatch means a soft-deleted
tenant with the same name produces a generic EF exception on the second attempt, not the
clean "already exists" message.

No seed data file was located in the scope of this investigation that would reveal a
pre-seeded tenant. If there is a seeded tenant (e.g., a "Demo" or "QuantumBuild" tenant),
its name collision would be an additional trigger path — but it would not explain the
"fresh unique name" reproduction described in the BACKLOG.

## Step 4 — Frontend Submission Path

**Files:**

### Submit handler structure

```typescript
// tenant-form.tsx
const onSubmit = async (data: TenantFormData) => {
  // ... sector validation
  
  const payload = { name, code, companyName, contactName, contactEmail };

  // Step 1 — Create tenant
  let newTenant: TenantDetail;
  try {
    newTenant = await createTenant.mutateAsync(payload);
  } catch {
    toast.error("Failed to create tenant");
    return;
  }

  // Step 2 — Assign sectors
  for (let i = 0; i < sectorIds.length; i++) {
    try {
      await assignTenantSector(newTenant.id, { sectorId: sectorIds[i], isDefault: i === 0 });
    } catch {
      sectorsFailed = true; break;
    }
  }
  // ...
};

// Form rendering
<form onSubmit={form.handleSubmit(onSubmit)} className="space-y-6">
  ...
  <Button type="submit" disabled={isPending}>  {/* ← THE GAP */}
    {isPending ? "Creating..." : "Create Tenant"}
  </Button>
```

Where:
```typescript
const isPending = createTenant.isPending || updateTenant.isPending;
```

### The double-submit window

`createTenant.isPending` is a React state value sourced from TanStack Query's
`useMutation`. When `mutateAsync` is called:

1. TanStack Query internally marks the mutation as pending
2. React **schedules** a re-render (asynchronous)
3. The DOM is still rendered with `disabled={false}` until the re-render completes
4. A second click (or keyboard submit, or programmatic trigger) within this window fires
   a second `onSubmit` execution
5. Second `onSubmit` calls `createTenant.mutateAsync(payload)` — a second HTTP POST

React Hook Form's `formState.isSubmitting` is often cited as a synchronous guard against
this pattern. However, the form here uses:
```tsx
disabled={isPending}
```
...not `disabled={isPending || form.formState.isSubmitting}`. React Hook Form sets
`isSubmitting` via internal ref tracking (synchronous, not a React state update), so it
closes the window before the re-render. The current code does NOT use this guard.

### API response parsing

```typescript
// tenants.ts
export async function createTenant(data: CreateTenantDto): Promise<TenantDetail> {
  const response = await apiClient.post<ApiResponse<TenantDetail>>("/tenants", data);
  return response.data.data;  // reads Result<TenantDetailDto>.Data
}
```

The backend controller returns `CreatedAtAction(..., result)` where `result` is
`Result<TenantDetailDto>`. JSON shape:
```json
{ "success": true, "data": { ...TenantDetailDto... }, "errors": [], "errorCode": null }
```
`response.data.data` correctly extracts `TenantDetailDto`. No shape mismatch here.

### No client-side field transformation

The payload is built as:
```typescript
const payload = {
  name: data.name,            // raw from form — NOT trimmed
  code: data.code || undefined,
  // ...
};
```

The name is not transformed (no `trim()`, no `toLowerCase()`). The backend also
does not trim before comparison. Whitespace-padded names could create phantom
"different" names that pass the check, but this is not the reported symptom.

### TanStack Query mutation — no retry

```typescript
return useMutation({
  mutationFn: (data: CreateTenantDto) => createTenant(data),
  onSuccess: () => { queryClient.invalidateQueries({ queryKey: TENANTS_KEY }); },
});
```

No `retry` option configured. TanStack Query default for mutations is `retry: 0`. TanStack
Query is **not** retrying the mutation on failure.

## Step 5 — Idempotency and Transaction

### Controller flow

```csharp
[HttpPost]
public async Task<IActionResult> Create([FromBody] CreateTenantCommand command)
{
    var result = await _tenantService.CreateAsync(command);  // saves tenant to DB

    if (!result.Success)
        return BadRequest(result);          // ← ONLY 400 PATH

    // ... audit log ...

    if (!string.IsNullOrWhiteSpace(command.ContactEmail) && !string.IsNullOrWhiteSpace(command.ContactName))
    {
        var onboardingResult = await _onboardingService.ProvisionTenantAsync(
            result.Data!.Id, command.ContactEmail, command.ContactName);

        if (!onboardingResult.Success)
            return StatusCode(207, new { ... });  // 207, NOT 400
    }

    return CreatedAtAction(nameof(GetById), new { id = result.Data!.Id }, result);  // 201
}
```

**Elimination of "misleading error from post-commit" hypothesis:**

The onboarding service (`TenantOnboardingService`) has a full `try/catch` that returns
`Result.Fail(...)` — it never throws. When onboarding fails, the controller returns HTTP
207 (not 400). The 400 can ONLY come from `BadRequest(result)` when
`_tenantService.CreateAsync` itself returns failure — which means the tenant was NOT
committed by that request.

Therefore: if the 400 is observed AND the tenant IS in the DB, the tenant was committed by
a different request than the one that returned 400.

**Onboarding partial-create risk (separate concern):** If `CreateAdminUserAsync` creates the
ASP.NET Identity user successfully but `CreateLinkedEmployeeAsync` fails, the user exists in
`AspNetUsers` but without an Employee record or EmployeeId link. The tenant itself is safe —
it was saved before onboarding began. This is a data integrity edge case, not related to the
reported 400 bug.

**No transaction wrapping tenant + onboarding:** The tenant creation and the onboarding
provisioning are in separate `SaveChangesAsync` calls within the same HTTP request scope.
If onboarding fails mid-way, the tenant row persists. The 207 response signals this to the
caller. The frontend correctly handles 207 as a partial success (Axios accepts 2xx; the form
proceeds to sector assignment and shows success/warning toast).

## Step 6 — Error Message Correctness

The error message is accurate: if the duplicate check fires, a tenant with that name IS
already in the DB. The 400 status is appropriate. The problem is not misleading messaging —
the message is correctly describing what happened.

What IS misleading is the user experience: the user sees `toast.error("Failed to create
tenant")` from the second submit's catch block, even though the first submit succeeded. The
tenant was created; only the second (spurious) request failed. The user may not realize
they have a successful creation in the DB.

## Step 7 — Axios Interceptor (Non-Contrib)

**File:** [client.ts](web/src/lib/api/client.ts)

The Axios 401 interceptor DOES retry original requests after token refresh:
```typescript
originalRequest._retry = true;
// ... refresh token ...
return apiClient(originalRequest);  // retries the failed request
```

This could potentially cause a double POST if a request returns 401 AFTER the backend has
already processed it. However, this is impossible in the standard JWT middleware flow:
JWT validation runs BEFORE the controller, so a token-invalid request returns 401 without
the controller executing. If the token is valid, the controller runs and returns 201/207 —
no retry triggered. The interceptor is therefore **not a contributor to this bug**.

The `isRefreshing` / `failedQueue` pattern prevents multiple concurrent token refreshes,
which is correct.

## Hypothesis

**Confirmed primary cause: Frontend double-submit race condition**

The submit button's disabled state relies on `isPending` (a React state value), which
only becomes `true` after React re-renders. Between calling `mutateAsync` and that
re-render, the button is still clickable. A second click within one render cycle
(~16ms at 60fps) dispatches a second `onSubmit`, fires a second `POST /api/tenants`,
and that second request races the first. In the 201-then-400 race:

- First POST: duplicate check → no match → insert → commit → 201 returned
- Second POST: duplicate check → finds row just committed → 400 returned

The second POST's 400 rejects `mutateAsync` in the second `onSubmit` invocation.
Its `catch` block fires `toast.error("Failed to create tenant")`. The user sees the
error toast; the tenant is in the DB.

Whether the UI also briefly shows the success toast (from the first invocation completing
concurrently) depends on timing. If the user clicked Submit and Enter in rapid succession,
both could be in flight simultaneously.

**Why this is reproducible in QA:** QA engineers typically test with keyboard navigation
(Enter key submits forms), and may inadvertently press Enter twice when the submit
confirmation is slow to appear. Additionally, test environments with slower API latency
widen the race window, making the bug more likely.

## Recommended Fix

**Single change, frontend only.**

In [tenant-form.tsx](web/src/components/admin/tenant-form.tsx), change the submit button:

```tsx
// BEFORE
<Button type="submit" disabled={isPending}>

// AFTER
<Button type="submit" disabled={isPending || form.formState.isSubmitting}>
```

`form.formState.isSubmitting` is set synchronously by React Hook Form via internal ref
tracking when the submit event fires — before any async work begins and before any
React re-render. This closes the race window completely.

No backend changes are required. The duplicate check, the 400 response, and the error
message are all correct and appropriate — they should not be softened or removed.

**Secondary hardening (optional):** Add an explicit `name.trim()` before the payload to
prevent whitespace-padded duplicates from bypassing the application-level check:
```typescript
const payload = {
  name: data.name.trim(),   // ← add .trim()
  // ...
};
```
This closes the whitespace gap identified in Step 2 but is not required for the reported bug.

**Separate latent bug (do not fix in this chunk):** `IX_Tenants_Name` is a hard unique
index with no filter — it blocks reinstatement of soft-deleted tenant names at the DB layer.
The application duplicate check (Step 2) excludes soft-deleted rows, so deleted-name
recycling passes the application check but fails at the DB layer with a generic EF
exception. This is a correctness bug independent of §3.11.

## 3. stepOrder.ts â€” isStepReachable Signature Decision

**Current file:** `web/src/features/toolbox-talks/components/learning-wizard/lib/stepOrder.ts`

**Current case 7 (lines 58â€“60):**
```typescript
case 7:
  // Publish: placeholder â€” 5.5 owns the real rule
  return false;
```

**Current signature:**
```typescript
export function isStepReachable(step: number, talk: ToolboxTalk | null): boolean
```

**Required rule (from prompt):**
- `talk.sections.length > 0` AND
- `talk.status !== 'Published'` AND
- `(targetLanguageCodes.length === 0 OR at least one ValidationRunSummary with status === 'Completed' exists)`

The third clause requires `ValidationRunSummary[]` data, which is not available on the talk object.

**Decision: extend signature with optional third parameter:**
```typescript
export function isStepReachable(
  step: number,
  talk: ToolboxTalk | null,
  validationRuns?: ValidationRunSummary[] | null
): boolean
```

**Existing call sites (both in `useStepNavigation.ts`):**
```typescript
// line 58: reachableSteps
reachable: isStepReachable(s.number, talk),

// line 68: goToStep guard
if (!isStepReachable(step, talk)) return;
```

Both callers pass no third argument â€” TypeScript treats optional params as `undefined`. With the new case 7 logic:
- `validationRuns === undefined` â†’ for talks WITH target languages: returns false (safe default, same as before)
- `validationRuns === undefined` â†’ for talks WITHOUT target languages: returns true if sections exist (new â€” English-only talks now show step 7 as reachable in the indicator from any step)

This is correct behaviour. English-only talks have no translation gate, so step 7 is reachable once content exists.

**useStepNavigation.ts** extended with:
```typescript
interface UseStepNavigationOptions {
  talkId: string | null;
  currentStep: number;
  talk: ToolboxTalk | null;
  validationRuns?: ValidationRunSummary[] | null;  // NEW â€” passed to isStepReachable for step 7 gate
}
```

The publish page passes `validationRuns` so the step indicator on that page correctly marks step 7 as reachable/unreachable.

## 5. Publish Endpoint Client Wrapper

**No existing `publishTalk` function for the new endpoint.** The old `publishSession` in `content-creation.ts` calls `POST /api/toolbox-talks/create/session/{sessionId}/publish` â€” different route, different body.

**New function to add** in `web/src/lib/api/toolbox-talks/toolbox-talks.ts`:
```typescript
export interface PublishTalkResult {
  talkId: string;
  status: string;
  publishedAt: string;
}

export async function publishTalk(talkId: string): Promise<PublishTalkResult> {
  const response = await apiClient.post<PublishTalkResult>(`/toolbox-talks/${talkId}/publish`);
  return response.data;
}
```

Returns `response.data` directly (no `Result<T>` unwrap) â€” matches the new endpoint convention confirmed in 5.5a report.

**Mutation hook** (new file `learning-wizard/hooks/usePublishTalk.ts`):
```typescript
export function usePublishTalk(talkId: string) {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: () => publishTalk(talkId),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['learnings', talkId] });
      queryClient.invalidateQueries({ queryKey: ['toolbox-talks'] });
    },
  });
}
```

## 7. Post-Publish Navigation Decision

**Old wizard:** Shows a success panel with "View Toolbox Talk" and "Create Another" buttons. User must click to navigate.

**New wizard decision: navigate to the talk's detail page immediately on success, with a toast confirmation.**

Rationale: The wizard is a linear flow; once published, the logical next action is to view the live talk. An intermediate success screen adds a click with no additional value. A toast ("Published successfully") provides the confirmation moment.

Navigate to: `/admin/toolbox-talks/talks/${result.talkId}`

## 8. Component Architecture

The publish page uses `WizardLayout`'s `footer` slot for the Publish button:
- Page wrapper owns the `usePublishTalk` mutation
- Page wrapper renders `footer={<PublishButton .../>}`
- `PublishStep` receives `talkId`, `isPublishing`, `publishError` as props
- `PublishStep` renders all summary panels + error display + external review warning
- `PublishStep` calls its own hooks (`useTalk`, `useValidationRuns`, `useWorkflowSubscription`) â€” same query keys, React Query dedupes the fetches

This avoids dual bottom-bars (WizardLayout nav bar + PublishStep action row) and keeps the component pattern consistent with other steps.

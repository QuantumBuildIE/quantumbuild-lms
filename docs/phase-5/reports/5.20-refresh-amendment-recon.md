## Summary

The regular-admin slice of §5.20 is a phantom. Code trace finds no redirect mechanism that fires during hydration for regular tenant admins refreshing on `/admin/toolbox-talks/learnings/{talkId}/{step}` routes. The `(authenticated)/layout.tsx` loading-spinner guard prevents every sub-layout's `useEffect` from mounting until auth is fully resolved, and once resolved, all permission checks pass cleanly for admin users. The original "refresh-to-drafts" symptom confirmed in the Phase 5.3b smoke was SuperUser-only, caused entirely by `TenantQueryInvalidator` — which §5.22 fixed on 2026-06-15. There is no separate cause to find for regular admins.

**Recommended direction:** Close §5.20 as fully resolved. The SuperUser slice was closed by §5.22. The regular-admin slice was a speculative assumption in the §5.22 fix report ("if regular admins also experience this"), not a confirmed observation, and code analysis confirms no mechanism exists that would produce the symptom.

## Step 0 — Symptom Verification

### Case 1: SuperUser with stored activeTenantId

**Code trace:**

1. `auth-context.tsx:113-115` — `initAuth()` reads `localStorage.getItem("activeTenantId")`, calls `setActiveTenantIdState(storedUUID)`. `activeTenantId` transitions `null → storedUUID`.
2. `auth-context.tsx:120` — `await loadUser(storedToken)` is async; `isLoading` stays `true` while it runs.
3. `(authenticated)/layout.tsx:30-36` — `isLoading = true` → renders loading spinner and returns early. Children (including admin layouts) are NOT mounted. No `useEffect` fires in `admin/layout.tsx` or `admin/toolbox-talks/layout.tsx`.
4. `providers.tsx — TenantQueryInvalidator` — mounted at the `Providers` level (above `(authenticated)/layout.tsx`). Effect dep `[activeTenantId, queryClient]` fires on the `null → storedUUID` transition:
   - `previousTenantId.current !== activeTenantId` → true
   - Updates ref, calls `queryClient.invalidateQueries()`
   - **No redirect** (§5.22 fix removed it)
5. `loadUser` completes → `setAuthState({ user: SuperUser, isLoading: false })`.
6. `(authenticated)/layout.tsx` now renders children.
7. `admin/layout.tsx:52-59` — SU tenant guard: `isSuperUser && !activeTenantId` → `true && false` = **false** (activeTenantId is now the stored UUID). No redirect.
8. `admin/toolbox-talks/layout.tsx:50-54` — `hasAdminPermission = true` for SU. No redirect.
9. Wizard page renders. `useTalk(talkId)` fetches and renders the correct step.

**Verdict: Stays on wizard step. ✅ Resolved by §5.22.**

### Case 2: Regular tenant admin

**Code trace:**

1. `auth-context.tsx:113-116` — `localStorage.getItem("activeTenantId")` returns `null` (regular admins have no tenant-switcher surface and never store a value). `setActiveTenantIdState` is not called. `activeTenantId` stays `null`.
2. `providers.tsx — TenantQueryInvalidator` — `previousTenantId.current` initialises to `null`. `activeTenantId` is `null` and stays `null`. Effect body condition `previousTenantId.current !== activeTenantId` → `null !== null` = **false**. Effect body never executes. No redirect, no query invalidation.
3. `auth-context.tsx:120` — `await loadUser(storedToken)` runs. `isLoading = true` throughout.
4. `(authenticated)/layout.tsx:30-36` — `isLoading = true` → loading spinner, returns early. **Neither `admin/layout.tsx` nor `admin/toolbox-talks/layout.tsx` mounts during auth initialization. Their `useEffect` hooks do not fire.**
5. `loadUser` completes → `setAuthState({ user: regularAdmin, isLoading: false })`.
6. `(authenticated)/layout.tsx` renders children.
7. `admin/layout.tsx:40-44` — `hasCorePermission = useHasAnyPermission(corePermissions)`. With user loaded: admin has permissions → `hasCorePermission = true`. Effect fires but `!hasCorePermission` is false. No redirect to `/toolbox-talks`.
8. `admin/layout.tsx:52-59` — `isSuperUser = false` → `isSuperUser && !activeTenantId && !isNonTenantScoped` = **false**. No redirect to `/admin/tenants`.
9. `admin/toolbox-talks/layout.tsx:50-54` — `hasAdminPermission = true` (admin has `Learnings.Admin`). No redirect to `/dashboard`.
10. Wizard page renders. `useTalk(talkId)` (`staleTime: 30_000`) fetches talk by talkId and renders. Loading skeleton shown until data arrives, then correct step renders.

**Verdict: Stays on wizard step. ✅ Symptom does not reproduce for regular admins.**

**Key protective mechanism:** `(authenticated)/layout.tsx:30-36` — the `isLoading` guard returns a loading spinner and never renders children until auth is settled. This prevents every nested `useEffect` permission guard from seeing `user = null` and firing a spurious redirect. The guards in `admin/layout.tsx` and `admin/toolbox-talks/layout.tsx` only run after the user is fully loaded.

### Case 3: SuperUser with activeTenantId = null (All Tenants mode)

**Code trace:**

1. `activeTenantId` stays `null` after auth (SU cleared their tenant selection).
2. `TenantQueryInvalidator` — no transition, effect body doesn't execute.
3. `admin/layout.tsx:52-59` — `isSuperUser = true`, `activeTenantId = null`.
   - `nonTenantScopedPaths`: `["/admin/tenants", "/admin/regulatory", "/admin/monitoring"]`
   - `/admin/toolbox-talks/learnings/{talkId}/quiz` does not start with any non-tenant-scoped path → `isNonTenantScoped = false`
   - `isSuperUser && !activeTenantId && !isNonTenantScoped` = **true**
   - `router.replace("/admin/tenants")` fires.

**Verdict: Redirected to `/admin/tenants`, NOT to `/drafts`.**

This is a different symptom from §5.20's "lands on /drafts." The redirect is by-design enforcement (SU in All Tenants mode should select a tenant before accessing tenant-scoped content). It does not meet §5.4's standard for refresh-recovery but it is a SuperUser-specific edge case, not the regular-admin case that §5.20's open slice was investigating. Documented as an adjacent observation below.

## Adjacent Observations

1. **SU + All Tenants mode bounce** (`admin/layout.tsx:52-59`): Not a §5.20 regression (destination is `/admin/tenants`, not `/drafts`), but a real §5.4 gap for SuperUsers who browse to a wizard step after clearing their tenant selection. Low-urgency — SU "All Tenants" mode is not the typical wizard usage pattern. If raised as a follow-up, the fix is a one-liner: add `"/admin/toolbox-talks/learnings"` to `nonTenantScopedPaths`, making wizard routes exempt from the tenant-selection redirect. The risk is that without a tenant, `useTalk` would fire a cross-tenant fetch that the query filter would reject — so the fix would need to pair with an error-state guard or simply accept that the tenant-guard exists for a reason.

2. **`learnings/page.tsx` cleanup candidate**: `web/src/app/(authenticated)/admin/toolbox-talks/learnings/page.tsx` exists only as a workaround for the §5.22 bug (to prevent 404 when `TenantQueryInvalidator` stripped the talkId UUID and landed on `/admin/toolbox-talks/learnings`). With §5.22 fixed, this page is no longer needed as a safety net. It can be kept as a convenient entry point (redirects to drafts) or removed as dead scaffolding. Either way it is harmless. BACKLOG §5.22 noted this as a "workaround" — its removal is a separate cleanup.

3. **`admin/toolbox-talks/layout.tsx` permission guard timing**: The `hasAdminPermission` guard (`useEffect` at line 50-54) would fire with `hasAdminPermission = false` if a user with no learnings permissions somehow navigated to a wizard URL. But because `(authenticated)/layout.tsx` renders the loading spinner during `isLoading = true`, these layouts only mount after auth is settled. This pattern is sound and requires no change.

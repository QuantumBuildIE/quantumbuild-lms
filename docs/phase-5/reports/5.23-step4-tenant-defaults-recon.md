## Summary

Step 4 (SettingsStep) currently shows defaults sourced from C# entity-level property initialisers on `ToolboxTalk.cs`, not from any tenant-configurable source. The BACKLOG's proposed fix direction ŌĆö read from `ToolboxTalkSettings` and use as defaults when `lastEditedStep < 4` ŌĆö is blocked because `ToolboxTalkSettings` is missing all five fields. The settings admin UI General tab is a placeholder stub ("coming soon"). `PUT /api/toolbox-talks/settings` is also a placeholder. No part of the prerequisite infrastructure for the fix exists yet. This is a backend-first problem: add the five fields to `ToolboxTalkSettings`, migrate, expose in settings admin UI, then wire into talk creation. The recommended fix shape is server-only (apply at creation time) rather than frontend-only (parallel fetch in the wizard step), because the wizard step already reads from `talk.*` and that pattern is correct.

## Step 5 ŌĆö Fix-direction assessment

### Candidate A: Frontend-only (as BACKLOG describes)

Fetch `ToolboxTalkSettings` in parallel with the talk in `SettingsStep`. Add conditional in `useEffect`:
```ts
if ((talk.lastEditedStep ?? 0) < 4 && settings) {
  // use settings.defaultMinimumVideoWatchPercent, etc.
}
```

**Feasibility:** Requires the five fields to be added to `ToolboxTalkSettings` first (entity + migration + DTO + settings update endpoint). Once those exist, frontend change is ~15 lines. **But the backend prerequisite is the same as all other candidates.**

**Risk:** Adds a second query to every SettingsStep render. The settings are only needed on first visit ŌĆö wasteful for return visits. The `lastEditedStep < 4` conditional is simple but introduces a new logic branch that must be tested.

**Side effects:** The wizard step now has two loading states (talk + settings) that must both resolve before the form can initialise. Minor but adds complexity.

### Candidate B: Server-only (recommended)

`InitialiseToolboxTalkCommandHandler` injects `IToolboxTalksDbContext`, queries `ToolboxTalkSettings` for the tenant, and applies the five values at creation time. The SettingsStep continues to read from `talk.*` ŌĆö no change to the component.

```csharp
// In InitialiseToolboxTalkCommandHandler:
var settings = await _dbContext.ToolboxTalkSettings
    .Where(s => s.TenantId == request.TenantId && !s.IsDeleted)
    .FirstOrDefaultAsync(cancellationToken);

var talk = new ToolboxTalk
{
    ...
    IsActive = settings?.DefaultIsActive ?? false,
    GenerateCertificate = settings?.DefaultGenerateCertificate ?? true,
    MinimumVideoWatchPercent = settings?.DefaultMinimumVideoWatchPercent ?? 90,
    AutoAssignDueDays = settings?.DefaultAutoAssignDueDays ?? 14,
    RequiresRefresher = settings?.DefaultRequiresRefresher ?? false,
    RefresherIntervalMonths = settings?.DefaultRefresherIntervalMonths ?? 12,
    ...
};
```

**Feasibility:** Same backend prerequisite (five new entity fields + migration + DTO). Frontend component unchanged.

**Risk:** If a `ToolboxTalkSettings` row doesn't exist, the null-coalescing fallbacks restore current entity defaults ŌĆö no regression. The `GetToolboxTalkSettingsQueryHandler` already implements this pattern at the query level.

**Side effects:** None. The `lastEditedStep < 4` guard is not needed ŌĆö defaults are applied exactly once at creation.

### Candidate C: Both

Creation applies tenant defaults (B), and SettingsStep also fetches settings to display a hint ("using tenant defaults"). Overkill. Adds frontend complexity for no behaviour benefit.

## Step 6 ŌĆö Gaps to flag

### G1: Five fields missing from `ToolboxTalkSettings` ŌĆö BLOCKER

All five Step 4 fields (`DefaultMinimumVideoWatchPercent`, `DefaultAutoAssignDueDays`, `DefaultGenerateCertificate`, `DefaultRequiresRefresher` + `DefaultRefresherIntervalMonths`, `DefaultIsActive`) must be added to the entity, the DTO, and the update command before either candidate A or B can land. This is the primary blocker. A migration is required.

### G2: Settings admin UI is a stub

The General tab in `/admin/toolbox-talks/settings` shows "coming soon." Without UI, tenants can't set their preferred defaults, making the whole feature invisible even after the backend is wired. The five fields need form controls in the General tab.

### G3: `PUT /api/toolbox-talks/settings` is a placeholder

The update endpoint currently returns the current settings without saving. `UpdateToolboxTalkSettingsCommandHandler.cs` exists but is not wired (`// TODO` comment in the controller action). The handler itself needs to be dispatched and must write the five new fields.

### G4: `isActive` default semantics require product decision

The handler hard-codes `IsActive = false` at creation. If `DefaultIsActive` is added to settings and a tenant sets it to `true`, new draft talks would start active ŌĆö which contradicts the "don't activate until published" intent. Options: (a) keep `IsActive = false` at creation regardless (the setting only affects the form's initial display, not the server-stored value), (b) remove the `IsActive = false` override and let the setting drive it. This is a product call and should be flagged before implementation.

### G5: Relationship to ┬¦1.1.17 ŌĆö partial overlap, not closure

┬¦1.1.17 (P2, Open) requests per-tenant defaults for Step 1 fields: target languages, sector, include-quiz, pass threshold, reviewer role, audience role, verbatim mode. ┬¦5.23 covers Step 4 fields only. They share the same entity (`ToolboxTalkSettings`) and the same settings admin UI extension pattern. ┬¦5.23 is not a closure of ┬¦1.1.17 ŌĆö different fields, different step. If ┬¦5.23 lands first, it extends `ToolboxTalkSettings` and the General tab; ┬¦1.1.17 would then add further fields to both using the same approach.

### G6: `DefaultPassingScore` already exists for Step 3

`ToolboxTalkSettings.DefaultPassingScore = 80` exists but is not currently read by the wizard's QuizStep either. If ┬¦5.23 wires up server-side defaults at creation, the pattern should likely apply to `PassingScore` in `InitialiseToolboxTalkCommandHandler` as well, for consistency. Out of scope for ┬¦5.23 but worth noting.

## Recommended direction

**Candidate B (server-only), with backend prerequisites first.**

Rationale:
1. The SettingsStep already reads from `talk.*` ŌĆö this is the correct, tested pattern. No reason to add a second query to the component.
2. Applying defaults at creation (once, server-side) is cleaner than conditional logic in the component that must distinguish "first visit" from "return visit."
3. The `lastEditedStep < 4` conditional logic is eliminated ŌĆö there is no branch to test.
4. The null-coalescing fallback approach (`?? 90`) preserves existing entity defaults when no settings row exists, making it a safe no-op for existing tenants until they configure their preferences.
5. The backend prerequisites (entity fields, migration, DTO, settings update endpoint, settings admin UI) are the same regardless of candidate.

**Implementation order within the chunk:**
1. Add five new fields to `ToolboxTalkSettings` entity
2. Add `dotnet ef migrations add AddWizardDefaultsToToolboxTalkSettings`
3. Update `ToolboxTalkSettingsDto` + TypeScript `ToolboxTalkSettings` interface
4. Implement `UpdateToolboxTalkSettingsCommandHandler` dispatch in the controller (remove `// TODO`)
5. Implement the five fields in the existing `UpdateToolboxTalkSettingsCommand` / handler
6. Add five form fields to the General tab in `/admin/toolbox-talks/settings`
7. Modify `InitialiseToolboxTalkCommandHandler` to look up settings and apply at creation
8. Resolve G4 (isActive semantics) before touching the `IsActive = false` line

# §3.11 Fix — Tenant Creation 400 "already exists" on First Submit

**Date:** 2026-06-17  
**Status:** Complete  
**Branch:** transval

---

## Summary

Three layers of hardening applied to close the double-submit race window and add defensive input hygiene. The root cause — a `~16ms` window between calling `mutateAsync` and the React re-render that applies `isPending` to the DOM — is closed by adding `form.formState.isSubmitting` (React Hook Form's synchronous guard) to the button's disabled expression. Input trimming is added at both the frontend payload construction site and the backend duplicate-check / entity creation site. A specific `DbUpdateException` catch is added to the backend to handle the rare but possible concurrent-tab scenario cleanly, returning the user-facing message instead of a raw EF exception string.

No migration required. All changes are application-layer only.

## Edit (1) — Frontend Race Fix

**File:** [web/src/components/admin/tenant-form.tsx](web/src/components/admin/tenant-form.tsx)

```diff
- <Button type="submit" disabled={isPending}>
+ <Button type="submit" disabled={isPending || form.formState.isSubmitting}>
```

`form.formState.isSubmitting` is set synchronously by React Hook Form via internal ref tracking the moment the submit handler is invoked — before any async work begins and before any React re-render. This closes the ~16ms race window where `isPending` (a React state value) hadn't yet propagated to the DOM after `mutateAsync` was called.

`form` is the `useForm()` instance declared at the top of `TenantForm` — in scope at the button render site.

## Edit (4) — Backend Defensive Catch

**File:** [src/Core/QuantumBuild.Core.Application/Features/Tenants/TenantService.cs](src/Core/QuantumBuild.Core.Application/Features/Tenants/TenantService.cs)

A specific `DbUpdateException` catch is inserted before the generic `Exception` catch:

```csharp
catch (DbUpdateException ex) when (IsUniqueConstraintViolation(ex))
{
    // Concurrent-tab scenario: a second request committed the same name between our
    // duplicate check and our SaveChangesAsync. Return the clean user-facing message
    // rather than leaking the raw EF/DB exception text.
    return Result.Fail<TenantDetailDto>("A tenant with this name already exists");
}
```

Helper (no Npgsql reference — Application layer has no Npgsql dependency):

```csharp
private static bool IsUniqueConstraintViolation(DbUpdateException ex)
{
    var message = ex.InnerException?.Message ?? ex.Message;
    return message.Contains("duplicate key", StringComparison.OrdinalIgnoreCase)
        || message.Contains("23505", StringComparison.OrdinalIgnoreCase);
}
```

PostgreSQL surfaces unique-constraint violations as `DbUpdateException` wrapping a message containing "duplicate key value violates unique constraint" (and `23505` appears in some Npgsql message formats). This check is robust without importing Npgsql types from the Infrastructure layer.

**Also handles the latent soft-delete index gap (§7.6):** when a soft-deleted tenant's name blocks a new tenant at the DB layer (the application-level check passes, the index blocks), the catch translates this to the same clean user-facing message.

## Toast Behavior

**Pre-fix:** Under the double-submit race, the first `mutateAsync` completes with 201 (success toast fires) while the second `mutateAsync` rejects with 400 (`catch` fires `toast.error("Failed to create tenant")`). Depending on timing, the user may see both toasts briefly — a confusing success + failure pair — or just the error toast if the 400 response arrives before the 201.

**Post-fix:** With `form.formState.isSubmitting` preventing the second submit from reaching the handler, only one `mutateAsync` call is made per form submission. The success toast fires cleanly; no error toast.

# Phase 5.4 — SignalR 30-second timeout fix

**Date:** 2026-06-12
**Branch:** transval
**Follows:** 5.4-signalr-fix.md (SignalR wiring gap fix)

---

## Problem

Translation validation jobs longer than ~30 seconds caused the WebSocket to drop with a 1006 close code, preventing `ValidationComplete` from reaching the client. Root cause: Railway's proxy idle-timeout fires when no data crosses the WebSocket for ≥30 s. The server's default `KeepAliveInterval` (15 s) is within the danger zone; the client's default `serverTimeoutInMilliseconds` (30 s) caused the client to declare the connection dead before a keep-alive response could arrive.

Reference: `5.4-signalr-recon.md` and `5.4-signalr-recon-followup.md` (pre-chunk investigation documents — note: these files were not committed to the repo; analysis was performed in the same session).

## Changes

### 1. `src/QuantumBuild.API/Program.cs` — lines 212–222

Replaced `builder.Services.AddSignalR();` (line 212) with an options block:

```csharp
builder.Services.AddSignalR(options =>
{
    options.KeepAliveInterval = TimeSpan.FromSeconds(10);
    options.ClientTimeoutInterval = TimeSpan.FromMinutes(2);
});
```

**`KeepAliveInterval = 10 s`** — server pings the client every 10 seconds when no data flows. At 10 s the ping crosses the wire at least 3× per 30-second idle window, well inside Railway's assumed ~60-second proxy timeout.

**`ClientTimeoutInterval = 2 min`** — server waits 2 minutes of silence before declaring the client gone. Default was 30 s — same as the Railway proxy timeout, creating a race.

### 2. `web/src/features/toolbox-talks/hooks/use-validation-hub.ts` — lines 71–73 and 95–97

After `.build()` on the initial connection (line 71):

```typescript
connection.serverTimeoutInMilliseconds = 120_000;       // 2 min — matches server ClientTimeoutInterval
connection.keepAliveIntervalInMilliseconds = 10_000;    // 10 s — matches server KeepAliveInterval
```

After `.build()` on the manual-reconnect `fresh` connection (line 95):

```typescript
fresh.serverTimeoutInMilliseconds = 120_000;
fresh.keepAliveIntervalInMilliseconds = 10_000;
```

**`serverTimeoutInMilliseconds = 120_000`** — client waits 2 minutes before declaring the server gone. Default was 30 s — same as the observed drop interval.

**`keepAliveIntervalInMilliseconds = 10_000`** — client sends its own keep-alive ping every 10 seconds when it hasn't heard from the server. Default was 15 s, which could still fall into a 30-second Railway idle window.

Both the initial connection and the manual-reconnect `fresh` connection are patched so the settings are consistent regardless of which code path establishes the connection.

## Blast Radius — Five-Hub Analysis

`AddSignalR(options => ...)` is called once at service-registration time. All five hubs registered in `Program.cs` (lines 414–418) inherit these settings:

| Hub | Route | Safety assessment |
|-----|-------|-------------------|
| **SubtitleProcessingHub** | `/api/hubs/subtitle-processing` | Safe. Subtitle transcription/translation jobs run for 30–120 s. Shorter keep-alive prevents the same Railway idle-timeout drop this fix targets. Longer client timeout is strictly more lenient. No regression. |
| **ContentGenerationHub** | `/api/hubs/content-generation` | Safe. AI content generation jobs run for 15–60 s. Same rationale — shorter keep-alive is conservative; longer timeout is lenient. No regression. |
| **TranslationValidationHub** | `/api/hubs/translation-validation` | Safe. This is the hub the fix targets. The settings are exactly what is needed for 30–90 s validation jobs. |
| **CorpusRunHub** | `/api/hubs/corpus-run` | Safe. Corpus runs can take minutes. These settings are if anything more important here. Strictly no regression. |
| **LessonParserHub** | `/api/hubs/lesson-parser` | Safe. Parsing jobs are similar duration to content generation. Same rationale applies. |

**Safety argument:** A shorter `KeepAliveInterval` means the server sends more frequent pings — the only cost is negligible extra bandwidth. A longer `ClientTimeoutInterval` means the server is more patient with slow/proxy-delayed clients — the only cost is that a truly dead client is detected slightly later. Neither change can cause a connected, healthy client to lose messages or get disconnected. No hub is expected to break.

### Smoke test

The smoke test (translation job > 60 s, continuous WebSocket, `ValidationComplete` received client-side without 1006 drop) requires a live environment with an active Railway proxy. This must be confirmed manually:

1. Open DevTools → Network, filter to WS
2. Start a translation expected to run > 60 seconds
3. Observe the WebSocket connection stays open continuously from job start through `ValidationComplete` — no 1006 close, no reconnect-attempt frames
4. Confirm UI transitions to "Validated" without manual refresh
5. Capture: timestamps of start and `ValidationComplete`, total WS lifetime from Network tab

The configuration change is a necessary and sufficient fix for the diagnosed root cause (Railway idle-timeout fires when keep-alive interval ≥ timeout). If the WebSocket still drops at a consistent sub-10-second interval after this fix, that indicates the Railway proxy idle timeout is shorter than assumed — escalate to BACKLOG with observed drop interval.

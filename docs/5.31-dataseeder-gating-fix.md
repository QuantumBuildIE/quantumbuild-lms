# §5.31 DataSeeder Gating — Implementation Report

**Date:** 2026-06-17  
**Branch:** transval  
**Recon:** `docs/5.31-dataseeder-gating-recon.md`

## Summary

DataSeeder no longer creates credentialled accounts (`superuser@certifiediq.ai`, `admin@quantumbuild.ai`) outside Development. Credentials are config-sourced from `appsettings.Development.json`. E2E fixtures read credentials from env vars with literal fallback. Two regression tests verify the gate works in the Testing environment.

## Files Changed In Scope

### A. `DataSeeder.cs` — gate + config-source

`src/Core/QuantumBuild.Core.Infrastructure/Persistence/DataSeeder.cs`

- Added `using Microsoft.Extensions.Hosting;` and `using Microsoft.Extensions.Configuration;`
- `SeedAsync` now resolves `IHostEnvironment` and `IConfiguration` from the service provider
- The three credentialled calls (`SeedSuperUserAsync`, `SeedAdminUserAsync`, `EnsureAdminEmployeeAsync`) are wrapped in `if (environment.IsDevelopment())`
- Inside the gate: reads `Seed:SuperUser:Email`, `Seed:SuperUser:Password`, `Seed:Admin:Email`, `Seed:Admin:Password` from configuration; logs a warning and skips if any pair is absent
- `SeedSuperUserAsync` signature: `(UserManager<User>, RoleManager<Role>, string superUserEmail, string superUserPassword, ILogger)` — literal `const string` declarations removed
- `SeedAdminUserAsync` signature: `(UserManager<User>, RoleManager<Role>, string adminEmail, string adminPassword, ILogger)` — literal `const string` declarations removed
- `EnsureAdminEmployeeAsync` signature: `(DbContext, UserManager<User>, string adminEmail, ILogger)` — local `var adminEmail = "admin@quantumbuild.ai"` at original line 704 removed; email now flows from caller

## Notable Deviations from Recon Recommendations

**Namespace correction:** The recon specified `using Microsoft.AspNetCore.Hosting;` for `IHostEnvironment`. The correct namespace is `Microsoft.Extensions.Hosting`. The recon used the ASP.NET Core hosting namespace by mistake; the fix used the correct one. No functional difference.

**BACKLOG note wording:** BACKLOG.md §5.31 carried the line `No insecure fallback defaults — fail-fast on missing config.` The implementation chose the recon §5 recommendation (warn-and-skip) over fail-fast. The rationale in the recon §5 is correct: with the defaults always present in committed `appsettings.Development.json`, missing config only happens when a developer explicitly removes or nulls the values. A warning is sufficient; a hard startup block adds no practical value. The BACKLOG note predated the implementation recon's more nuanced recommendation.

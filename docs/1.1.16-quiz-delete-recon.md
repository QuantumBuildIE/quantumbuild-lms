# §1.1.16 — Quiz Deletion / Back-Nav Data Loss — Recon

_Date: 2026-06-23 | Status: Read-only recon_

---

## 1. One-line summary

The two wizard variants have **opposite bugs** — divergent fixes are needed, not a single
shared pattern. The legacy wizard deletes immediately with no undo; the new wizard holds
deletions locally but silently undoes them on back-nav. The recommended fix is:

- **Legacy wizard** → Toast Undo (sonner action button)
- **New wizard** → Save-on-Back (persist before navigating away from the step)

The §1.1.8 pattern (snapshot + re-entry preference) is not the right fit for either case;
see §4 for why.

## 2. Current state

### 2.1 File locations

| Wizard | File |
|--------|------|
| New wizard (URL-per-step) | `web/src/features/toolbox-talks/components/learning-wizard/steps/QuizStep.tsx` |
| Legacy wizard (SPA) | `web/src/features/toolbox-talks/components/create-wizard/steps/QuizStep.tsx` |

The two components share no code beyond both importing from `sonner` and `shadcn/ui`. They
use different state models, different API hooks, and different deletion mechanisms.

Delete triggers — both surface a trash icon on the question card. Neither requires a
confirmation dialog for individual question deletion (the "Regenerate All" button does, but
single-question delete does not). The Explore agent noted an `AlertDialog` wrapper but that
is on the **Regenerate All** button in both files; single-question delete fires directly on
click.

### 2.2 New wizard deletion flow

```ts
// learning-wizard/steps/QuizStep.tsx:140-149
const handleDeleteQuestion = useCallback(
  (index: number) => {
    const current = form.getValues('questions');
    const updated = current
      .filter((_, i) => i !== index)
      .map((q, i) => ({ ...q, questionNumber: i + 1 }));
    form.setValue('questions', updated, { shouldValidate: true, shouldDirty: true });
  },
  [form]
);
```

- Uses React Hook Form (`form.setValue`) — **no API call on delete**
- No toast on delete (silent)
- API call happens only in `handleContinue` → `updateQuestionsMutation.mutateAsync(payload)`

**Back-nav data loss mechanism:**

The component initializes questions from `talk.questions` (server data) on mount via a
`useEffect` guarded by `initializedRef.current`. That ref is instance-local — it resets on
every component unmount. When the user:

1. Arrives at the quiz step → form initialized from `talk.questions` (say 5 questions)
2. Deletes one → form now has 4, server still has 5 (no API call made)
3. Clicks Back → component unmounts, form state is lost
4. Clicks Continue on the previous step → component remounts, `initializedRef.current = false`
5. `useEffect` fires → `talk.questions` still returns 5 → form reinitializes with 5

The deletion is silently undone. The user has no indication this happened.

### 2.3 Legacy wizard deletion flow

```ts
// create-wizard/steps/QuizStep.tsx:217-226
const handleDeleteQuestion = useCallback(
  (questionId: string) => {
    const newQuestions = questions.filter((q) => q.id !== questionId);
    setQuestions(newQuestions);
    if (editingQuestionId === questionId) setEditingQuestionId(null);
    saveQuestionsToServer(newQuestions);
    toast.success('Question deleted');
  },
  [questions, editingQuestionId, saveQuestionsToServer]
);
```

`saveQuestionsToServer` (lines 193–205):

```ts
const saveQuestionsToServer = useCallback(
  (updatedQuestions: QuizQuestion[]) => {
    if (!sessionId) return;
    // From Validated: questions are held locally and saved atomically at handleContinue
    if (session?.status === 'Validated') return;
    updateQuestions.mutate(
      { sessionId, questions: updatedQuestions },
      { onError: () => toast.error('Failed to save questions') }
    );
  },
  [sessionId, updateQuestions, session?.status]
);
```

**Behavior by session status:**

| Status | Delete behaviour | Server call | Toast |
|--------|-----------------|-------------|-------|
| Parsed / QuizGenerated | Immediate `PUT /questions` | ✅ Immediate | `'Question deleted'` (no undo) |
| Validated | Local only (blocked from server) | ❌ None | `'Question deleted'` (misleading — not yet saved) |

**For non-Validated sessions:** the deletion is permanently committed to the backend on
click. `UpdateToolboxTalkQuestionsCommandHandler` performs a hard delete:
```csharp
await _dbContext.ToolboxTalkQuestions
    .Where(q => q.ToolboxTalkId == talk.Id && !incomingIds.Contains(q.Id))
    .ExecuteDeleteAsync(ct);
```
`ToolboxTalkQuestion` has no `IsDeleted` flag — this is irrecoverable at the DB layer.

**For Validated sessions:** the deletion is held in `useState<QuizQuestion[]>`. If the user
clicks Back, the `QuizStep` component unmounts and this local state is lost. On return, the
`useEffect` that hydrates from `quizData` re-runs and restores the server data (which still
has the original questions). Result: the deletion appears to have succeeded (toast fired,
question disappeared) but is silently restored.

## 3. §1.1.8 pattern comparison

§1.1.8 fixed the parse step by introducing a `parsedSectionsSnapshot` in `WizardState`
(parent component, survives step changes) and changing the re-entry hydration logic to
prefer local wizard state over server data when the snapshot is non-null.

**Why this doesn't transfer cleanly to the quiz step:**

| Dimension | §1.1.8 (parse step) | Quiz step |
|-----------|---------------------|-----------|
| State location | `WizardState` (parent) | New wizard: RHF form (instance-local). Legacy: `useState` (instance-local) |
| Back-nav survival | Yes — `WizardState` persists | No — both form state and `useState` are lost on unmount |
| Re-entry trigger | "Don't re-parse if client state non-empty" | Re-entry always re-hydrates from server |
| Fix mechanism | Snapshot + prefer-local on remount | Would need to lift state into parent or save-on-back |

To apply §1.1.8 verbatim to the new wizard, you would need to:
1. Move the questions array into a parent `WizardState` field (e.g., `pendingQuestions`)
2. On remount, prefer `WizardState.pendingQuestions` over server `talk.questions`
3. On Back/Continue, flush `pendingQuestions` to the server and clear the field

This is a medium-sized refactor of the new wizard's step architecture. For the legacy
wizard, the same lift would be needed plus rewriting `handleDeleteQuestion` to not call
`saveQuestionsToServer` immediately.

There is a simpler fix for each wizard that does not require lifting state. See §6.

## 4. Two-pattern comparison

### Toast Undo

- Show `toast.success('Question deleted', { action: { label: 'Undo', onClick: restore } })`
  where `restore` re-adds the deleted question to local state and re-fires `saveQuestionsToServer`
- Sonner supports the `action` field on `toast.success`; no new dependency needed
- Addresses the primary user complaint: "I deleted by accident"
- **Does not address back-nav data loss in either wizard** — if the user navigates away
  before the toast expires, undo is no longer available and the session is gone
- For the new wizard specifically, a toast undo is particularly awkward: the question
  was never saved, so clicking Undo just un-filters the form; but if the user already
  navigated away, the form has been reinitialized from server data anyway (which
  contains the question), making undo a no-op

### Persist-on-Continue (Save-on-Navigate for the new wizard)

The new wizard is already "persist-on-Continue" by design — the deletion only touches form
state, the API call only happens in `handleContinue`. The missing piece is that the Back
button navigates away without saving. The fix is one clause:

```ts
// In the new wizard Back handler
const handleBack = useCallback(async () => {
  const questions = form.getValues('questions');
  if (form.formState.isDirty && questions.length > 0) {
    await updateQuestionsMutation.mutateAsync(/* payload */);
  }
  onBack();
}, [form, updateQuestionsMutation, onBack]);
```

Once the deletion is committed to the server on Back, the server data on re-entry is
correct, and the `initializedRef` guard works as intended.

For the legacy wizard, persist-on-Continue is a larger change (removing the immediate save
from `handleDeleteQuestion`, lifting state). Toast Undo achieves the same recoverable-
delete goal with less disruption.

### Recommendation

Apply both patterns, one per wizard:

1. **New wizard**: **Save-on-Back** (persist-on-navigate). Fixes the silent data loss.
   No change to the Continue path — it already saves. No toast needed; the Save & Continue
   label already communicates that saves happen on explicit action.

2. **Legacy wizard**: **Toast Undo**. Keeps the immediate-save semantics (which are clear
   to users — they see the question disappear and a "deleted" toast). Adds an undo window
   of ~8 seconds. Hold the deleted question in a `deletedQuestionRef` for the undo restore.
   The Validated-session back-nav loss (questions held locally then lost on unmount) is a
   secondary issue that can be tracked separately; it is not the primary complaint.

## 5. Implementation step list

### Fix A — New wizard save-on-Back

**Files to change:** `learning-wizard/steps/QuizStep.tsx`

1. Extract the question payload builder (currently inline in `handleContinue`) into a
   shared `buildPayload()` helper (avoids duplication).
2. Add a `handleBack` callback that:
   a. Builds the payload from current form values
   b. If `form.formState.isDirty`, calls `await updateQuestionsMutation.mutateAsync(payload)`
   c. Catches and swallows errors (don't block back navigation on a save failure)
   d. Calls `onBack()`
3. Wire `handleBack` to the Back button's `onClick` instead of `onBack` directly.
4. Replace the Back button's `disabled={isSaving}` guard with
   `disabled={updateQuestionsMutation.isPending}` to cover the new save path.

**Backend changes:** None. The same `PUT /toolbox-talks/{id}/questions` endpoint is used.

**New tests:** One unit test: navigate Back with dirty form → verify mutation is called.
One unit test: navigate Back with clean form → verify mutation is NOT called.

---

### Fix B — Legacy wizard Toast Undo

**Files to change:** `create-wizard/steps/QuizStep.tsx`

1. Add a `deletedQuestion` ref: `const lastDeletedRef = useRef<QuizQuestion | null>(null)`.
2. Modify `handleDeleteQuestion`:
   a. Store the question being deleted in `lastDeletedRef.current`
   b. Filter local state and call `saveQuestionsToServer` as before
   c. Replace `toast.success('Question deleted')` with:

```ts
toast.success('Question deleted', {
  duration: 8000,
  action: {
    label: 'Undo',
    onClick: () => {
      const deleted = lastDeletedRef.current;
      if (!deleted) return;
      const restored = [...questions, deleted].sort(
        (a, b) => (a.sectionIndex ?? 0) - (b.sectionIndex ?? 0)
      );
      setQuestions(restored);
      saveQuestionsToServer(restored);
      lastDeletedRef.current = null;
    },
  },
});
```

3. `lastDeletedRef` only holds one question (the most recent delete). If the user deletes
   a second question before the first toast expires, the first undo action silently does
   nothing (ref has been replaced). This is acceptable — stack-of-undos would add
   complexity without proportional benefit.

**Backend changes:** None.

**New tests:** One unit test: delete → undo within toast window → verify restored.
One unit test: delete → delete again → verify only latest is undoable.

## 6. Risks and edge cases

### New wizard — save-on-Back when form is invalid

If the user has added a question with no text and clicks Back, `form.formState.isDirty`
is true but validation would fail if triggered. The save-on-Back path should NOT trigger
`form.trigger()` — it should save whatever is in the form. The backend accepts incomplete
questions (it's still in Draft). Validation is only enforced on Continue.

### New wizard — double save on Back → Continue

If a user deletes a question, clicks Back (save fires, server updated to 4 questions),
then clicks Continue again:
- `initializedRef.current = false` → form reinitializes from `talk.questions` which now
  shows 4 (because Back already committed the deletion)
- `handleContinue` fires, saves 4 again — idempotent, no problem

### Legacy wizard — undo after Validated

In the Validated session state, `saveQuestionsToServer` is a no-op. If the user undoes,
the call to `saveQuestionsToServer` is also a no-op. The question is restored in local
state but not on the server. This is correct behaviour — the Validated path already
deliberately holds all changes locally until the user clicks Continue and confirms the
cascade-reset dialog.

### Legacy wizard — multiple rapid deletes

Each delete replaces `lastDeletedRef.current`. Only the last deletion is undoable. The
previous toast's undo button calls `onClick` but `lastDeletedRef.current` is now a
different question — the undo will restore the wrong question. Mitigation: capture the
deleted question in the closure via a local `const captured = question` before the `toast`
call and reference `captured` in `onClick` rather than reading from `lastDeletedRef` inside
the action handler.

Revised approach:

```ts
const handleDeleteQuestion = useCallback(
  (questionId: string) => {
    const captured = questions.find((q) => q.id === questionId) ?? null;
    const newQuestions = questions.filter((q) => q.id !== questionId);
    setQuestions(newQuestions);
    if (editingQuestionId === questionId) setEditingQuestionId(null);
    saveQuestionsToServer(newQuestions);
    if (captured) {
      toast.success('Question deleted', {
        duration: 8000,
        action: {
          label: 'Undo',
          onClick: () => {
            const restored = [...newQuestions, captured];
            setQuestions(restored);
            saveQuestionsToServer(restored);
          },
        },
      });
    }
  },
  [questions, editingQuestionId, saveQuestionsToServer]
);
```

This closes the multiple-delete case: each toast closes over its own `captured` and
`newQuestions`, so each undo is independent.

### Toast duration

The default sonner duration is 4 seconds. Quiz question deletion is more consequential
than a typical "item deleted" action. Recommend `duration: 8000` (8 s) to give the user
enough time to read the toast and decide.

### Back button availability

The new wizard's Back button is currently wired directly to `onBack`. If the Back-nav save
is running (async), the button should show a loading spinner and be disabled until the
mutation resolves — same pattern as the Save & Continue button.

## Appendix: Backend endpoint

| Field | Value |
|-------|-------|
| Endpoint | `PUT /api/toolbox-talks/{id}/questions` |
| Command | `UpdateToolboxTalkQuestionsCommand` |
| Handler | `UpdateToolboxTalkQuestionsCommandHandler` |
| Delete mechanism | `ExecuteDeleteAsync` — hard delete, no `IsDeleted` flag |
| Re-use for batch | Yes — same endpoint used whether deleting 1 or 10 questions |

The endpoint accepts the **full desired question list**; questions absent from the list are
hard-deleted. No separate delete-by-id endpoint exists or is needed for either pattern.
